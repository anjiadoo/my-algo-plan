> **定位**：这不是产品经理的"用户研究教程"，是写给项目总负责人 / 技术 leader / 架构师的"需求体检手册"。目标：当老板抛下一个"我们也要做 AI Agent / 数字员工 / 中台 / 大模型平台"的指令时，**能不被裹挟、能识别出真问题、能在投入大量工程资源之前就把伪需求拦截掉**。读完这篇你应该能回答三个问题：**老板说要做的事到底是不是真问题？为什么用户调研得很辛苦却调研出了假信号？什么时候应该勇敢地告诉老板"这个项目不该做"？**
>
> ****
>
> **核心论点**：在大厂里，**90% 的项目失败不是输在执行，是输在"问题定义"那一关——而问题定义错了，后面的架构设计、技术选型、组织协同都是在精确地解决错误的问题**。架构设计方法论让你"把事情做对"，工程行为方法论让你"把事情做下去"，技术演进让你"选对技术"——但都假设了一个前提：**这件事本身是真的值得做的**。这个前提一旦不成立，前面三套方法论越熟练，烧掉的钱越多。
>

## 核心认知：需求空间 vs 解空间
| / | 需求空间（Problem Space） | 解空间（Solution Space） |
| :---: | :---: | :---: |
| **回答的问题** | 我们到底在解决什么问题？ | 我们用什么方式解决？ |
| **核心物料** | 用户、场景、痛点、动机、约束 | 架构、技术栈、产品形态、流程 |
| **思维模式** | 发散 → 收敛 | 收敛 → 落地 |
| **失败模式** | 把"伪需求"当真需求 | 把对的问题解错了 |
| **承担方** | 项目总负责人 / 业务 leader | 架构师 / 技术 leader |
| **资源弹性** | 低成本、可迭代 | 高成本、改起来贵 |
| **典型时长** | 数周到数月反复打磨 | 数月到数年持续投入 |


**关键洞察**：工程师 / 架构师习惯生活在解空间里——拿到任何需求第一反应是"怎么做"，跳过"为什么做"。但**解空间的最优解，建立在需求空间的正确问题之上**。需求空间错了，解空间做得再漂亮，整个系统都是在为错误的目标做精确优化。**这是为什么很多大厂的 AI Agent / 中台 / 数字员工项目，技术评审都过、上线也成功、但 6 个月后就没人用——因为它们解决的是一个伪问题**。

> 一句话：**需求挖掘的目标不是"让用户满意"，而是"识别出值得用工程资源去解决的真问题"。前者是产品经理的活，后者是项目总负责人的命。**
>

## 第一部分：诊断——为什么大厂的 AI Agent 系统大多都在烂尾
在讲方法之前，先看清楚"敌人"。所有需求挖掘的失败，都可以归到下面 7 种伪需求模式。把它们当成"需求系统的 P0 故障"来记。

### 1.1 七种典型伪需求模式（个人框架）
| 伪需求模式 | 表现 | 根因 | 类比 |
| :---: | :---: | :---: | :---: |
| **跟风型需求** | "友商都在做、我们也要做"，但说不清具体要解决什么 | 战略焦虑，把"不缺席"当价值 | 看到别人买股票就跟着买，不知道这家公司业务 |
| **老板冲动型需求** | 老板某次外出交流 / 听了一个分享后回来要做某件事 | 把"老板的好奇心"当"组织的战略" | 把热搜当 KPI |
| **指标倒推型需求** | "今年要做 N 个 AI 落地案例"，于是凑案例 | 把"完成指标"当"产生价值"，目标和动作错位 | 为了让监控变绿而关掉告警 |
| **解决方案先行** | "我们已经买了 GPU / 大模型，必须找场景落地" | 沉没成本陷阱，资产逼着找问题 | 锤子找钉子 |
| **平均用户型需求** | 调研出"用户平均怎样"，然后做一个谁都不爱用的产品 | 把统计平均当真实用户，没有具象人 | 给身高 1.7 米的"平均人"做衣服 |
| **声音最大型需求** | 把客服工单 / 高管访谈里最响的诉求当全民诉求 | 信号噪音过滤失败 | 把告警面板里出现最多的当成最严重的 |
| **理论上有用型需求** | "这个功能用户应该会喜欢吧"——但没人验证过 | 把"我觉得"当"用户要"，闭门造车 | 没压测就上线，理论上能扛 |


### 1.2 三句话记住根因
1. **用户从来不会主动告诉你他真正要什么**——他只会告诉你他遇到了什么麻烦、他用过的工具有什么不好。**真正的需求要靠"听+看+验证"三段才能逼出来**，光问问不出来。
2. **老板的需求 ≠ 用户的需求 ≠ 业务的需求 ≠ 系统的需求**——这四种需求经常被混为一谈，但它们的优先级、资源、衡量标准完全不一样。**项目总负责人最大的工作量是"翻译"和"对齐"这四类需求，而不是直接执行其中任何一个**。
3. **能在立项前就否掉的伪需求，比上线后再纠错便宜 1000 倍**——别人催你"快点开干"的时候，你在做的"再多花两周搞清楚到底要做什么"，是整个项目最划算的投资。**敢于慢，是为了真的快**。

### 1.3 项目总负责人最容易栽的三个认知陷阱
| 陷阱 | 错误信念 | 真实情况 |
| --- | --- | --- |
| **执行陷阱** | "我的工作是把老板交代的事做好" | 你的工作是**在老板交代的事变成"做好做对"之前，先帮老板厘清那件事到底是什么**。**只做执行的人是工头，做问题定义的人才是 leader**。 |
| **共识陷阱** | "立项会上没人反对，那就是真需求" | 大厂立项会的"沉默"通常意味着两件事之一：（1）大家没听懂、（2）大家觉得提反对意见的代价比配合演戏的代价高。**会上的沉默不是共识，是程序正义**。 |
| **数据陷阱** | "调研报告显示 70% 的用户都需要这个功能" | 70% 的用户在问卷里勾选"我需要" ≠ 70% 的用户在产品上线后会用。**Stated preference ≠ Revealed preference**——人嘴上说要的和真愿意付出代价用的，往往是两件事。 |


### 1.4 中国大厂特有的"需求扭曲场域"（暗规则）
上面 7 种伪需求模式 + 3 个认知陷阱是"通用层"。落到中国大厂的具体语境，还有一组**只有踩过坑才知道**的"暗规则"——它们写不进任何流程文档，但**忽视它们，再科学的需求方法论都会失灵**。

| 暗规则 | 说明 | 落地动作 |
| --- | --- | --- |
| **"老板说"自带最高优先级，但老板说的不一定是老板真要的** | 老板在某个场合说"我们也搞个 AI Agent 试试"，可能只是表达了"我对这个方向有好奇"，而不是"我要 All-In"。但传到下面就变成了 P0 战略项目 | 任何老板需求落地前，**回拨给老板问 3 个问题**：（1）这事您打算解决什么具体业务问题？（2）愿意投多少资源？（3）半年后您想看到什么结果？三个问题答不出来 → 老板自己也没想清楚，先别动手 |
| **"立项 PPT" 比"用户访谈"权重高 10 倍** | 大厂里项目能不能通过，往往取决于 PPT 写得多漂亮、和老板战略关键词的契合度多高，而不是用户研究做得多扎实 | **接受这是规则，但要双轨并行**：PPT 必须写得让老板能拍板（用对方的语言），但**底层用户研究文档必须独立维护**——后者是你做"杀掉判断"时唯一能依赖的真凭据 |
| **"组织面子" 经常压倒"用户里子"** | "公司年报里要写我们已经在 AI 落地了"——这种压力会让一个本可以 6 个月想清楚的项目，被压缩到 3 个月内必须出 demo | **把"对外宣传需求"和"真实落地需求"拆成两个 track**：前者用最小成本满足（一个 demo + 一个新闻稿），后者按真实节奏推进。**不要让对外宣传需求挤占真实落地的资源** |
| **"中台 / 平台 / 基座" 是需求的黑洞** | 一旦项目被定位为"中台"，所有人都可以来提需求，但没有任何一个用户对它的成败负责 | **拒绝"为所有人服务"的中台立项**——必须明确**第一批"灯塔用户"**（最多 2-3 个具体业务团队），用他们的真实使用作为成功标准。没有具体灯塔用户的中台 = 烂尾的预定 |
| **"赛马机制" 让需求在内部就被扭曲** | 一个真实业务问题可能有 3 个团队同时在解决，每个团队都说自己的方案最好，但没人有动力去搞清楚"这个问题到底存不存在" | **赛马的前提是问题已经被验证存在**。**先和潜在赛马团队联合做 1-2 周的需求验证**，确认问题真实之后再开始赛马。否则三匹马在终点都会发现没有奖品 |
| **"内部用户 ≠ 外部用户"，但常被混为一谈** | 内部员工被强制使用一个新系统时，会"配合演出"——给好评、按例使用、写表扬信。这些反馈和外部用户的真实选择毫无可比性 | **内部用户的反馈要打 30% 折，并且必须叠加"行为数据"作为对照**——他们说"很好用"但日活率 5%，前者是政治表态，后者是真相 |
| **"竞品做了我们也要做" 是最难拒绝的伪需求** | 竞品做了一个功能，业务方要求"我们也要做"——但他们做这个功能的上下文（用户结构、商业模式、组织能力）和我们完全不同 | **建一份"竞品功能溯源表"**——每次跟进竞品功能前，先回答 3 个问题：（1）他们为什么做这个？（2）他们做了之后用户行为变化了吗？（3）我们的用户结构和他们一致吗？答不出来 → 跟风的代价是你来背 |


**总结一句**：在中国大厂，**用户需求挖掘从来不是一件"纯方法论"的事——它发生在战略焦虑、组织面子、竞品恐惧、KPI 压力、赛马博弈这些"暗场域"里**。**只懂用户研究方法、不懂组织扭曲场的人，调研做得再认真也会被组织系统反向裹挟**。这不是教你愤世嫉俗，是教你认清"在哪里做需求挖掘"——只有看清场域，方法才能落地。

### 1.5 当前实证：2025-2026 年企业级 AI Agent 已批量进入烂尾期
这套方法论之所以现在尤其紧迫，是因为**这场跟风潮的失败已经开始大规模显现，不再是"理论上可能失败"，而是"已经在烂尾了"**。

2024 年是国内大厂"All-in AI Agent / 数字员工"的立项高峰——几乎每家头部互联网公司、银行、运营商、央企都启动了至少一个亿级预算的"企业级 Agent / AI 中台"项目。**到 2026 年中，这批项目里大量已经悄悄进入下面三种状态之一**（基于公开报道、行业访谈、社区讨论的合并观察）：

| 烂尾形态 | 典型表现 | 占比估算 |
| :---: | :---: | :---: |
| **缩水降级** | 从"通用 Agent 平台"降级为"某个具体场景的内部工具"——比如从"全员 AI 助手"缩水成"客服话术辅助" | 较高 |
| **静默下线** | 上线后日活长期 < 5%，团队不再投入新功能，但也不公开宣布关停（避免"承认失败"） | 中等 |
| **改头换面** | 项目名字保留、PPT 还在讲、对外宣传仍在做，但实际投入大幅收缩，仅留 1-2 人维护 | 中等 |
| **真正用起来** | 找到具体场景、有持续 DAU、有真实业务价值 | **少数** |


**这批烂尾项目的共同特征**（与 §1.1 七种伪需求模式高度重合）：
1. 立项时主要论据是"友商都在做"或"AI 是未来"
2. 立项 PPT 写得漂亮，但回答不出"我们的目标用户是谁、他的具体场景是什么"
3. 没有杀掉条件、没有 Kill Criteria、没有"什么情况下停"的预案
4. 上线后用"PV / 累计用户数"这种易刷指标作为主 KPI，而不是留存 / 完成核心 Job 的次数

**为什么这是这套方法论的"实证基础"**：
- 2023-2024 年讲"AI Agent 项目要做需求体检"是预防性建议，听众容易觉得"危言耸听"
- 2025-2026 年讲同样的话，已经有大量真实案例可对照——**今天读这篇方法论的人，大概率自己公司里就能找到至少一个上述形态的项目**
- 这套方法论的真正定位不是"未来的指南"，是"**当下烂尾潮里的止损工具**"——能用得越早，烧掉的钱越少

> **观察**：**2026 年企业级 AI 项目的成功率，比 2024 年立项时所有人预期的低一个数量级**。但这不是 AI 不行、也不是技术不行——**是大量项目从一开始就没解决"为谁解决什么真问题"这个最基本的问题**。这套方法论的全部目的，就是不让你的项目成为下一个统计数字。
>

## 第二部分：用户与场景——先搞清楚"为谁解决什么问题"
所有需求挖掘的第一步，不是"调研"，而是把这两个问题答清楚：**用户是谁？他在什么场景下遇到什么问题？**

很多项目卡在这一步——比如"企业级 AI Agent 系统"，听起来很清晰，但拆开问就发现：用户是 CEO？是部门 leader？是一线员工？是 IT 运维？这四个用户的需求差异比"做不做这个项目"还大。

### 2.1 四象限：用户 / 客户 / 决策者 / 付费方
B 端项目最容易混淆的概念。在 C 端这四者通常是同一个人，但在 B 端 / 大厂内部，它们是四种不同的角色，**需求和评价标准完全不同**。

| 角色 | 定义 | 关心什么 | 典型场景：企业级 AI Agent |
| :---: | :---: | :---: | :---: |
| **用户（User）** | 真正使用产品的人 | 用起来顺不顺手、能不能解决眼前的事 | 一线员工：用 Agent 写周报、查数据 |
| **客户（Customer）** | 接受产品交付的人（B 端常是采购方 / 业务负责人） | 业务指标有没有改善、合规风险大不大 | 部门负责人：员工效率有没有提升 |
| **决策者（Decider）** | 决定要不要买 / 要不要立项的人 | 战略契合度、对外形象、ROI | CTO / CEO：公司是否在 AI 转型上"不掉队" |
| **付费方（Payer）** | 真正出钱的人（可能是 CFO / 财务 / 业务部门预算） | 成本可控、可量化、不超预算 | CFO / 业务 BP：投入产出比、隐性成本 |


**典型错位（这是大量项目失败的根源）**：

+ **决策者下单 + 用户不愿用**：CEO 拍板要做 AI Agent，员工拿到工具发现还不如自己手写——上线 3 个月，日活 < 5%
+ **客户买单 + 付费方反对**：业务部门负责人觉得有用，但 CFO 看不到 ROI 报表，第二年预算被砍
+ **用户喜欢 + 决策者不感冒**：一线员工真的爱用，但老板看不到"亮点"和"成绩单"，项目得不到资源升级

**落地动作**：项目立项之前，**强制画一张"四象限对照表"**：

```plain
项目：企业级 AI Agent

【用户】       一线员工（写文档/查数据/跑流程）
              成功标准：每周节省 2 小时
              失败模式：用起来比不用还累

【客户】       各部门 TL
              成功标准：团队人效 +10%
              失败模式：员工骗他说有用，他被骗了

【决策者】     CTO + CEO
              成功标准：公司 AI 战略落地有"标杆案例"可对外讲
              失败模式：年报里写不进去

【付费方】     CFO（IT 预算）+ 各业务 BP
              成功标准：单 token 成本可控、没有越界使用
              失败模式：年中预算超支被通报

⚠️ 风险：四个角色的成功标准之间有没有矛盾？
   - 用户成功 vs 决策者成功：可能冲突（员工要好用，老板要漂亮）
   - 客户成功 vs 付费方成功：可能冲突（部门要 AI 多用，CFO 要省钱）
```

**关键判断**：**四个角色的成功标准如果存在结构性矛盾，必须立项前就解决——不是上线后再"协调"。** 上线后协调的代价是 100 倍。

### 2.2 Jobs-to-be-Done：用户雇你做的"工作"是什么
**JTBD 框架（Jobs-to-be-Done）** 由 Clayton Christensen 在 2003 年《The Innovator's Solution》中系统提出，2016 年《Competing Against Luck》进一步完善。核心命题：

> "用户不是在购买产品，而是在'雇佣'一个产品来完成某个'工作'（Job）。理解 Job，比理解产品本身更重要。"
>

经典案例：Christensen 调研奶昔销量低的原因，发现早晨购买奶昔的用户雇它做的"工作"不是"喝早餐"，而是"在通勤路上一手开车一手有件事干、不会饿到 10 点"。理解了这个 Job，奶昔的产品改进方向（更稠、更慢、更耐喝）才浮出水面。

**JTBD 的标准句式**

```plain
当 [情境/触发条件] 时，
我想 [完成某个动作 / 达到某个状态]，
这样我可以 [获得某种回报 / 避免某种损失]。
```

**用 JTBD 改写"AI Agent 需求"**

❌ 模糊版（典型立项语言）：
> "员工需要一个 AI Agent 来提升办公效率。"

✅ JTBD 改写后（暴露真实需求）：
> "当我每周五下午要写一份给老板的工作周报时（情境），
> 我想快速从 OA、IM、文档系统里把这周做的事捞出来汇总（动作），
> 这样我可以不用花 2 小时手动整理、还能提早下班接孩子（回报）。"

**对比**：第一个写法可以做出 100 种产品形态——通用 Agent、IM 助手、文档总结、流程自动化……每种成本千万级别，但都不一定解决问题。第二个写法把范围收敛到"周报场景的多源信息聚合 + 自动撰写"——成本可控、价值可验证、是不是真问题也容易测。

**JTBD 三层细化**

每个 Job 还可以拆成三层：

| 层次 | 含义 | "周报场景" 例子 |
| --- | --- | --- |
| **功能 Job** | 任务本身 | 把这周做的事汇总成结构化文本 |
| **情绪 Job** | 用户内心想感觉到什么 | 不想周五加班、不想被老板觉得"工作量不饱满" |
| **社交 Job** | 用户希望别人怎么看自己 | 周报有结构、有数据、显得专业可靠 |


**关键洞察**：**很多 B 端项目只优化了"功能 Job"，但用户真正在乎的是"情绪 Job"和"社交 Job"**。比如周报场景：技术上做一个"自动汇总"很简单，但用户最深的诉求是"不想被老板觉得我这周没干事"——这个诉求决定了周报的语气、详略、亮点提取，而不是技术能力本身。

### 2.3 痛点三层次：症状 / 问题 / 根本动机
用户说出来的"痛点"，往往只是冰山一角。**真正能驱动产品决策的，是把痛点剥到"根本动机"那一层**。这是个人框架，但本质上是 5 Whys（详见 §3.1）的简化版本。

| 层次 | 用户的表达方式 | 例子（周报场景） | 工程师的对应反应 |
| --- | --- | --- | --- |
| **症状（Symptom）** | "这事很烦/很慢/很累" | "每周写周报要 2 小时" | ❌ 想用技术直接消除症状 |
| **问题（Problem）** | "因为 X 所以累" | "因为我要从 4 个系统里捞数据，每个系统格式都不一样" | ⚠️ 开始接近本质，但还不够 |
| **根本动机（Job-to-be-done）** | "我真正想要的是 Y" | "我想用 30 分钟搞完，剩下时间陪家人 / 干别的事 / 不被老板觉得我加班还摸鱼" | ✅ 这是产品决策的真实输入 |


**为什么必须挖到第三层**

+ 停在症状层 → 做出来的产品是"自动化周报工具"，可能根本没人用（因为周报本身可能是个伪需求）
+ 停在问题层 → 做出来的产品是"多源数据聚合器"，技术上很厉害，但用户不感冒（因为他要的不是聚合，是"快速完成"）
+ 挖到动机层 → 做出来的产品可能根本不是"AI Agent"，而是"周报模板 + 一键填充 + 智能润色"——形态简单、价值清晰、用户真用

**实操：挖动机的话术（个人提炼）**

不要直接问"你为什么想要这个"——这种问题用户答不上来或会编一个理由。用下面三句话循环追问：

1. **"上次你做这件事的时候，具体是什么场景？周几？什么时间？前后在做什么？"**——把抽象需求拽回具体情境
2. **"如果今天没有这个工具，你会怎么应付？最差的情况是什么？"**——挖出"避免损失"维度的动机（人对损失的敏感度高于收益 2 倍——Kahneman, 1979）
3. **"如果这件事彻底解决了，你最想做的下一件事是什么？"**——挖出"获得收益"维度的动机，往往这才是真正的驱动力

### 2.4 场景描述模板（5W1H + 频次 + 现状解法 + 替代品）
有了 JTBD 还不够，**需求要落地到"具体场景"**——抽象 Job 是产品方向，具体场景是设计输入。一个合格的场景描述模板需要覆盖 10 个要素（5W1H + 频次 + 现状解法 + 替代品 + 红线）：

```plain
# 场景：员工写周报（个人框架模板）

## 5W1H
- Who（谁）：研发岗员工，工作 1-5 年，每周需要写工作周报
- When（何时）：每周五下午 4-6 点，或周一上午前提交
- Where（哪里）：在公司办公电脑上、IM 里、有时在家
- What（做什么）：把这周做的工作汇总成 300-500 字的结构化文本
- Why（为什么）：老板要求 + 公司流程 + 个人绩效记录
- How（怎么做）：手动翻 Jira、IM 群、文档历史、邮件，复制粘贴整理

## 频次
- 100% 员工每周必做 1 次
- 平均耗时 2 小时
- 80% 员工觉得这是"必要但讨厌的事"

## 现状解法（用户当前怎么应对）
- 自己手动整理（70%）
- 复制上周改一改（20%）
- 找助理代写（5%）
- 干脆不写或写得敷衍（5%）

## 替代品（用户可能用什么替代你的方案）
- 通用大模型自己手动喂内容（已经有员工在用 DeepSeek / 豆包 / 通义千问 / 文心一言 / Kimi / ChatGPT 等）
- 公司原有 OA 周报模板
- 不写 / 应付了事（最强的"替代品"是"什么都不做"）

## 不能错的红线
- 周报内容必须真实，不能瞎编（涉及绩效）
- 不能泄露代码 / 数据到外部（合规）
- 不能让老板觉得"周报是 AI 写的"（社交 Job）
```

> **2026 年的反讽现实**：很多公司高层一边要求"AI 全员渗透率 ≥ 80%、必须用 AI 写周报"，一边一线员工却担心"周报被认出是 AI 写的会被认为不走心"——**这两条红线方向是相反的，且通常不会被公开讨论**。这种"高层方向"和"一线感受"的撕裂本身就是真实痛点：员工真正需要的产品形态可能不是"AI 写得越多越好"，而是"AI 帮我准备素材、但保留我自己的语气和表达"——**这正是 §2.2 提到的"社交 Job"在 AI 时代的具体表现**。做这类产品时，不识别这种撕裂，就会做出"高层验收满意但一线偷偷停用"的产品。
>

**关键：把"替代品"那一栏写好**

很多项目失败的原因是**忽视了"什么都不做"和"用户已经有的土办法"也是替代品**。如果你的产品价值不能显著高于这两个替代品，就不该上。

> **金句**："用户最大的竞品是'什么都不做'——能让用户从'不做'变成'做'的产品，才是真正解决问题的产品。"——这是产品社区长期沉淀的共识表述（在 Des Traynor、Ash Maurya、Andy Rachleff 等多位实践者的论述中均出现过类似表述），不一定能归到某一位作者名下。
>

### 2.5 用户分群：从"平均用户"到"具象人物"
"平均用户"是需求挖掘里最大的陷阱之一。**真实的用户群体永远是分层的，统计平均往往没有任何一个真人对应**。

**分层的三种维度（按需选用）**

| 分层维度 | 说明 | 适用场景 |
| --- | --- | --- |
| **按使用频次** | 重度 / 中度 / 轻度 | 工具类产品、SaaS、内部系统 |
| **按使用阶段** | 新手 / 上手期 / 熟练 / 专家 | 复杂工具、需要学习曲线的产品 |
| **按角色** | 决策者 / 执行者 / 协作者 / 旁观者 | 协作类产品、B 端工作流产品 |


**Persona 模板（Persona 概念由 Alan Cooper 在 1998 年《The Inmates Are Running the Asylum》中提出）**

```plain
# Persona：研发周报重度用户"小李"

基础画像：
- 28 岁，男，工作 4 年，研发岗（后端）
- 已婚，孩子 1 岁，住公司 30 分钟通勤距离

工作场景：
- 每周参与 3-5 个项目，跨多个 Jira 看板
- IM 群 30+ 个，每周收发消息 2000 条
- 周报必须有"具体数据 + 项目进展 + 风险预警"三段

痛点（按严重程度排序）：
1. 跨系统翻找信息——平均花费 60 分钟
2. 把工程语言"翻译"成老板能看懂的话——25 分钟
3. 担心遗漏重要事项被老板觉得"没干事"——情绪负担

不会做的事：
- 不会用 ChatGPT（公司禁止）
- 不会用 Notion / 国外 SaaS
- 不会写 Python 自动化脚本（虽然是研发但嫌麻烦）

愿意付出的成本：
- 时间：愿意花 5 分钟设置 / 学习
- 金钱：公司付费的话愿意用，自己不愿付
- 隐私：不愿把周报内容上传到外网

衡量"好用"的标准：
- 节省时间 > 50%
- 老板看了不会觉得"敷衍"
- 不用学新工具
```

**关键技巧：每个 Persona 必须配 1-2 个真实的人作为"原型"**

Persona 不是凭空捏造的"理想用户"——它必须基于你访谈过的真实的人。**找不到真人作为原型的 Persona，就是你想象中的用户**。

落地动作：每个 Persona 旁边写"原型：王某（XX 部门）+ 李某（YY 部门）"——这样以后产品决策有疑问时，可以回到这两个真人去问。**抽象的 Persona 容易被组织讨论稀释，具体的真人会守住产品方向**。

### 2.6 场景边界：In Scope vs Out of Scope vs Future
比"做什么"更重要的是"明确不做什么"。一个项目立项时如果不显式声明"不做什么"，后续会被无数"小需求"灌死。

**三个边界清单（必须写进立项文档）**

```plain
# 项目：企业级 AI Agent v1（个人框架模板）

## In Scope（首批必做，2026 H2 上线）
- 周报场景：跨 Jira / IM / 文档自动汇总 + 智能撰写
- 适用人群：研发岗 + 产品岗（约 3000 人）
- 典型用例：每周一次、单次 < 1 分钟生成

## Out of Scope（明确不做，避免被 sneak in）
- ❌ 客服 / 销售 / HR 等其他岗位的工作流
- ❌ 多轮对话型 Agent（开放任务）
- ❌ 跨语言（仅中文）
- ❌ 移动端（仅 PC Web）
- ❌ 接入外部 SaaS（仅公司内系统）
- ❌ 替代周报这件事本身（即不质疑"周报"流程，仅优化工具）

## Future（v2 候选，立项时不承诺）
- 月报、季报扩展（取决于 v1 验证结果）
- 销售岗周报（不同 JTBD，需要重新研究）
- 移动端入口（取决于 PC 端日活）

## Kill Criteria（什么情况下整个项目终止）
- 上线 3 个月内日活 < 30% 目标用户
- 用户满意度 NPS < 30
- 老板的"对外宣传"诉求已被替代方案满足，本项目失去战略意义
```

**为什么 Out of Scope 比 In Scope 更值钱**

In Scope 是给团队的指南，Out of Scope 是给老板和合作方的"防御工事"。**没有显式 Out of Scope 的项目，会在每次评审中被加进新需求，最后做成一个谁都不爱用的"All-in-One"**。

> **个人观察**：在大厂里，"不做"比"做"难 10 倍——因为"做"是政绩，"不做"是失语。但**长期看，"敢于不做"的项目总负责人，最终交付的产品价值反而最高**。
>

## 第三部分：从模糊到清晰——核心方法论工具箱
有了用户和场景，接下来是真正"剥洋葱"的环节——把**模糊的诉求**逐层挖到**可执行的产品决策**。下面 6 个工具按使用顺序排列，每个解决一个特定问题。**不是全都要用——按你项目的复杂度按需取用**。

### 3.1 5 Whys——剥洋葱式追问
**5 Whys** 起源于丰田汽车工业实践——丰田佐吉提出"问五次为什么"的原始思想，丰田生产系统（TPS）创始人大野耐一（Taiichi Ohno）在 1950s-1970s 把它系统化为 TPS 的核心问题分析工具，1988 年大野耐一在《Toyota Production System》英文版中正式向全球普及。核心命题：**对任何问题连续追问 5 次"为什么"，能从表面症状挖到根本原因**。

**经典案例（丰田生产线）**：

```plain
现象：机器停了。
Why 1：保险丝烧了。
Why 2：因为电流过大。
Why 3：因为轴承润滑不足。
Why 4：因为润滑泵抽不上油。
Why 5：因为润滑泵的轴磨损了。
→ 根本原因：润滑泵设计缺陷，需更换型号
   而不是"换保险丝"——那只是治标
```

**用 5 Whys 拆"我们要做 AI Agent"**

```plain
Why 1：为什么我们要做 AI Agent？
→ 因为公司战略要 AI 转型。

Why 2：为什么公司战略要 AI 转型？
→ 因为友商都在做、董事会问起来不好交代。

Why 3：为什么董事会问起来不好交代？
→ 因为缺乏可对外讲的"AI 落地案例"。

Why 4：为什么缺乏案例？
→ 因为之前的 AI 投入分散、没有标杆项目。

Why 5：为什么没有标杆项目？
→ 因为业务方和技术方都在等"对方先动"，没人愿意承担试错成本。

→ 真正的问题不是"做不做 AI Agent"，而是
  "谁来承担首批 AI 落地的试错成本，怎么把试错转化成可对外讲的故事"
```

**关键洞察**：5 Whys 不是为了问出"答案"，是为了把**模糊的口号还原成"可被工程化的问题"**。原始问题"做 AI Agent"工程上无法启动；最终问题"试错成本+案例叙事"则可以拆出明确的方案——找一个低风险高曝光的场景做灯塔、明确试错预算上限、提前规划对外宣传节奏。

**5 Whys 的三个常见误区**

| 误区 | 表现 | 修正 |
| --- | --- | --- |
| **过早停止** | 问到第 2 层就觉得"想清楚了" | 至少问到第 5 层；如果答案变成"这就是常态"——再问一次，常态本身值得质疑 |
| **追问个人** | 问"为什么 X 没做好"，矛头指向个人 | 永远问"为什么这个机制 / 这个流程产生了 X 结果"——追问机制不追问人 |
| **直线追问** | 只沿一条路径往下挖 | 重要节点用"5 Whys 树"——同一层多个原因都展开，避免被单线思维误导 |


### 3.2 第一性原理——回到根本约束
**First Principles Thinking** 来自亚里士多德，被 Elon Musk 在 SpaceX/Tesla 推广为现代工程语境中的"砍价方法论"。核心命题：**抛开类比和经验，把问题分解到不可再分的物理 / 经济 / 用户行为基本要素，然后从基本要素重新组合**。

**经典案例（SpaceX）**：

```plain
传统思路（类比思维）：
"火箭以前都要 6 千万美元一发，业内成本就这么贵"
→ 找便宜供应商砍 10%，已是极限。

第一性原理：
"火箭由铝合金、钛、铜、碳纤维构成，
 这些原材料按市场价计算 = 200 万美元
 → 那 5800 万美元差价是哪来的？"
→ 制造工艺、供应链、垄断溢价
→ 自己造、复用、垂直整合
→ 单发成本降到 600 万美元
```

**用第一性原理拆"AI Agent 提升办公效率"**

```plain
模糊命题："AI Agent 能提升办公效率"

第一性拆解：
1. "效率"由什么构成？
   → 单位时间产出 = 产出量 / 投入时间

2. 现在的瓶颈在哪一边？
   → 不是产出量（员工已经做了大量事）
   → 是投入时间（员工 30% 时间花在"信息搬运"：
     找文档、汇总数据、发提醒、写格式化文本）

3. "信息搬运"中哪一类是 AI 真正擅长的？
   → 跨系统取数 + 格式转换 + 内容总结
   → 但 NOT 创造性判断、跨人协调、政治表态

4. 那些"擅长"的部分占员工总时间多少？
   → 估算 8-12%（不是 30%）

5. 8-12% 的时间节省，按员工成本算，
   投入产出比是多少？
   → 单员工 30 万年薪 × 10% = 3 万节省/年
   → 1000 员工 × 3 万 = 3000 万/年节省
   → vs Agent 系统建设 + 运维成本 = 2000 万/年
   → 净 ROI = 1000 万/年（前提：覆盖率到 100%）

→ 真正可工程化的命题：
   "选一类 AI 擅长的'信息搬运'任务，
    把它的覆盖率推到 80% 以上"
   而不是泛泛的"提升办公效率"
```

**关键洞察**：第一性原理的真正威力在于**把模糊的战略口号拆成"可估算 / 可证伪"的子命题**。一旦命题可估算，老板拍板的成本和你的执行风险都会下降一个数量级。**模糊命题的危险在于"无法被证伪"——做不出来都可以归咎于"努力不够"**。

**第一性原理的"三层拆解模板"**

```plain
原始命题：[老板的口号 / 业界的流行语]
   ↓
拆解到"用户行为"层：用户每天做什么？花多少时间？
   ↓
拆解到"经济"层：每个动作值多少钱？省下来值多少？
   ↓
拆解到"物理 / 技术"层：技术上能做到的边界在哪？
   ↓
重组：在三层约束下，最高 ROI 的切入点是什么？
```

### 3.3 Outcome-Driven Innovation——从"想要什么结果"倒推
**Outcome-Driven Innovation（ODI）** 由 Tony Ulwick 在 1990s 提出、2005 年《What Customers Want》系统化。核心命题：**用户不在乎你的产品，只在乎"完成 Job 时能达到的结果指标"**。

**ODI 的核心步骤（简化版）**

```plain
1. 识别用户的 Job（来自 §2.2 JTBD）
2. 列出 Job 的"期望结果"（Desired Outcomes）
   - 每个结果用标准句式：[最小化/最大化] 某个 [可度量指标]
3. 调研用户对每个结果的"重要性"和"满意度"
4. 计算"机会分数" = 重要性 + max(0, 重要性 - 满意度)
5. 高机会分数 = 真痛点 = 真值得做
```

**用 ODI 拆"周报场景"**

```plain
Job：每周完成给老板的工作汇报

期望结果（按 ODI 标准句式）：
1. 最小化"汇总信息所需时间"
2. 最小化"周报被老板要求重写的概率"
3. 最小化"漏报重要事项的概率"
4. 最大化"周报中数据的准确率"
5. 最大化"老板对'我这周干了事'的认知"
6. 最小化"被同事看到周报后觉得我'吹牛'的概率"

调研（n=50 员工）：
                    重要性  满意度  机会分
1. 汇总时间          9.0     3.5    14.5
2. 被要求重写        7.5     6.0     9.0
3. 漏报重要事项      8.5     5.0    12.0
4. 数据准确率        8.0     7.5     8.5
5. 老板的认知        9.5     5.0    14.0   ← 最高优先级之一
6. 同事的看法        6.0     5.5     6.5

→ 最值得攻克的是 #1 + #5：
  "节省时间" + "让老板看见我"
  这两条决定了产品要不要做"亮点提取 + 数据可视化"，
  而不是单纯的"内容自动生成"
```

**关键洞察**：ODI 把"用户喜欢什么功能"这个不靠谱的问题，转换成了"哪些结果是用户最痛但还没满足的"——后者既客观又可量化。**这是把"用户调研"从艺术变成科学的关键转折**。

**机会分数的解读**

| 机会分数 | 含义 | 行动 |
| --- | --- | --- |
| **> 15** | 极度未被满足的痛点 | 必做、有显著差异化潜力 |
| **10-15** | 重要但部分被满足 | 选择性做、看竞争格局 |
| **5-10** | 一般 | 不要重点投入 |
| **< 5** | 已经满足或不重要 | 不做 |


### 3.4 Kano 模型——区分基础 / 期望 / 兴奋需求
**Kano Model** 由日本东京理科大学（Tokyo University of Science）狩野纪昭（Noriaki Kano）教授在 1984 年提出。核心命题：**用户需求不是线性的——同样的功能，按"基础 / 期望 / 兴奋"分类，对用户满意度的影响曲线完全不同**。

**Kano 三类需求**

```plain
满意度 ↑
       │              ╱─── 兴奋需求（Excitement）
       │            ╱      没有→无感；有→惊喜
       │          ╱
       │        ╱─────────  期望需求（Performance）
       │      ╱             功能越好→越满意（线性）
       │    ╱
       │──╱─────────────→  实现度
       │╱
       │   ─────────────── 基础需求（Basic / Must-be）
       │                   有→不抱怨；没有→愤怒
满意度 ↓
```

| 需求类型 | 用户的态度 | "周报场景" 例子 |
| :---: | :---: | :---: |
| **基础需求** | 有不夸、没有就骂 | 数据准确不出错、不能泄露公司机密 |
| **期望需求** | 越好越满意 | 生成速度越快越好、自动汇总越完整越好 |
| **兴奋需求** | 没有不会怪、有了惊艳 | 自动检测"周报亮点"并加粗、自动适配老板的偏好语气 |


**关键洞察**：**做产品最大的浪费是把基础需求当兴奋需求做、把兴奋需求当基础需求做**：

+ ❌ 基础需求当兴奋需求：花大量资源去"优化"数据准确率从 99% 到 99.5%——用户根本无感（因为这本来就该有）
+ ❌ 兴奋需求当基础需求：把"老板偏好语气适配"做成默认开关，结果用户觉得"这玩意儿管太多"

**Kano 调研的标准问法（必须双向问）**

每个候选功能问两个问题：

1. **正向问**："如果产品**有**这个功能，你的感受是？"
2. **反向问**："如果产品**没有**这个功能，你的感受是？"

每题 5 个选项：①很喜欢 ②理应如此 ③无所谓 ④勉强接受 ⑤不喜欢

根据正反两题的组合，落到 Kano 二维矩阵里——这才是 Kano 调研的精髓，单向问问不出真正的需求类型。

### 3.5 价值主张画布——把用户和产品挂上钩
**Value Proposition Canvas** 由 Alex Osterwalder 在 2014 年《Value Proposition Design》提出，是商业模式画布的延伸。核心命题：**用户侧（Customer Profile）和产品侧（Value Map）必须显式对齐——任何脱节的功能都是浪费**。

**画布结构**

```plain
┌─────────── 产品侧（Value Map） ─────────┐  ┌──── 用户侧（Customer Profile） ────┐
│                                          │  │                                       │
│  Products & Services                     │  │  Customer Jobs                        │
│  （我们提供什么）                          │  │  （用户要完成什么）                     │
│                                          │  │                                       │
│  Pain Relievers                          │  │  Pains                                │
│  （我们如何缓解痛苦）          ←─对齐─→   │  │  （用户的痛苦）                         │
│                                          │  │                                       │
│  Gain Creators                           │  │  Gains                                │
│  （我们如何创造收益）                      │  │  （用户期待的收益）                      │
│                                          │  │                                       │
└──────────────────────────────────────────┘  └───────────────────────────────────────┘
```

**用 VPC 填"AI 周报助手"**

```plain
【用户侧】

Customer Jobs：
- 每周完成 300-500 字的工作周报
- 让老板看到"我这周做了事"
- 不影响周末时间

Pains：
- 跨 4-5 个系统找信息（耗时）
- 怕漏报重要事项被老板觉得没干事（焦虑）
- 把工程语言翻译成老板能懂的话很累
- 周报写得太"敷衍"会被绩效打折

Gains：
- 30 分钟内搞完
- 周报有结构、有数据、显得专业
- 老板看了能记住我做的几件大事
- 不用学新工具

【产品侧】

Products & Services：
- AI 周报助手（Web 端，集成 Jira/IM/文档）

Pain Relievers：
- 一键拉取本周所有任务（缓解"找信息"痛苦）
- 智能识别"重要事项"打 ⭐（缓解"漏报"焦虑）
- 自动把技术语言转成业务语言（缓解"翻译累"）

Gain Creators：
- 模板化结构 + 自动数据嵌入（创造"专业感"收益）
- 自动提取本周 TOP 3 亮点（创造"被老板看见"收益）
- 0 配置即用，IM 入口（创造"不用学新工具"收益）

⚠️ 对齐检查：
- 每个 Pain Reliever 必须对应至少一个 Pain ✅
- 每个 Gain Creator 必须对应至少一个 Gain ✅
- 反向：每个 Pain / Gain 都被覆盖了吗？
  → "怕周报敷衍被绩效打折" 还没对应方案 ⚠️
  → 需要加一个 Pain Reliever：周报"完整度"评分提示
```

**关键技巧：对齐检查反向做**

写完画布后，**强迫自己反向检查**——每一条 Pain 和 Gain 是不是都有对应的 Pain Reliever 和 Gain Creator？没有的就是产品价值的"漏洞"。**80% 的产品做了一堆无人在乎的功能，同时漏掉了用户真正在乎的痛点——VPC 反向检查就是用来填这个漏洞的**。

### 3.6 用户故事地图——把零散需求编织成产品骨架
**User Story Mapping** 由 Jeff Patton 在 2014 年《User Story Mapping》系统化（早期实践可追溯到 2005 年）。核心命题：**用户故事不是 backlog 里一堆扁平卡片，而应该按"用户旅程 + 优先级"组织成二维地图**。

**故事地图结构**

```plain
横轴：用户旅程（按时间顺序的"史诗活动"）
纵轴：优先级（从核心到锦上添花）

┌──────────┬──────────┬──────────┬──────────┬──────────┐
│  打开工具 │ 拉取数据  │ 编辑内容  │  审阅周报 │  发送提交 │ ← 史诗活动
├──────────┼──────────┼──────────┼──────────┼──────────┤
│  IM 入口  │ Jira 集成 │ 模板填充  │ 一键预览  │ 一键发送  │ ← MVP（第一刀切线，必须连成端到端）
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ Web 入口  │ IM 集成   │ 智能润色  │ 拼写检查  │ 定时发送  │ ← v2 增强
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ 桌面客户端│ 邮件集成  │ 多语言    │ 协作评论  │ 多人审批  │ ← v3 锦上添花
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

**关键洞察**：用户故事地图最大的价值是**保证 MVP 是"端到端可用"的，而不是某个单点完美**。常见的产品失败模式：

+ ❌ **柱子型 MVP**：一个环节做得超完美（比如智能润色），但其他环节没做（拉取数据要手动），用户没法形成完整闭环 → 没人用
+ ✅ **横线型 MVP**：每个环节都做最简版本但能跑通——拉取数据虽然只支持 Jira，编辑虽然只有模板，但用户能完整走一遍流程 → 可以验证

**实操：故事地图的画法（个人提炼）**

1. **第一步：横轴只画"动词" + "宾语"**——"打开工具 / 拉取数据 / 编辑内容"，不要画"使用 IM 集成"。横轴是用户视角的活动，不是产品视角的功能
2. **第二步：每个活动下面贴 3-10 张卡片**——用最简单的纸条，写"这一步用户具体做什么"
3. **第三步：横切第一刀 = MVP**——选每个活动的最低可用版本，连成端到端
4. **第四步：横切第二刀、第三刀 = v2、v3**——每一刀都必须保证端到端，不能只增强单个环节

## 第四部分：调研与访谈——避免被假信息误导
有了工具箱还不够。**80% 的调研失败不是因为没用对工具，是因为问错了问题、信错了答案**。这一部分讲清楚怎么从用户嘴里和数据里提取真信号，避开假信号。

### 4.1 The Mom Test——三条铁律
**The Mom Test** 由 Rob Fitzpatrick 在 2013 年《The Mom Test》一书提出。书名的来源是个直白的设定：**如果你问妈妈"我这个产品想法怎么样？"，她一定会说"很棒，宝贝！"——但这个答案毫无价值**。

Rob 的核心洞察：**任何"问别人觉得你的想法怎么样"的访谈都是垃圾**——人天生不愿伤别人的心，会下意识给好评。**好的访谈不应该让被访者表态，应该让他讲事实**。

**三条铁律（直接背下来）**

| 铁律 | 错误问法 | 正确问法 |
| --- | --- | --- |
| **① 谈他的生活，不谈你的想法** | "你觉得 AI 周报助手怎么样？" | "上次你写周报花了多久？卡在哪一步？" |
| **② 问具体过去，不问抽象未来** | "你以后会用我们的产品吗？" | "上一次你用 X 工具是什么时候？为什么放弃了？" |
| **③ 少说多听**（说话比例 ≤ 30%） | 滔滔不绝介绍产品 30 分钟 | 准备 5 个开放式问题，让对方占 70% 时间 |


**反例：典型的"垃圾访谈"**

❌ 错误访谈：
> 你：我们想做一个 AI 周报助手，能自动从 Jira / IM 拉数据，然后用大模型生成周报，你觉得有用吗？
>
> 对方：哇听起来挺好的，应该会有用吧。
>
> 你：那你愿意每月付 50 块用吗？
>
> 对方：嗯……如果好用的话可以试试。
>
> 你：太好了！我们这周就开始做。

**这段访谈的所有信息都是假的**：对方在客气，"挺好"、"试试"、"如果"都是社交润滑词，不是承诺。

✅ 正确访谈：
> 你：上周五你写周报，能给我讲讲过程吗？比如几点开始、做了什么？
>
> 对方：大概 4 点半开始，先去翻 Jira 找这周关单的票，然后翻 IM 群找几个讨论记录……大概搞到 6 点多。
>
> 你：6 点多搞完之后，有没有重新看一遍？
>
> 对方：看了一眼，发现漏了一个昨天和产品的讨论，又加了一段。
>
> 你：那这个"漏了"的事经常发生吗？最近一次让你有点焦虑的漏报是什么？
>
> 对方：上个月漏报了一个我推动的小项目，老板月度复盘时点名问，挺尴尬的。
>
> 你：那次之后你做了什么改变吗？
>
> 对方：我现在每周三会先列一个 todo list 提醒自己……（暴露真实痛点）

**这段访谈得到的信息都是真的**：具体场景、具体痛点、具体补救方案，全是事实。

**Mom Test 的扩展铁律（个人提炼）**

| 扩展铁律 | 落地动作 |
| --- | --- |
| **永远索要"承诺货币"** | 让对方付出一点真实代价（时间 / 介绍朋友 / 试用 / 预付）——口头答应不算数 |
| **聊到"这个东西要 50 块"对方眼睛亮 ≠ 真要付** | 真要付的人会立刻问"怎么付、什么时候开始、能否给我账号"——眼睛亮但不追问 = 客套 |
| **任何"将来会"都打 80% 折** | "我以后会用"打 80% 折；"我现在就想要"打 0% 折；"我朋友 X 一定会用"打 100% 折 |


### 4.2 好问题 vs 坏问题清单（含话术）
延续 Mom Test 思路，下面整理一份"问题词典"——**每一组都对应一个真实场景，可以直接拿去用**。

**A. 挖痛点的好问题**

```plain
✅ 好问题：
- "上次你做 [X 任务] 的时候，最让你挫败的瞬间是什么？"
- "如果今天没有 [现有工具]，你会怎么应付？"
- "上一次你为了完成 [Y] 加班到很晚，是因为什么？"
- "你身边有没有人也在做 [X]？他们怎么做？"

❌ 坏问题：
- "你觉得这个功能有用吗？"  ← 抽象、引导性
- "你愿意付钱吗？"  ← 假设、无承诺
- "如果我们做了这个，你会用吗？"  ← 未来时、零成本表态
```

**B. 验证假设的好问题**

```plain
✅ 好问题：
- "上个月有几次你 [发生 X 情况]？"  ← 频次确认
- "上次发生 [X] 后，你做了什么？"  ← 行为追溯
- "为了解决 [X]，你试过哪些方法？为什么没继续用？"  ← 替代品调研
- "如果我说有个工具能 [Y]，你的第一反应是哪些场景能用？"  ← 让对方自己映射场景

❌ 坏问题：
- "你觉得 X 是不是问题？"  ← 引导
- "如果有 Y 你会用吗？"  ← 未来时
- "你最大的痛点是什么？"  ← 太宽泛、对方编一个
```

**C. 索要"承诺货币"的好问题**

```plain
✅ 好问题（真的能筛出真意愿）：
- "你愿意花 30 分钟和我们的产品经理深聊吗？"  ← 时间承诺
- "我们做了一个原型，你愿意试用一周给我们反馈吗？"  ← 行为承诺
- "你能介绍 1-2 个和你一样有这个痛点的同事给我们吗？"  ← 社交承诺
- "如果产品做出来收 50 元/月，你愿意预付一个月吗？"  ← 金钱承诺（最强）

❌ 坏问题：
- "你以后会用吗？"
- "你愿意付钱吗？"
- "你觉得多少钱合适？"
```

> **核心心法**：一个真有需求的人会主动给你**时间、行为、社交、金钱**这四种承诺中至少一种。**只给"嘴上承诺"的人是 100% 的伪用户**——他们的意图调研报告会让你做出错误的产品。
>

### 4.3 三类数据：一手 / 二手 / 反信号
做需求挖掘时，数据来源决定了结论的可信度。**不同来源的数据有完全不同的可信度权重**，混用会导致严重误判。

| 数据类型 | 来源 | 可信度 | 用法 |
| :---: | :---: | :---: | :---: |
| **一手数据** | 自己做的访谈、用户行为日志、自己的实测 | 最高 | 决策依据 |
| **二手数据** | 调研报告、行业白皮书、内部数据团队 | 中 | 验证 / 对照，不单独决策 |
| **反信号数据** | 用户没说的话、用户没用的功能、用户流失原因 | 极高（但容易被忽略） | 必看 |


**反信号数据：项目挖掘里最值钱的"暗物质"**

工程师习惯于看"用户做了什么"，但真正的洞察往往在"用户**没**做什么"里：

| 反信号 | 含义 | 例子 |
| --- | --- | --- |
| **用户没说的话** | 调研里完全没人提的话题 | 调研 50 个员工，没有一个人主动提"AI 周报助手"——说明这不是高优痛点 |
| **用户没点的功能** | 上线后日活率极低的功能 | "智能润色"按钮 30 天点击率 < 1%——说明这不是兴奋点 |
| **用户没回的反馈** | 调查问卷的开放题空白率 | 开放题"你最希望增加什么功能"，60% 的人留空——说明用户不缺这个 |
| **用户流失的原因** | 流失用户的访谈 | 流失用户都说"不是不好用，是用不上"——说明场景命中率有问题 |
| **没付费的"愿意付费用户"** | 嘴上说要付费、上线后 0 付费 | 调研 60% 说愿付费、实际付费率 < 5%——典型的伪需求 |


**关键技巧：流失访谈的 ROI 是新增访谈的 5 倍**

绝大多数公司只访谈"留存用户"和"潜在用户"，不访谈流失用户。**但流失用户给的信息密度远高于留存用户**——他们有真实的"放弃理由"，不需要客气。

落地动作：上线后 90 天，**强制每周访谈 3 个流失用户**，问三个问题：
1. 你最后一次用我们产品是什么时候？为什么打开的？
2. 之后为什么没再用？
3. 现在你怎么解决那个问题？

**这三个问题答完，比任何调研都更接近"真实用户需求"**。

### 4.4 内部访谈 vs 外部访谈：信号差异
B 端产品，尤其是大厂内部工具（如 AI Agent / 内部中台），访谈对象往往是**自己公司的同事**。这里有一个巨大的陷阱：**内部用户给的信号严重失真**。

| 信号差异 | 内部用户 | 外部用户 |
| --- | --- | --- |
| **诚实度** | 低（怕得罪、怕被打绩效） | 高（无利害关系） |
| **配合度** | 高（被领导拉来访谈） | 低（需要给报酬） |
| **代表性** | 低（自己公司文化的样本） | 高（更接近真实市场） |
| **行为真实度** | 低（被强制使用，会"配合演出"） | 高（自由选择） |


**内部访谈的"打折机制"（个人框架）**

| 内部用户来源 | 信号打折比例 |
| --- | --- |
| **同部门同事**（你的人） | 打 5 折——他们不愿伤你 |
| **跨部门有合作关系的人** | 打 7 折——客气 |
| **跨部门无合作关系的人** | 打 8 折——较真实 |
| **来自老板要求"必须用"的部门** | 打 3 折——他们的"好评"几乎是政治表态 |


**实操：让内部访谈逼近真实**

1. **匿名化 + 第三方**：让 PMO 或外部咨询机构做访谈、保证匿名——内部用户更敢说真话
2. **行为数据 > 嘴上反馈**：日活、留存、功能点击是真实的，问卷调查不是
3. **"反向求证"**：访谈后追踪具体行为——他说"会推荐给同事"，看 30 天后他真的推荐了几个
4. **找"逆耳忠言型"用户**：每个公司都有几个"敢说真话"的人——主动去找他们，他们一句话顶 10 个客气的反馈

### 4.5 访谈数量与"饱和度"判断
最常见的问题：**到底要访谈多少人才够？**

学术界的答案：5-12 人能识别 80% 的主要主题（Nielsen 1993 经典研究）。但工程实践有更简单的判断方法——**信号饱和度**。

**饱和度判断标准（个人提炼）**

```plain
访谈第 1 人：信号 100% 是新的
访谈第 2 人：信号 80% 新 + 20% 重复第 1 人
...
访谈第 N 人：信号 < 20% 新 + > 80% 重复

→ 当连续 3 个访谈对象提供的"新信号"占比都 < 20% 时
   → 该群体的需求挖掘已饱和，停止访谈
```

**典型的群体饱和数（经验值）**

| 用户群体 | 饱和访谈数 |
| --- | --- |
| **同质性高的群体**（如某个具体岗位） | 5-8 人 |
| **中等同质性**（如"研发岗"） | 10-15 人 |
| **异质性高**（如"全公司员工"） | 20-30 人，且必须分层 |


**关键技巧：如果 5 个人就饱和，说明你的用户群体已经够窄、值得做**

很多人觉得"才访谈 5 个人，样本太少"——这是用学术标准评判工程问题。**工程上，5 个人迅速饱和 = 这个用户群体的需求高度集中 = 产品定位精准**。如果访谈了 30 个人还在出新信号，说明用户群体没切对——**先窄化再调研**。

**饱和后的下一步：定量验证**

定性访谈饱和 → 写出 5-10 条"假设"（比如"研发岗每周写周报平均花 2 小时"）→ 用问卷 / 数据 / A/B 测试做定量验证。**定性挖掘问题、定量验证规模——这是需求挖掘的标准动作链**。

---

## 第五部分：需求验证——立项前必做的"价值证明"
访谈做完了、Persona 画好了、JTBD 也理清了——**但这都还停留在"我相信"的层面**。需求挖掘的最后一公里是：**用最低成本、最快速度，把"我相信"变成"已被证伪 / 已被证实"**。

这一部分的核心命题：**任何一个项目立项之前，必须明确"哪些是已知事实、哪些是假设、假设错了什么时候停"**。**没有这三件事的项目立项 = 用真金白银做没有止损线的赌博**。

### 5.1 Working Backwards / PR-FAQ——Amazon 的反向叙事
**Working Backwards** 是 Amazon 内部 20 多年的产品立项方法，核心是：**先写产品上线后的"新闻稿"，再写"FAQ"，最后才写技术方案**。**写不出"打动人的新闻稿"= 这个产品没价值**。

**PR-FAQ 模板（适配企业项目）**

```plain
# 内部新闻稿（Press Release）
（假设是上线后写给全公司的）

【标题】（一句话能说清楚价值）
公司发布"AI 周报助手"，员工平均节省 70% 周报时间

【副标题】（说清楚谁在用、用得怎么样）
首批 1000 名研发员工试用 3 个月，
75% 用户表示"不再讨厌写周报"

【导语】（最重要的 3 句话——卖点）
2026 年 12 月，公司正式发布面向研发岗员工的 AI 周报助手。
该工具自动从 Jira / IM / 文档中聚合信息，
让员工 30 分钟内完成原本 2 小时的周报工作。

【问题描述】（用户当前的困境）
公司研发员工平均每周花 2 小时写工作周报。
80% 员工调研显示"周报是讨厌但又不能不做的事"。
更糟的是，跨系统手动汇总信息容易遗漏关键事项，
导致月度复盘时被老板点名。

【解决方案】（产品如何解决）
AI 周报助手通过深度集成公司 Jira / IM / 文档系统，
一键聚合本周所有工作记录，
通过大模型自动生成初稿并保留员工编辑权。

【关键人物声言】
"以前周五下午都是焦虑的开始，现在 30 分钟搞定，
 还能 5 点准时下班接孩子。"
                          —— 张小李，研发工程师

"作为部门 leader，我现在能看到团队真实的工作脉络，
 而不是一份份'看起来都很努力'的模板周报。"
                          —— 王 TL，研发经理

【How to Get Started】
登录公司 IM，搜索"周报助手"机器人，开箱即用。

# FAQ（必备 10-20 条）

Q1. 这个工具会自动发送周报吗？
A: 不会。所有周报必须用户主动点击"发送"。AI 只生成草稿。

Q2. 老板会不会发现周报是 AI 写的？
A: 工具会保留员工的写作风格指纹，并强制要求至少 30% 的人工编辑。

Q3. 数据安全怎么保证？
A: 所有数据在公司内网处理，不调用任何外部 API。已通过安全合规评审。

Q4. 没有 Jira 的非研发岗能用吗？
A: v1 仅支持研发岗。其他岗位见 Future Roadmap。

Q5. 如果 AI 总结错了，怎么办？
A: 每条总结旁边有"原始来源"链接，用户可一键溯源。

...（继续 10-20 条预设的反对意见 / 疑虑 / 边界问题）

# 不做（Out of Scope）—— 显式声明
- 不做多轮对话
- 不做替代人工判断
- 不做销售 / HR 等其他岗位
- 不做替代周报这个流程本身
```

**为什么 Working Backwards 有效**

1. **PR 强迫"用户视角"——写不出来 = 没价值**：如果你写不出一段让目标用户激动的话，这个产品本身就没有打动人的价值。**很多项目卡在 PR 这一步——这正是它的筛选作用**。
2. **FAQ 强迫"对抗思维"——回答不出 = 没想清楚**：FAQ 部分 20 条 Q&A，相当于让你提前回答所有可能的反对意见。一个回答不出的问题，就是项目的一个潜在炸弹。
3. **Out of Scope 强迫"边界感"——划不清 = 后期被灌死**：写完 PR-FAQ，必须在最后明确列出"这个项目不做什么"。这是项目最重要的"防御工事"。

**与第二章 6-Pager 的关系**：6-Pager 适合工程方案，PR-FAQ 适合产品立项——前者讲怎么做，后者讲为什么做。**两个工具配合：先 PR-FAQ 立项，确认值得做；再 6-Pager 设计，确认能做到**。

### 5.2 Pre-mortem——反向推演项目失败
工程行为方法论 §3.6 已经详细介绍过 Pre-mortem 的具体做法。这里只补充**需求阶段特有的 Pre-mortem 用法**。

**需求阶段的 Pre-mortem 提问框架**

```plain
开场白：
"假设 6 个月后这个项目上线了，但失败了——
 上线 3 个月日活 < 5%，老板宣布暂停项目。
 请每个人独立写下：你认为最可能的失败原因是什么？"

需求阶段 Pre-mortem 必问的 5 类问题：
1. 用户根本没用 / 用一次就放弃了——为什么？
2. 用户用了，但反馈"还不如手动"——哪里不达预期？
3. 老板看了 demo 不满意——他真正想要的是什么我们没给？
4. 内部有竞争团队做了类似的，把我们干掉了——他们的优势是什么？
5. 上线后才发现一个我们没考虑的合规 / 安全 / 政策问题——是什么？
```

**关键技巧：把 Pre-mortem 答案写进 Risk Register**

Pre-mortem 的输出不是"开完就完"——每一条预设的失败原因，必须转化成 Risk Register 里的一条风险，并配上"早期信号 + 缓解措施 + Owner"。**Pre-mortem 的真正价值在于把"未来的后悔"变成"今天的行动项"**。

### 5.3 MVP / 烟雾测试 / 假门测试——最低成本的验证
**MVP（Minimum Viable Product）** 由 Eric Ries 在 2011 年《The Lean Startup》系统化（早期实践来自 Steve Blank 2005 年《The Four Steps to the Epiphany》）。核心命题：**用最低成本做出一个能验证关键假设的产品，而不是憋大招做一个完美产品**。

**MVP 的五种形态（按成本递增）**

| 形态 | 描述 | 成本 | 适用阶段 |
| :---: | :---: | :---: | :---: |
| **烟雾测试** | 落地页 + "感兴趣点这里" 按钮 | 极低（1-3 天） | 早期：验证用户是否点击 |
| **假门测试** | 看起来是产品的入口，点进去显示"敬请期待 + 留邮箱" | 低（3-7 天） | 中期：验证用户是否愿意留下联系方式 |
| **绿野仙踪 MVP** | 前端是产品、后端是人工 | 中（1-2 周） | 验证场景：用户体验对了，但还不值得自动化 |
| **核心功能 MVP** | 端到端最简版本，但每个环节都是真实的 | 较高（4-8 周） | 验证产品价值 |
| **完整 MVP** | 满足核心场景的最完整版本 | 高（2-3 月） | 准备规模化 |


**用 MVP 拆"AI 周报助手"**

```plain
阶段 1：烟雾测试（成本：1 天）
做法：
- 在公司 IM 群发一条消息："我们在做一个 AI 周报工具，
  能 5 分钟搞定周报，感兴趣的朋友扫码登记体验"
- 提供一个简单表单，收集愿意试用的人

验证假设：员工有"想用 AI 写周报" 的诉求
信号判断：
- 200 人群里，有 30 人扫码 → 强信号（15%）
- 200 人群里，有 5 人扫码 → 弱信号（2.5%），考虑放弃

阶段 2：假门测试（成本：3 天）
做法：
- 做一个简单的 Web 页面，看起来是周报工具，输入项目信息后
  点击"生成周报"
- 实际后端是个 Google 表单，收集用户输入
- 用户提交后页面显示"内测中，工程师会在 24h 内人工生成你的周报"

验证假设：员工愿意把工作信息输入给 AI 工具
信号判断：
- 30 个登记用户里，有 20 个真正提交了内容 → 强信号
- 大量人在"输入信息"那一步流失 → 暴露关键障碍（隐私？复杂度？）

阶段 3：绿野仙踪 MVP（成本：2 周）
做法：
- 工程师配合 5-10 个用户，每周五人工帮他们把 Jira / IM 的内容
  汇总成周报草稿
- 用 ChatGPT / 大模型辅助但不自动化
- 持续 4 周

验证假设：用户拿到生成的周报后，真的会改一改就发出去
信号判断：
- 用户拿到草稿后改动 < 30% → 强信号（产品价值高）
- 用户拿到草稿后改动 > 70% → 弱信号（不如自己写）
- 用户拿到草稿后压根不发 → 否定信号（场景错了）

阶段 4：核心功能 MVP（成本：8 周）
做法：
- 真正实现 Jira / IM 自动拉取 + 大模型生成 + IM 入口
- 限定 100 个种子用户

验证假设：自动化版本的留存和满意度
```

**MVP 的"杀掉条件"必须前置定义**

每个阶段开始前，必须明确：**什么样的数据 = 通过、什么样的数据 = 杀掉**。否则数据出来了，所有人都会用"再观察一下"拖延决策。

### 5.4 假设—验证—决策循环
把上面所有工具串起来，需求验证的标准工作流是一个**假设—验证—决策**的循环（这是 Lean Startup "Build-Measure-Learn" 循环的需求阶段简化版）。

**循环结构**

```plain
1. 列出关键假设
   ↓
2. 按"假设错了的代价" 排序
   ↓
3. 选最高代价的假设，设计最低成本的验证
   ↓
4. 执行验证（访谈 / 数据 / MVP）
   ↓
5. 看数据 → 决策（继续 / 调整 / 停止）
   ↓
6. 回到第 1 步，更新假设清单
```

**用循环拆"AI 周报助手"立项**

```plain
关键假设清单（按代价排序）：

H1. 员工真的觉得周报是痛点（不是老板觉得的）
    → 错了的代价：整个项目方向错，几千万投入打水漂
    → 最低成本验证：访谈 8 个员工 + 烟雾测试（5 天，1 万成本）

H2. 员工愿意把工作内容交给 AI 工具
    → 错了的代价：技术做出来没人用
    → 最低成本验证：假门测试 + 隐私顾虑访谈（1 周，3 万成本）

H3. 大模型生成的周报质量足够好（用户改 < 30%）
    → 错了的代价：技术不达标，需要重新选型
    → 最低成本验证：绿野仙踪 MVP（4 周，10 万成本）

H4. 自动化版本能保持质量
    → 错了的代价：从手动到自动化的工程量浪费
    → 最低成本验证：100 用户 MVP（8 周，30 万成本）

H5. 该项目对公司"AI 战略对外宣传"有价值
    → 错了的代价：项目失去战略资源
    → 最低成本验证：和 PR / 战略部门 1on1 访谈（1 周，0 成本）

总成本累计：H1+H2+H3+H4+H5 ≈ 44 万 + 14 周（串行执行，理想节奏）
含 buffer 上限：约 50 万 + 16 周（考虑访谈排期、合规审查等不可控因素）
任何一步失败 → 停止后续步骤，止损
全部通过 → 才进入正式立项
```

**关键洞察**：**用 50 万 / 16 周 验证一个 5000 万 / 2 年的项目，是任何理性投入应该做的事**。但大厂里，这一步往往被跳过——因为"老板等不及"、"友商已经发布"、"我们 PPT 已经讲出去了"。**敢顶住这种压力做完验证再立项的人，就是真正的项目总负责人**。

### 5.5 杀掉条件（Kill Criteria）——立项时必须明确
**Kill Criteria** 不是单一框架，是综合 Lean Startup（pivot/persevere）、Stage-Gate（Cooper 1986）、OKR 季度复盘等实践，落到工程语境下的实践版本。核心命题：**任何项目立项时，必须显式声明"什么情况下我们会主动放弃"**。

**为什么必须在立项时就定**

立项时所有人都乐观，所以杀掉条件最容易被诚实地写下来。**项目跑了一半，所有人都已经投入了感情和资源（沉没成本），那时候谈"杀掉"会被视为对团队的背叛**——所以必须**前置**。

**杀掉条件三层模板（个人框架）**

```plain
# 项目：AI 周报助手 - Kill Criteria

## 第一层：技术不可行性（Hard Kill）
- 大模型生成周报的"用户改动率"持续 > 70%（连续 4 周）
- 端到端延迟 > 60 秒，且用户反馈"等不及"
- 数据隐私 / 合规审查未通过

→ 触发 = 立刻停止，不再投入

## 第二层：用户不接受（Soft Kill）
- 上线 3 个月，目标用户日活 < 30%
- NPS（净推荐值）< 30
- 用户访谈中 50%+ 反馈"不如手动"

→ 触发 = 暂停 2 周分析，要么大改方向、要么停项目

## 第三层：组织不再支持（Strategic Kill）
- 老板对外不再讲"AI 落地"故事
- 公司战略转向，AI 投入收紧
- 关键合作团队（IT / 数据 / 安全）撤回支持

→ 触发 = 走"优雅退出"流程（详见工程行为方法论 §3.8）
```

**杀掉条件的"自我反思测试"**

写完杀掉条件后，问自己一个问题：**"这些条件里，哪一条今天就已经接近触发了？"**

如果答案是"没有，都还很远"——可能你定的条件太宽松（避免了真正的止损线）。
如果答案是"已经有 1-2 条接近触发了"——立项时就要重新评估"是否真该立这个项目"。

**关键洞察**：**好的 Kill Criteria 不是"项目跑不下去时的逃生通道"，而是"项目跑得下去但已经偏航时的清醒按钮"**。**项目失败 90% 不是因为没办法跑下去，是因为"偏航了但所有人都不愿承认"**——杀掉条件就是用来对抗这种集体不愿承认的。

## 第六部分：需求优先级——有限资源下的取舍
需求挖掘的产物从来不是一个，而是一堆——访谈结束、Pre-mortem 完成、MVP 跑过几轮，桌上往往有 30+ 个候选场景 / 候选功能。**项目总负责人的真本事是从这堆里挑出"必做的 5 个 + 不做的 25 个"**。这一部分讲清楚怎么做这件事。

### 6.1 RICE 评分——量化的优先级
**RICE** 由 Intercom 团队（Sean McBride）在 2016 年通过 Intercom 博客对外公开。核心命题：**用 4 个可估算的指标给每个需求打分，按分数排序就是优先级**。

**RICE 公式**

```plain
RICE Score = (Reach × Impact × Confidence) / Effort

Reach（触达）：每月 / 每季度会接触这个功能的用户数
Impact（影响）：单用户单次的影响（0.25 / 0.5 / 1 / 2 / 3 五档）
Confidence（信心）：这些数字的可信度（50% / 80% / 100%）
Effort（投入）：人月成本
```

**实操示例**

```plain
项目：AI 周报助手 - 候选功能优先级

功能                 Reach    Impact   Confidence   Effort   RICE
────────────────────────────────────────────────────────────────
1. Jira 一键拉取     1000/月   3        100%         2 人月    1500
2. IM 入口           1000/月   2        100%         1 人月    2000
3. 智能润色          1000/月   1        80%          3 人月    266
4. 多模板切换        500/月    1        100%         1 人月    500
5. 老板偏好适配      1000/月   2        50%          4 人月    250
6. 移动端           300/月    2        80%          4 人月    120

→ 优先级：2 → 1 → 4 → 3 → 5 → 6
```

**RICE 的三个常见误区**

| 误区 | 表现 | 修正 |
| --- | --- | --- |
| **Reach 算成总用户数** | 把"全公司员工"都算成 Reach | Reach 是"会用这个功能的人"——不是所有员工都会用所有功能 |
| **Impact 凭感觉打分** | 团队所有人讨论后投票 | 给 Impact 配具体定义：3 = 用户会主动推荐、2 = 用户会反复用、1 = 用户偶尔用、0.5 = 用户用一次就忘 |
| **Confidence 总是 100%** | 大家都不愿"显得不专业" | Confidence < 80% 是常态——只有有数据支撑的才打 100% |


### 6.2 Kano 落地与权重分配
§3.4 介绍了 Kano 模型，这里讲怎么把 Kano 结果变成优先级。

**Kano 与优先级的映射**

| Kano 类型 | 优先级 | 资源分配 |
| :---: | :---: | :---: |
| **基础需求**（Must-be） | 必做、做到 80 分就够 | 30% 资源 |
| **期望需求**（Performance） | 持续投入、做到行业前列 | 50% 资源 |
| **兴奋需求**（Excitement） | 选 1-2 个深度做、不撒胡椒面 | 20% 资源 |


**关键洞察：兴奋需求会随时间退化**

Kano 模型的隐含动力学：**今天的兴奋需求，3 年后变期望需求，5 年后变基础需求**。

举例 1（消费电子）：智能手机的"指纹解锁"，2013 年是兴奋需求（iPhone 5s 引入引发惊艳），2017 年变期望需求（用户期望都有），2020 年变基础需求（没有就被骂）。

举例 2（AI 时代，更近）：**LLM 内容自动总结 / 摘要**——2023 年是兴奋需求（GPT-4 出现引发惊艳），2025 年变期望需求（每个文档/IM/邮件工具都内置），**2026 年正在变基础需求**（没有反而会被吐槽"这工具怎么没有 AI"）。**这个退化节奏明显比指纹解锁那一代更快——AI 时代的 Kano 退化周期已从 7 年压缩到 3 年左右**。

**对策**：
1. **基础需求一定要满足，但不要过度投入**——它的天花板是"不被骂"
2. **期望需求需要长期投入**——它是用户满意度的主要驱动力
3. **兴奋需求要持续创造新的**——上一代的兴奋点会贬值

### 6.3 North Star Metric——用一个指标统一团队
**North Star Metric（北极星指标）** 由 Sean Ellis 在 2010 年代推广（"增长黑客"运动的核心方法论）。核心命题：**整个团队应该用一个最重要的指标统一方向，所有功能开发都围绕"这个指标涨不涨"**。

**好的北极星指标的特征**

| 特征 | 说明 |
| --- | --- |
| **直接反映用户价值** | 不是"DAU"（用户来了不一定有价值），而是"完成核心 Job 的次数" |
| **领先于商业指标** | 北极星涨 → 营收会涨；不是反过来 |
| **团队能影响** | 团队的工作能直接推动这个指标 |
| **足够具体** | 不是"用户满意度"——太虚 |


**典型公司的北极星指标**

| 公司 | 北极星指标 |
| --- | --- |
| **Facebook** | 月活用户（早期）→ 7 天内加 10 个好友的用户（细化）|
| **Airbnb** | 预订夜数 |
| **Spotify** | 总收听时长 |
| **Slack** | 发了 2000 条消息的团队数 |
| **Whatsapp** | 发送的消息数 |


**用北极星指标改"AI 周报助手"**

```plain
❌ 不好的指标：
- DAU（用户打开了不一定用）
- 累计用户数（用过一次就算）
- 满意度评分（自报数据，失真）

✅ 好的北极星指标候选：
A. 每周完成周报的用户数（直接反映核心价值）
B. 用户用工具完成周报的"采纳率"
   = 工具生成的周报内容被保留的比例
C. 月度留存率（连续 4 周都用）

→ 选 A 作为北极星：每周完成周报的用户数
   - 直接反映 Job 完成（写周报）
   - 团队能影响（产品好用 → 用的人多）
   - 领先指标（用的人多 → 满意度涨 → 老板满意）
```

**关键洞察**：**很多项目失败的根因是"指标错位"——团队天天在优化的指标，和老板真正关心的指标，和用户真正受益的指标，是三个完全不同的东西**。北极星指标的真正价值是**强迫这三者对齐到同一个指标上**。

### 6.4 OKR 对齐——把需求变成跨团队的"共享内存"
工程行为方法论 §7.4 已经讲过 OKR 对齐对协作的价值。这里补充**需求阶段特有的 OKR 用法**。

**需求—OKR 对齐三步法（个人框架）**

```plain
Step 1：把核心需求映射到一个北极星指标
   → "AI 周报助手" → "每周完成周报的用户数"

Step 2：把北极星指标拆成可分配的 KR
   → KR1：研发岗周报采纳率达到 60%（产品负责）
   → KR2：周报生成端到端延迟 < 30 秒（工程负责）
   → KR3：用户改动率 < 30%（算法负责）
   → KR4：合规审查通过、零数据泄露事件（安全负责）

Step 3：找每个合作团队 OKR 的"重叠点"
   → 数据团队 OKR 里有"建设员工行为数据中台"
     ← 你的项目能给他们提供"周报场景"作为标杆案例
   → 安全团队 OKR 里有"AI 应用合规框架落地"
     ← 你的项目能成为"首个走完合规流程的 AI 应用"
   → 给每个合作团队都找到"重叠点"，写进自己的 OKR
```

**关键洞察**：**没有 OKR 重叠点的合作请求 = 必然被排到 backlog 末尾**。这不是合作方"不配合"，是组织规则。**反向利用：找不到重叠点的项目，本身就是一个红色信号——说明这个项目对组织缺乏吸引力，即使做出来也得不到长期支持**。

### 6.5 路线图设计："三层折叠"
有了优先级清单和 OKR 对齐，最后一步是把它们排成一份**可对外讲、可被老板理解、可被团队执行**的路线图。**好的路线图既要给老板信心，又要给团队回旋余地**。

**三层折叠路线图（个人框架）**

```plain
# AI 周报助手 - 路线图 v1（2026 H2）

## 第一层：对外承诺层（Now）
（已经向老板和合作方明确承诺、必须完成）
- ✅ M1：MVP 上线（2026/8/1）
- ✅ M2：1000 种子用户接入（2026/9/15）
- ✅ M3：研发岗全量覆盖（2026/12/31）

→ 这一层的特点：
  - 颗粒度细到月
  - 每个里程碑有验收标准
  - 失败 = 项目失败
  - 不轻易调整

## 第二层：方向探索层（Next）
（已经规划但保留调整空间）
- 🔄 多模板支持（取决于种子用户反馈）
- 🔄 移动端入口（取决于 PC 端日活）
- 🔄 老板偏好适配（取决于产品成熟度）

→ 这一层的特点：
  - 颗粒度到季度
  - 验收标准灵活
  - 调整需要团队评审、不需要老板审批

## 第三层：愿景层（Later）
（更远期的可能性，不承诺）
- 💭 销售 / 客服 / HR 岗位扩展
- 💭 多语言支持
- 💭 跨组织流程自动化（更像 Agent 而非工具）

→ 这一层的特点：
  - 不写日期、不承诺范围
  - 用来对外讲愿景、对内统一长期方向
  - 调整频率：每年 review 一次
```

**为什么"三层折叠"有效**

1. **对外有承诺、对内有弹性**：第一层让老板和合作方放心，第二层让团队有迭代空间，第三层让所有人有共同愿景
2. **避免"过度承诺"**：很多项目把所有功能都写进路线图、所有日期都给死——结果第一次延期就开始信任崩盘
3. **避免"无路线图"**：另一极端是只写"持续探索"——团队没有方向感，老板没有耐心

**关键技巧：路线图的"层间转移"必须有显式仪式**

每季度做一次路线图 review：
- 第二层的某个功能"成熟"了 → 加日期 + 验收标准 → 升到第一层
- 第三层的某个想法"被验证" → 加资源 + 范围 → 升到第二层
- 第一层的某个里程碑"被证伪" → 不要悄悄删除 → 公开 review、解释原因、降级或杀掉

**层间转移的公开化** = 项目长期信用的来源。**最忌讳的是"路线图悄悄变了"**——这会让所有人都对你的承诺打折。

## 第七部分：中国大厂特色——战略型 vs 跟风型需求识别
前 6 部分讲的是"通用"方法论。但中国大厂有一个特别的需求来源——**老板需求**。这一类需求往往伴随着"战略焦虑 + 友商压力 + 对外宣传 + 上市预期 + KPI 倒推"等复合因素，**用通用方法论硬接，要么得罪老板、要么烧错钱**。这一部分专门讲怎么处理这类需求。

### 7.1 老板需求"翻译"——从"做 AI Agent"到真正的业务问题
老板抛出一句"我们要做 AI Agent"，下面理解的人有 3 种典型反应：

| 反应类型 | 典型行为 | 后果 |
| :---: | :---: | :---: |
| **执行型**（最常见） | 立刻开始招人 / 选型 / 立项，3 个月内出 demo | 90% 概率烧钱后烂尾——因为老板自己也没想清楚 |
| **回避型** | 觉得老板"不懂技术、瞎指挥"，消极抵抗 | 短期没事但失去战略机会 + 老板觉得你"格局小" |
| **翻译型**（高级别 leader） | 主动拆解老板的"原始诉求 → 真实诉求 → 可工程化的问题" | 大概率成功，并且是项目总负责人的真正价值 |


**老板需求翻译四步法（个人框架）**

```plain
Step 1：还原老板的原始触发
   问自己：老板是在什么场合说这话的？
   - 董事会？→ 大概率是"对外宣传焦虑"
   - 产品评审？→ 大概率是"产品差异化焦虑"
   - 1on1？→ 大概率是真有具体痛点要解决
   - 看了某个分享？→ 大概率是"被启发的临时想法"

Step 2：识别"真实诉求层"（往往隐藏在表象之下）
   表面诉求："做一个 AI Agent"
   可能的真实诉求：
   ✓ 公司年报需要"AI 落地案例"——对外宣传诉求
   ✓ 老板担心团队不够"AI Native"——组织能力诉求
   ✓ 友商发布了相似产品——竞争防御诉求
   ✓ 看到了某个具体业务低效——降本增效诉求
   ✓ 某个 VC / 客户 / 上司点过名——外部压力诉求

Step 3：回拨给老板做"诉求确认"
   不要直接问老板"您到底想要什么"——会让老板觉得你不专业。
   用 PR-FAQ 的方式确认：
   "老板，我准备了一个三页的 PR-FAQ，
    里面假设了 3 种可能的产品方向。
    您能花 15 分钟，帮我看下哪个方向最符合您的预期吗？"

   每个方向给一个"新闻稿 + 投入估算 + 时间线"
   让老板在具体方向之间选，而不是从零编

Step 4：把"老板确认的方向" 翻译成可工程化的需求
   一旦老板确认了方向，就可以走 §3-§6 的标准方法论
   （JTBD / Pre-mortem / MVP / Kill Criteria）
```

**实战示例：老板说"做 AI Agent"**

```plain
【Step 1：还原触发】
老板是在和某 VC 吃完饭后说的——VC 提到"现在不做 Agent 的公司都过时了"
→ 大概率是"竞争防御 + 对外宣传"诉求，不是真有具体业务痛点

【Step 2：识别真实诉求】
列出 5 种可能：
A. 真的想用 Agent 解决某个具体业务问题
B. 想要一个"标杆案例"对外讲故事
C. 想让公司技术团队"AI 化" 转型
D. 单纯想"不掉队"，没有具体目标
E. 想给某个具体场景（如客服）降本

【Step 3：PR-FAQ 回拨】
准备 3 个 PR-FAQ：
1. "客服 Agent v1"——降本场景，6 个月、800 万
2. "员工 AI 助手"——内部赋能，9 个月、1500 万
3. "对外开放 Agent 平台"——商业化场景，18 个月、5000 万

每个 PR-FAQ 给老板看：
- 预期对外宣传效果（标杆 / 案例 / 媒体覆盖）
- 资源投入
- 风险和回报

【老板的真实回应】
老板看完后说："其实我最想要的是 #2，
让公司每个员工都觉得'我们公司在 AI 上是领先的'。
但你们也别忘了 #1 的客户价值。"

→ 老板的真实诉求 = "组织能力诉求 + 一点对外宣传"
→ 可工程化的项目 = 内部 AI 助手平台，先做"周报场景"作为灯塔
→ 同时低成本支持 #1 客服场景（一两个外包项目）

【Step 4：转入标准方法论】
现在可以走 JTBD → Persona → MVP → Kill Criteria 全流程
```

**关键洞察**：**老板需求的"翻译"比"执行"重要 10 倍**。一个能把"做 AI Agent"翻译成"内部 AI 助手平台 + 周报灯塔场景"的 leader，省下来的钱可能是几千万。**这才是项目总负责人真正的杠杆**——不是带多少人、不是写多少代码，是把模糊的战略压力转化成清晰的工程命题。

### 7.2 跟风型需求的 5 个识别信号
除了老板需求，还有一类常见的"危险需求"——跟风型。这一类需求的特征是**"看起来很有道理，但拆开看一切都是别人的故事"**。**识别它的关键不在于内容，而在于来源和论证方式**。

**5 个识别信号（个人框架）**

| 信号 | 表现 | 解读 |
| --- | --- | --- |
| **① "友商都在做"是核心论据** | 立项 PPT 第一页就放友商对标 | 你在抄答案，不是在解题 |
| **② 完全没有"用户访谈" 数据** | 全部数据是行业报告 / 二手数据 | 没有人和真实用户聊过 |
| **③ ROI 论证基于"如果"** | "如果用户都用，能省 X 千万" | 关键变量是"如果"，没有验证 |
| **④ 时间线异常激进** | 6 个月内必须上线，原因是"不能比友商晚" | 时间不是来自需求，是来自焦虑 |
| **⑤ "杀掉条件"答不上来** | 问"什么情况下我们会停止" → 沉默 | 没有止损线 = 这是个赌博，不是项目 |


**5 个信号 ≥ 3 个 = 高度跟风嫌疑**

如果一个项目命中 ≥ 3 个信号，**你必须站出来质疑它，哪怕得罪老板**。在大厂里，这种质疑短期会被视为"消极"，长期会被视为"靠谱"——**因为最终烂尾的项目，所有人都会去翻"当时谁说过反对意见"**。

**反过来：战略型需求的 5 个特征**

为了对照，下面是真正值得做的"战略型需求"的特征：

| 特征 | 表现 |
| --- | --- |
| **① 有具体的用户和痛点** | 能说清楚"X 岗位的 Y 任务，每周浪费 Z 小时" |
| **② 有一手用户数据** | 已经访谈过 10+ 真实用户、有访谈纪要 |
| **③ ROI 论证基于"已观察"** | "现在已经有 N 个用户在自己用 Z 工具凑合，他们愿意付 X 元" |
| **④ 时间线来自需求紧迫性** | "用户的 Y 痛点每天都在发生，越早解决越好" |
| **⑤ 有明确的杀掉条件** | "如果上线 3 个月日活 < N，我们停" |


### 7.3 给老板的"需求体检报告"
当你识别出一个项目有"跟风型需求"嫌疑时，**直接说"我反对" 是最低效的姿态**——会被视为消极、不配合。**最高级的反对是"用专业的方式让老板自己看到问题"**。

**需求体检报告模板（个人框架）**

```plain
# 需求体检报告：XX 项目

## 体检结论（一句话）
该项目当前处于 [跟风型 / 半跟风型 / 战略型] 状态，
建议在投入正式资源前，先用 [X 周 / Y 万] 完成补强体检。

## 体检维度（6 项）
1. 用户和场景清晰度：[A / B / C / D]
   - A：有具体 Persona + 一手访谈数据
   - B：有用户群体定义但无具体 Persona
   - C：用户群体描述模糊
   - D：完全没有用户层面信息

2. 真实痛点支撑度：[A / B / C / D]
3. 一手数据完整度：[A / B / C / D]
4. ROI 论证可信度：[A / B / C / D]
5. 杀掉条件明确度：[A / B / C / D]
6. 组织安全性（既得利益 / 数据归属 / 流程归属 / 合规 / 战略红线）：[A / B / C / D]

## 体检总分
当前得分（示例）：B / C / D / D / D / B
建议：补强 #2-5 后再立项

## 补强方案（最低成本）
- 4 周访谈 30 个目标用户（10 万）
- 2 周烟雾测试 + 假门测试（3 万）
- 1 周 Pre-mortem + Kill Criteria 制定（0 万）
- 总计：7 周 + 13 万 → 决定是否值得继续投入

## 不补强的风险
- 项目按当前方向直接投入：约 5000 万 + 12 个月
- 失败概率：估算 70%（基于体检维度）
- 失败后的隐性代价：团队信用、跨部门关系、老板对你的判断
```

**为什么这个报告有效**

1. **它不否定项目**——只是说"现在还不能拍板，需要补强"
2. **它给老板"出口"**——让老板觉得"是我在做明智决策"，不是"被下属反对"
3. **它有具体的补强方案 + 成本**——把"反对"变成"投资 13 万换一个明智决策"
4. **它显式提示风险**——让老板看到"不补强的代价"

**关键技巧：把体检报告和 PR-FAQ 配套用**

体检报告是"否定的婉转表达"，PR-FAQ 是"重新校准方向的工具"。配合起来：
- 体检报告：让老板看到"原方向有问题"
- PR-FAQ：让老板在"经过补强后的几个新方向"里做选择

**这是项目总负责人对老板"管理"的最高境界——既保住了老板的体面，又把项目从悬崖边拉回来**。

### 7.4 需求的"组织安全性"——立得住才推得动
中国大厂的另一个特色：**一个需求即使技术上可行、用户上有价值，如果它"组织上不安全"——会冲撞既得利益、踩政治红线、跨越敏感边界——也推不动**。

**组织安全性的 5 个维度（个人框架）**

| 维度 | 风险表现 | 缓解 |
| --- | --- | --- |
| **既得利益冲突** | 你的产品会让某个团队的 KPI / 预算 / 编制受损 | 提前谈判、找替代价值、走联合立项 |
| **数据归属冲突** | 你的产品需要的数据归属另一个团队，他们不愿共享 | 换数据源 / 走数据协议 / 升级到上层老板协调 |
| **流程归属冲突** | 你的产品要改某个被另一个团队"拥有"的流程 | 让对方团队做 co-owner、共享成功 |
| **合规 / 安全红线** | 涉及个人隐私、跨境数据、未脱敏内容 | 提前 6 个月走合规流程、不要硬上 |
| **战略红线** | 涉及未公开的业务方向、并购计划、上市计划 | 立项前和老板单独确认、不写进公开文档 |


**典型案例：AI 周报助手的组织安全性体检**

```plain
1. 既得利益冲突？
   → OA 团队会不会觉得"被替代"？
   → 提前找 OA TL 谈：你的产品是"OA 之上的 AI 层"，OA 仍是基础
   → 让 OA 团队做联合 owner，共享对外曝光

2. 数据归属冲突？
   → 需要 Jira 数据，归属研发效能团队
   → IM 数据，归属 IT 基础设施团队
   → 提前 socialize、走数据协议、签 SLA
   → 不要等开发到一半才去要数据

3. 流程归属冲突？
   → 周报这个流程，归属 HR
   → 你不改流程本身（HR 仍然定义周报规范），
     你只是提供工具——明确边界，HR 反而欢迎

4. 合规 / 安全红线？
   → 周报内容是"工作信息"，可能涉及未公开的产品计划、
     客户数据、内部冲突
   → 提前 3 个月启动安全合规审查
   → 设计"敏感词检测 + 不传外网"机制

5. 战略红线？
   → AI 写周报会不会让某些岗位"显得多余"？
     是否触发"AI 替代人"的敏感话题？
   → 老板单独确认对外口径："效率工具，不是替代"
   → 内部宣传重点：员工解放，而不是裁员
```

**关键洞察**：**组织安全性的检查必须在立项之前完成，而不是在出问题之后补救**。**很多技术上完美的项目死于组织安全性**——比如某个 AI 系统用了未脱敏的客户数据，技术上跑得很好、上线 1 个月被合规叫停——这种案例在大厂里太常见了。

### 7.5 KPI 反向设计——避免被指标反向裹挟
最后一个中国大厂特色场景：**很多项目立项时被给一个"KPI"，但这个 KPI 设计得不合理，反而会逼着团队做错的事**。

**典型反例**

```plain
项目：AI 周报助手
被给的 KPI：年内累计使用人数 ≥ 5000 人

→ 这个 KPI 看起来合理，但会反向裹挟团队：
  - 团队为了完成 KPI，会大量推送、强制使用、刷量
  - "用过一次就走"的用户也算 KPI
  - 但产品价值（节省时间、提升满意度）一点都没体现

→ 最后的"成功"：5000 人使用过、但没有人留下
   = 烧钱完成 KPI，没创造价值
```

**KPI 反向设计的 4 步法（个人框架）**

```plain
Step 1：识别"被裹挟"的可能性
   - KPI 是不是"易刷"指标？
   - KPI 是不是和用户价值脱节？
   - KPI 是不是和北极星指标冲突？

Step 2：提出"组合 KPI" 替代单一 KPI
   - 量的指标 + 质的指标
   - 例：使用人数 + 留存率 + 满意度

Step 3：和老板谈判调整 KPI
   "我理解您要看到 5000 人的规模感，
    但单看人数会让我们失去对质量的关注。
    我建议改成：5000 人使用 + 30% 月留存 + NPS > 30，
    三个指标一起看。"

Step 4：写进项目共识文档，所有合作方对齐
   不要等老板"想起来"再改 KPI
   立项时就把"KPI 组合"白纸黑字写下来
```

**关键洞察**：**KPI 不是被动接受的，是项目总负责人和老板共同设计的**。**敢于在立项时就和老板谈 KPI、能谈出"既保规模感又保质量"的复合 KPI，是高级 leader 的标志**。**只会接 KPI 不会设计 KPI 的人，永远会被 KPI 反向裹挟做错事**。

## 第八部分：失败模式手册（Anti-Patterns）
正确做法已经讲完，现在固化错误做法。**下面 8 个反模式，每一个我都见过项目栽在上面**。

### 8.1 解决方案先行（先选 AI 再找问题）
**症状**：先决定"我们要用 AI / 区块链 / 大模型 / Agent"，再去找哪个场景能用。

**为什么是反模式**：
+ 技术不是用户需求——用户不在乎你用什么技术
+ "锤子找钉子"会做出形态扭曲的产品（用 AI 强行嵌套到不需要 AI 的场景）
+ 技术决策应该是问题的输出，不是问题的输入

**典型表现**：
- "我们买了 GPU，必须找场景把 GPU 用起来"
- "大模型这么火，我们要把所有产品都接入大模型"
- "AI Agent 是未来，我们必须现在就做"

**正解**：**永远从用户问题倒推技术方案**。如果用户的问题用规则引擎就能解决，就不要硬上 AI。**好的工程师手里有锤子时，会先看清楚有没有钉子，而不是把所有东西都当钉子敲**。

### 8.2 老板需求 = 真需求（盲信）
**症状**：把老板说的每一句话当圣旨，立刻执行不做翻译。

**为什么是反模式**：
+ 老板的话往往是"模糊的方向感"，不是"具体的需求"
+ 老板自己也在试错——他需要你帮他校准
+ 把老板的"想法"做成"产品"，是低级 leader 的做法

**正解**：参考 §7.1 老板需求翻译四步法。**真正高级的 leader 是能把老板的模糊想法变成清晰命题的人，而不是闷头执行的人**。

### 8.3 用户说要 = 用户真要（亨利·福特陷阱）
**症状**：用户问卷说"想要 X"，立刻去做 X。上线后发现没人用。

**福特名言（虽然真伪存疑，但寓意有效）**：
> "如果我去问用户想要什么，他们会说：'更快的马'。"

**为什么是反模式**：
+ 用户不擅长想象自己未见过的解决方案
+ 用户会说"我想要"，但不会真的为它付代价
+ 用户的"需要"和"想要"是两件事

**典型表现**：
- 调研问卷：80% 用户说"想要 AI 周报助手"
- 上线后：日活 5%
- 复盘：用户嘴上说要，但真到要花时间学习新工具就放弃了

**正解**：**Stated preference 必须和 Revealed preference 交叉验证**。问卷数据要打 50% 折扣，必须用行为数据（烟雾测试 / 假门测试）来验证真实意愿。**用户的承诺货币（时间、行为、金钱）比嘴上的话可信 10 倍**。

### 8.4 平均用户陷阱
**症状**：根据"用户平均"数据做产品决策，结果做出一个"理论上最优、现实中谁都不爱用"的产品。

**为什么是反模式**：
+ 真实用户群体永远是分层的、双峰甚至多峰的
+ "平均用户"在数学上存在，在物理上不存在
+ 给"平均用户"做的产品，对每一个真实用户都不够好

**经典例子**（美国军方 1950s）：美国空军飞行员飞机驾驶舱按"平均飞行员尺寸"设计，结果发现 4063 名飞行员里没有一个完全符合 10 个维度上的"平均"。后来改用"可调节驾驶舱"，事故率大幅下降。

**正解**：**永远具体化到 Persona，永远找真人原型**。如果做不出"3 个具体的真人都觉得这个好"，就不要假装"5000 个平均用户都会喜欢"。

### 8.5 数据完美主义
**症状**：调研、访谈、数据分析、报告——做了 6 个月，还在"完善需求文档"，迟迟不立项、不上线、不验证。

**为什么是反模式**：
+ 完美的需求文档不存在——你永远缺一个数据
+ "再多调研一周"是焦虑型 leader 最常用的拖延策略
+ 真实需求只有上线后才能被真正检验
+ 等你"准备好"，机会窗口可能已经过了

**正解**：**70 分的需求 + MVP 验证 > 95 分的文档但没有验证**。MVP 是最便宜的"调研工具"——花 2 周做出来给 50 个用户用，比花 6 个月写文档更接近真相。

> **金句改编**："Done is better than perfect. Validated is better than thought-through."
>

### 8.6 内部 PMF 假象（自家人都觉得好）
**症状**：内部员工试用反馈很好，于是觉得"已经验证了"——上线给外部 / 全公司，发现根本没人用。

**为什么是反模式**：
+ 内部用户被强制使用 / 不愿伤你的心 / 给好评是"配合演出"
+ 你的同事是"特殊用户"——他们懂技术、懂背景、有内部支持
+ 真实的目标用户没有这些"特殊条件"

**典型表现**：
- 团队内部测试：90% 满意
- 上线给全公司：日活 5%
- 复盘：内部用户和外部用户的"使用门槛"完全不同

**正解**：**内部测试和外部上线是两个完全不同的验证阶段**。内部测试只能验证"产品能不能跑通"，不能验证"用户会不会用"。**真正的 PMF 验证必须在"陌生用户"群体里做**——找 30 个完全不认识你的用户、给他们工具、看他们用不用。

### 8.7 一次性需求调研
**症状**：项目立项前做了一次大规模调研，之后再也不做了。立项 6 个月、12 个月、24 个月，需求文档还是当初那一版。

**为什么是反模式**：
+ 用户需求是动态的——市场、竞争、用户习惯都在变
+ 立项时的"用户痛点"，可能 12 个月后已经被竞品解决了
+ 没有持续调研 = 在一张过期地图上开车

**正解**：**需求挖掘是持续的，不是一次性的**。每个季度至少做一次"需求复检"：
- 用户访谈 5-10 人，看痛点是否还存在
- 行业扫描，看有没有新竞品 / 新替代方案
- 行为数据复盘，看产品的实际使用是否符合假设
- 不符合 → 立刻调整方向，不要等"重大里程碑"

### 8.8 拒绝杀掉（Sunk Cost Trap）
**症状**：项目已经投入了 1000 万、9 个月，数据明明显示"用户不接受"，但没人敢说"停"。

**为什么是反模式**：
+ 沉没成本不应该影响未来决策——这是经济学常识，但人性会反着来
+ "再投入 500 万搏一搏"几乎永远是错的
+ 拒绝杀掉的项目变成"组织级僵尸"——耗资源、占编制、毁士气

**典型表现**：
- 数据明明在变差，每周复盘还在说"再观察一下"
- "我们已经投入了这么多，不能放弃"
- 老板自己都不再关心这个项目，但下面没人敢提"停"

**正解**：**Kill Criteria 必须前置定义、严格执行**（参考 §5.5）。**敢杀掉自己项目的 leader，长期信用远高于"咬牙救活一个垃圾项目"的 leader**。前者是判断力，后者只是消耗。**这是工程行为方法论 §3.8 末节"何时该止损"的延伸——杀掉项目不是失败，是清醒**。

## 第九部分：实战推演——以"企业级 AI Agent 系统"为例端到端复盘
理论说完，下面用一个完整的虚构但贴近真实的项目案例，把方法论串起来。**这个案例就是文章开头你提到的——"老板说要做企业级 AI Agent，但没人说得清要解决什么问题"**。

> **项目背景**：你所在公司是一家 5000 人的中大型互联网公司，CEO 在某次外部行业大会回来后，向 CTO 和 CIO 抛下一句："我们也要搞个企业级 AI Agent 系统，半年内出成果"。CTO 把这事压到你头上——"项目总负责人"，"全权负责"。**钱有、人有、时间紧、目标模糊**。

### 9.1 阶段一：拒绝直接接需求，先做"伪需求体检"（Week 1-2）
**很多人的本能**：立刻拉群、招人、选型、调研大模型 / Agent 框架。

**正确的本能**：**先停 2 周，做需求体检**。

**Week 1：还原老板触发 + 真实诉求识别**

```plain
Day 1-2：老板触发还原（§7.1 Step 1）
- 单独约 CTO 喝咖啡 30 分钟
- 不要直接问"老板到底想要什么"
- 用问题引导：
  "您觉得 CEO 是在什么场景下提的这个？"
  "他提的时候在场的还有谁？"
  "他对'成果'的预期是什么样的？数据？产品？媒体曝光？"

获得的信息：
- CEO 是在某次行业峰会，听了某竞品 CEO 的分享后说的
- 在场还有董事会某成员，对方点头表示赞同
- "成果"的真实含义是"年报里能写一页关于公司 AI 转型"

→ 真实诉求 = 对外宣传 + 不掉队焦虑（约 70%）+ 一点点真实业务效率诉求（约 30%）

Day 3-5：跨部门"诉求扫描"
- 见 CIO：他关心成本，担心又是一个烧钱项目
- 见 PR / 战略：他们确实需要"年报亮点"，但更想要"客户能感知的故事"
- 见 HR / 行政：他们对"内部效率工具"有真实诉求（招聘、入职、报销自动化）
- 见业务线 1（客服）：客服 TL 关心"成本压缩"，已经在自己用大模型试水
- 见业务线 2（销售）：销售 TL 不感冒——"我们要的是客户，不是 AI"
- 见研发：研发员工有"个人提效"诉求，但担心"被 AI 监控"
```

**Week 2：写需求体检报告 + 三个候选方向的 PR-FAQ**

把 Week 1 的信息整理成一份《企业级 AI Agent 项目需求体检报告》，给 CTO + CIO 看。

```plain
# 需求体检报告

## 体检结论
当前项目处于"半跟风型"状态——
有真实的对外宣传诉求 + 部分真实业务痛点，
但具体落地方向尚未收敛。

直接立项有 70% 概率烂尾。
建议在投入正式资源前，用 6 周完成方向收敛。

## 体检维度
1. 用户和场景清晰度：D（无具体 Persona）
2. 真实痛点支撑度：C（有零散信号）
3. 一手数据完整度：D（无访谈数据）
4. ROI 论证可信度：D（"如果"句式）
5. 杀掉条件明确度：D（不存在）
6. 组织安全性：B（无明显冲突，但需进一步评估）

## 三个候选方向（PR-FAQ 各一份附后）
A. 客户侧：客服 / 售前 Agent v1
   - 投入：6 个月、800 万
   - 对外宣传：✓✓（有客户案例可讲）
   - 真实业务价值：✓✓（客服成本下降 20%）
   - 风险：客户体验失控

B. 员工侧：内部 AI 助手平台 + 灯塔场景（如周报）
   - 投入：9 个月、1500 万
   - 对外宣传：✓（有"组织 AI 化"故事）
   - 真实业务价值：✓✓（员工效率 + 文化）
   - 风险：内部接受度

C. 平台侧：对外开放 Agent 开发平台（PaaS）
   - 投入：18 个月、5000 万
   - 对外宣传：✓✓✓（行业级故事）
   - 真实业务价值：✗（短期看不到）
   - 风险：极高，技术 + 商业模式都未验证

## 建议路径
1. CEO / CTO 在 A / B / C 中选 1-2 个方向
2. 选定后 6 周做需求收敛 + Pre-mortem + Kill Criteria
3. 6 周后再决定是否正式立项
```

**关键动作**：**把这份报告呈给 CTO 之前，先在 1on1 里口头聊一遍**。让 CTO 觉得"是他在做明智决策"，不是"被你反对"。

**老板的真实回应（典型剧本）**：

> "我同意先做一下方向收敛。AI Agent 这事我也没完全想清楚——你说的对，A 和 B 比 C 现实。我倾向 B，但 A 也别完全放掉。给我一个 B 为主、A 为辅的方案吧。"

**→ 项目从"做 AI Agent"翻译成了"做内部 AI 助手平台 + 周报灯塔场景，同时在客服场景做轻量验证"——清晰度提升 10 倍**。

### 9.2 阶段二：用户与场景重定义（Week 3-8）
方向定了，接下来 6 周做需求收敛。

**Week 3-4：JTBD + Persona + 痛点访谈**
- 访谈 30 个目标用户（研发岗 20 人 + 产品岗 10 人）
- 用 The Mom Test 三铁律：只问过去、不问未来、问行为不问态度
- 输出：3 个 Persona（重度 / 中度 / 轻度）+ 一份痛点优先级表
- 关键发现：**真正的痛点不是"周报本身"，是"跨系统找信息" + "被老板看见"**

**Week 5-6：MVP 设计 + 烟雾测试**
- 设计 3 种产品形态：浏览器插件 / IM 机器人 / Web 工具
- 在公司 IM 群投放烟雾测试
- 数据：浏览器插件 12% 报名 / IM 机器人 28% 报名 / Web 工具 5% 报名
- **决策**：选 IM 机器人作为主形态

**Week 7-8：假门测试 + 绿野仙踪**
- 上线"周报机器人 v0"——前端是机器人，后端是 5 个工程师手动 + 大模型辅助
- 50 个种子用户使用 4 周
- 数据：用户改动率 35%（接近目标 30%），日活 65%，NPS 42
- **结论**：核心假设成立，可以进入正式开发

### 9.3 阶段三：MVP 设计与杀掉条件（Week 9-10）
6 周收敛完成，回头给 CTO 做汇报，正式立项。

**立项文档（PR-FAQ + 6-Pager 配套）**

```plain
# 项目正式立项：内部 AI 助手平台 v1

## 范围（In Scope）
- 第一阶段：研发岗 + 产品岗的"周报场景"
- IM 机器人 + Web 工具双入口
- 集成 Jira / IM / 文档系统
- 时间：6 个月（2026 H2）
- 预算：1500 万

## 范围（Out of Scope，显式声明）
- ❌ 多轮对话型 Agent
- ❌ 销售 / 客服 / HR 等其他岗位（暂不做，v2 候选）
- ❌ 移动端
- ❌ 替代周报这件事本身

## North Star Metric
每周完成周报的用户数 / 目标用户数 ≥ 60%

## Kill Criteria（杀掉条件）
- 上线 3 个月，目标用户日活 < 30% → 暂停 review
- 用户改动率持续 > 70% → 重新评估技术路线
- 关键合作团队（数据/安全）撤回支持 → 优雅退出
- 预算超支 30% 且未达里程碑 → 升级 CTO 决策

## 组织安全性已确认
- ✅ OA 团队：联合 owner，共享对外曝光
- ✅ 数据团队：Jira / IM 数据接入协议已签
- ✅ 安全团队：AI 应用合规框架走通（v1 灯塔）
- ✅ HR：流程不变，只是工具升级
```

### 9.4 阶段四：验证、扩量、迭代机制（Month 4-12）
立项后的 9 个月不是"开发完了就上"，而是按"验证—扩量—迭代"的节奏走。

**M4-M6：核心功能 MVP（100 种子用户）**
- 端到端实现，但只放给 100 个用户
- 每周 5 个流失用户访谈（§4.3）
- 月度 NPS 测量
- **关键里程碑**：M6 末，决定是否扩量

**M7-M9：第一次扩量（1000 用户）**
- 扩量条件：种子用户 NPS > 40 + 留存 > 50% + 用户改动率 < 35%
- 扩量后建立"自助接入"流程
- **关键里程碑**：M9 末，决定是否全量

**M10-M12：研发岗全量（5000 用户）**
- 全量后启动 v2 规划
- 启动"对外宣传"配套（PR / 案例 / 媒体）
- 启动其他岗位（销售 / HR / 行政）的需求挖掘——**回到第二阶段，重新做 JTBD**

### 9.5 复盘：从"老板想要"到"用户真用"的转化路径
项目上线 12 个月后，做一次完整复盘：

```plain
# 项目复盘：内部 AI 助手 v1

## 时间线
- M0：CEO 抛出"做 AI Agent"
- M0+0.5（W1-W2）：完成需求体检，方向从"AI Agent"翻译为"内部 AI 助手 + 周报灯塔"
- M0+2（W3-W8）：完成需求收敛（JTBD / 烟雾 / 假门 / 绿野仙踪）
- M0+2.5（W9-W10）：正式立项，进入开发
- M0+8（M6 节点）：核心功能 MVP 上线，100 种子用户验证通过
- M0+11（M9 节点）：扩量到 1000 用户
- M0+14（M12 节点）：研发岗全量 5000 用户

## 哪些做对了
1. 没有直接接"做 AI Agent"，先做了 2 周需求体检——避免了 70% 烂尾概率
2. 用 PR-FAQ 让 CEO 在具体方向中选，而不是泛泛讨论
3. 杀掉条件前置定义，遇到 M3 改动率偏高时主动降速
4. 找到 OA / 数据 / 安全做联合 owner，避免组织冲突
5. 流失用户访谈（每周 3-5 个）暴露了 M5 的关键问题

## 哪些做错了
1. M2 阶段"绿野仙踪"投入了 5 个工程师，事后看 3 个就够
2. M4 选型时低估了 IM 机器人的工程量，延期 2 周
3. 对"老板对外宣传需求"的优先级估计不足，M8 临时加了 PR 案例

## 走运了什么
1. 友商发布了相似产品但产品形态错了——
   反而验证了我们的方向
2. 公司年报正好需要"AI 案例"，我们的项目刚好踩点

## 沉淀的"组织资产"
1. 内部 AI 应用合规框架（首例走通，后续项目可复用）
2. 用户需求挖掘方法论（这份文档）
3. 30 个跨岗位 Persona 库（其他 AI 项目可复用）
4. 流失用户访谈机制（推广到其他产品）

## Action Items
- [ ] 把"需求体检报告"模板化、推广给其他项目（Owner: 我，Due: M+2）
- [ ] 启动其他岗位的需求挖掘（Owner: 产品 TL，Due: M+1）
- [ ] 把 v1 经验教训整理成"内部 AI 应用 Playbook"（Owner: 我，Due: M+3）
```

**核心心法（这个案例的最大启示）**：

**项目最大的价值不是"按老板要求做出了 AI Agent"，而是"把老板模糊的诉求翻译成了清晰的、可工程化的、能创造真实价值的命题"**。这个翻译过程才是项目总负责人的真正杠杆——它的价值远高于"招了多少人、写了多少代码、上线了多少功能"。

**反过来说**：如果你接到 CTO 那句"做 AI Agent"立刻开干，6 个月后大概率交付一个技术上能跑、产品上没人用、对外讲不出故事、对内消耗信任的"半成品"——而你会被复盘为"执行力强但缺乏产品判断"。**这就是为什么前面所有方法论都重要**。

## 第十部分：可立即落地的 Checklist + 话术速查
把整篇方法论压缩成可以直接拿去用的清单。

### 10.1 接到老板需求 Checklist（黄金 48 小时）
```plain
[ ] 不要立刻拉群、招人、写方案
[ ] 先单独和直属老板 1on1：
    [ ] 还原老板触发场景（在哪、对谁、什么背景下说的）
    [ ] 确认"成果"的真实含义（产品？数据？媒体？故事？）
    [ ] 确认资源边界（钱、人、时间）
[ ] 跨部门"诉求扫描"——找 5-10 个相关方了解他们的看法
[ ] 写需求体检报告 + 2-3 个候选方向 PR-FAQ
[ ] 1on1 呈给老板，让老板在具体方向中选
[ ] 永远不要在 48 小时内承诺具体方案 / 时间 / 范围
```

### 10.2 需求挖掘 Checklist
```plain
[ ] 用户/客户/决策者/付费方四象限对照表已画
[ ] 每个核心场景都用 JTBD 句式写出来了（情境/动作/回报）
[ ] 痛点已经挖到第三层（症状/问题/根本动机）
[ ] 至少访谈 8-15 个真实用户，且达到"信号饱和"
[ ] 每个 Persona 有具体真人原型
[ ] In Scope / Out of Scope / Future / Kill Criteria 显式声明
[ ] 至少做了一轮 MVP 验证（烟雾/假门/绿野仙踪三选一）
[ ] 关键假设清单 + 验证成本 + 验证顺序已排好
```

### 10.3 立项前 Checklist
```plain
[ ] PR-FAQ 已写（含 10-20 条 FAQ）
[ ] North Star Metric 已定义（不是 DAU / 累计用户）
[ ] 关键合作团队的 OKR 重叠点已找到
[ ] 组织安全性 5 维度已评估
[ ] Kill Criteria 三层已写
[ ] 立项前已做 Pre-mortem 至少 1 轮
[ ] 路线图三层折叠（Now / Next / Later）已写
[ ] KPI 是组合指标，不是单一指标
```

### 10.4 项目运行期 Checklist
```plain
[ ] 每月至少访谈 3-5 个流失用户
[ ] 每季度做一次"需求复检"（用户痛点是否还在？竞品是否变化？）
[ ] North Star Metric 周报必看
[ ] Kill Criteria 月度对照（接近触发的要主动 escalate）
[ ] 路线图季度 review，公开宣布层间转移
[ ] 老板对项目的关注度变化（从 1on1 频率、提及次数判断）
```

### 10.5 话术速查
**A. 应对老板"为什么这么慢，不是说一个月就能出成果吗"**

> "老板，我知道您着急。但我们花 2 周时间做需求体检，是想避免 6 个月后发现方向错了——那个代价比 2 周大 50 倍。我可以每 3 天给您同步一次进展，您看怎么样？"

**B. 应对老板"友商已经发布了，我们要赶紧跟"**

> "老板，我看了友商的方案——他们做的是 X，我们要做的是 Y，用户群完全不同。如果我们硬跟，6 个月内能出 demo，但用户不会买账。建议我们用 4 周做用户研究，找到真正适合我们的切入点，反而能更快出真正的成果。"

**C. 应对合作方"你这个项目对我有什么好处"**

> "[张哥]，这个项目和你们 H2 OKR 里的 [具体 KR] 有重叠点。具体来说：[列出对对方组织/团队/个人三层价值]。我希望我们能一起做这个事，不是您配合我，是我们一起做出对双方都有价值的东西。"

**D. 应对老板"杀掉条件什么的太消极了，我们怎么能立项就想着失败"**

> "老板，杀掉条件不是想着失败——它是给我们一个'清醒的标准'。如果项目跑得好，杀掉条件根本不会触发；如果跑得不好，我们能比所有人都更早发现并止损。这其实是给您和团队的'保险机制'，不是消极。"

**E. 应对用户"你们做这个有什么用，我们现在不也好好的"**

> "我不是来推产品的——我想了解下，您上次写周报花了多久？卡在哪一步？我们想看看真实的工作流程是什么样的，再决定要不要做工具，怎么做。"

**F. 应对调研对象"我觉得这功能挺好的，应该会用"**

> "谢谢您的反馈。能问个具体的问题吗——上一次您遇到 [类似问题] 是什么时候？您当时是怎么解决的？如果我们做出这个工具，您愿意试用一周给我们反馈吗？"
> （把"觉得好"翻译成"具体行为"）

**G. 应对项目内部"再调研一周吧，数据还不够"**

> "我们已经有 X 个数据点了，足够支持初步判断。再等一周不会让数据质量提升 50%，但会让我们错过 [具体机会窗口]。建议我们用 70 分的数据先做出 MVP，让真实用户用起来——MVP 的反馈会比再调研一周更有价值。"

## 第十一部分：心法——三句话总结整篇
如果只能记三句话：

1. **需求挖掘的目标不是"满足用户"，是"识别值得用工程资源去解决的真问题"**——这两个目标看起来相近，实际差 10 倍。前者是产品经理的活，后者是项目总负责人的命。**老板说要做、用户说想要、调研说支持——三者都不等于"真问题"**。真问题的标志只有一个：**用户愿意付出真实代价（时间、行为、金钱）来获得解决方案**。

2. **解空间的最优解，建立在需求空间的正确问题之上**——架构设计方法论让你"把事情做对"，工程行为方法论让你"把事情做下去"，技术演进让你"选对技术"，这三套都假设了"问题本身是对的"。**而问题对不对，是项目总负责人的最高责任**。把模糊的老板诉求翻译成清晰的工程命题，是 leader 真正的杠杆——比写代码、带团队、推架构都重要。

3. **敢于慢，是为了真的快；敢于不做，是为了把资源投在真做的事上；敢于杀掉，是为了不被沉没成本拖死**——这三个"敢"是项目总负责人面对组织压力时唯一的护城河。**最厉害的项目总负责人，做出来的项目数量往往不是最多的，但活下来的项目比例是最高的、价值是最持续的**。这才是真正的"工程影响力"——不是用蛮力推动一切，而是用判断力让对的事情发生、错的事情提前停止。

---

> **写在最后**：这套方法论和《架构设计方法论》《工程行为方法论》《后端技术演进全景》是一组互补的工具集——
>
> - **用户需求挖掘方法论**：决定"做不做"
> - **架构设计方法论**：决定"怎么建模"
> - **后端技术演进全景**：决定"怎么选型"
> - **工程行为方法论**：决定"怎么推动"
>
> 四套方法论合起来，覆盖了一个项目从"老板抛话"到"上线运营"的完整生命周期。**单独用任何一套都会有盲点；合起来用，才是一个完整的项目总负责人的"操作手册"**。
