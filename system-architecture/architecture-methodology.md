# 系统架构通用方法论

> **本文不是教程，是心法。** 目标：让你在任何高并发系统设计场景下，都能用一套统一的思维框架，从零推导出一个扛得住、救得回、扩得开的方案。读完这篇你应该能回答：**为什么所有互联网系统的架构长得都差不多？** —— 因为所有高并发系统面对的物理约束是一样的，而人类对抗物理约束的招数就那几种。

---

**核心认知：架构的本质**

**架构不是"设计出来"的，是"算出来 + 逼出来"的。** 架构的每一个决策，都是被物理约束和业务 SLA 联合逼出来的最优解。三条铁律决定了所有架构的形态：

| 铁律 | 具体含义 |
|------|---------|
| **① 单机物理极限** | MySQL 写 3000~5000 TPS；Redis 单分片 10 万 QPS；万兆网卡 10 Gbps；SSD 随机写 10 万 IOPS；CPU 上下文切换微秒级 |
| **② 分布式不可靠** | RPC 会超时/丢包/乱序；节点会突然宕机；时钟有漂移 |
| **③ 业务 SLA 是死命令** | 可用性 99.99%（年停 52min）；P99 < 200ms；丢数据 < 1e-7；峰值 = 平均 × 5~100 倍 |

**架构师的工作，就是在这三条铁律的夹缝中，用有限资源做出满足 SLA 的系统。** 所有高并发架构的"套路"不是巧合，是物理决定论的必然结果。

---

## 1. 需求澄清 —— 先划定"战场"再打仗

**90% 的架构失败在需求阶段。** 架构设计第一原则：**解空间取决于约束空间**，不知道约束就设计不出合理方案。

### 1.1 需求三问

**① 功能边界**：核心业务流程 / 核心实体和动作 / MVP vs nice-to-have。

**② 非功能 SLA**（必须量化）：

| 维度 | 示例量化 |
|------|---------|
| 可用性 | 99.9%（8.76h）/ 99.99%（52min）/ 99.999%（5min） |
| 延迟 | P99 < 200ms / P999 < 1s |
| 吞吐 | 平均 QPS × 峰值系数（3~100x） |
| 一致性 | 强一致 / 最终一致 / 秒级延迟 |
| 数据规模 | 热 30 天 + 温 1 年 + 冷归档 |

**③ 明确禁行**（最容易被忽略但最关键）：禁行收敛解空间，避免过度设计。例：延时任务禁止 ORDER BY 实时扫描；红包禁止浮点金额；短链禁止自增 ID 暴露；推送禁止对大 V 写扩散；评论禁止 OFFSET 翻页。

**为什么"禁行"比"功能"更重要**：功能可以慢慢加，但错误的技术选型一旦上线就是灾难。禁行本质是"我们已经预判了哪条路是坑，不要走"。好的架构师面对需求时，先亮起的不是"我能怎么做"，而是"我绝对不这么做，因为会死"。

### 1.2 需求阶段的终极三问
1. **这个系统存在的理由是什么？**（别设计没用的东西）
2. **如果系统挂一天，谁会骂街？骂多狠？**（决定 SLA）
3. **业务 3 年后会长什么样？**（决定扩展性预留）

## 2. 容量评估 —— 用数字代替感觉

**没算容量就画架构 = 没看地形就打仗。** 所有架构决策背后都是数字：分库因为单库 TPS 不够；缓存因为 DB QPS 不够；CDN 因为回源带宽不够；异步因为同步链路超时；分片因为单机内存不够。**容量评估不是形式主义，是让你看清哪里会先撑爆。**

### 2.1 容量推导六步公式

```
Step 1：峰值 QPS = (DAU × 人均次数 / 86400) × 峰值系数
Step 2：带宽(Gbps) = 峰值 QPS × 单请求字节 × 8 / 10⁹ × 冗余(2x)
Step 3：存储(TB) = 日新增条数 × 单条字节 × 保留天数 / 10¹²
Step 4：分库数 = 目标 TPS / 3000 → 向上取 2 的幂次
Step 5：Redis 分片数 = 峰值 QPS × (1-本地命中率) / 10 万 → 2 的幂次
Step 6：服务节点数 = 峰值 QPS / (单机 QPS × 0.7 水位)
```

### 2.2 必须记死的基准常量（"物理常量"，跨公司通用）

| 组件 | 安全水位 | 组件 | 安全水位 |
|------|---------|------|---------|
| MySQL 单库写 | 3000 TPS | Kafka 单分区 | 5000~10000 TPS |
| MySQL 单库读 | 10000 QPS | 服务节点 | 70% 负载 |
| MySQL 单表 | < 1 亿行 | 万兆网卡 | 8 Gbps |
| Redis 单分片 | 10 万 QPS / ≤20GB | 同机房 RTT | < 1ms |
| Redis 单 Key | ≤ 10 MB | 跨机房/地域 RTT | 1~5ms / 30~100ms |

### 2.3 峰值系数（最容易被低估）

**系统是被峰值压垮的，不是被平均值压垮的。**

| 场景 | 峰值系数 | 场景 | 峰值系数 |
|------|---------|------|---------|
| 工具类均匀访问 | 3~5x | 整点活动（电商促销） | 20~50x |
| 社交日活变化 | 5~10x | 突发热点（明星发微博） | 100~1000x |
| 系统雪崩重试 | 无上限 |  |  |

### 2.4 实战例子：点赞系统

DAU 1 亿、人均 5 次、峰值系数 10x：
```
平均 QPS = 1亿 × 5 / 86400 ≈ 5787；峰值 QPS ≈ 5.8 万
DB 写 TPS = 5.8 万 → 20 主库 → 向上取 32 库
Redis（80% 命中）= 1.16 万 → 4 分片留余量
带宽 = 5.8万 × 200B × 8 / 10⁹ ≈ 0.1 Gbps（不是瓶颈）
服务节点 = 5.8万 / (2000 × 0.7) ≈ 42 台
3 年存储 = 5 × 1亿 × 365 × 3 × 100B ≈ 55 TB → 必须冷热分层
```
架构雏形就出来了：**32 分库 + 4 Redis 分片 + 42 应用节点 + 冷热分层**。没有数字支撑的架构都是耍流氓。

## 3. 架构分层 —— 隔离变化的终极武器

**分层的本质是"隔离变化"。** 不同层的变化频率不同（接入层几年不动 / 应用层天天改 / 中间件数年稳定 / 存储层变化最慢但代价最高），不隔离就会互相绑架。**把变化频率相近的东西放一层，不同频率的东西之间用接口隔离。**

### 3.1 标准四层架构

| 层 | 职责 | 特点 |
|----|------|------|
| **接入层** | LVS/Nginx/Gateway：SSL 卸载 / 限流 / 鉴权 / 路由 | 无状态、水平扩展 |
| **应用层** | 业务服务集群：业务逻辑、编排 | 无状态、快速迭代、K8s 弹性 |
| **中间件层** | Redis/Kafka/ETCD：缓存、异步、配置、协调 | 有状态、集群化、稳定 |
| **存储层** | MySQL/HBase/OSS：数据持久化 | 强一致、高可靠、迁移成本极高 |

### 3.2 分层三铁律 + 关键澄清

- **① 下层不依赖上层**：存储不知道谁在用它
- **② 同层最小耦合**：两个应用服务最好不直接 RPC，用事件
- **③ 跨层走标准协议**：应用→DB 必走 ORM/DAO，不裸 SQL 四处散

**分层 ≠ 拆服务**：分层是技术维度（物理布局），拆服务是业务维度（能力独立演进），两者正交。分层错了扩不动，拆服务错了跑不快。

## 4. 领域建模 —— 实体/事件/读模型三分法

**不要照搬 DDD 的 8 种 building block**（大部分项目用不上，徒增心智负担）。真正有效的是三分法：

| 类别 | 本质 | 职责 | 对应物 |
|------|------|------|-------|
| **实体** | 有独立生命周期的业务对象 | 负责正确性（状态机、强一致） | MySQL 主表（聚合根） |
| **事件** | 发生过的动作（只可追加） | 负责可追溯（审计、对账、回放） | Kafka + HBase |
| **读模型** | 为查询优化的反范式视图 | 负责性能（允许冗余、延迟、可重建） | Redis + ES + ClickHouse |

**最大误区**：三者混在一张表 → 读写互相影响、一致性和性能二选一、改需求就要改核心表。**物理隔离是必须的。**

### 4.1 建模的最高境界：找到"业务不变式"

识别永远成立的等式：
- 账户：`借方 = 贷方`（复式记账）
- 红包：`已抢 + 剩余 = 总额`
- 库存：`可售 + 锁定 + 已售 = 总库存`
- 订单：`总价 = sum(商品) + 运费 - 优惠`

**不变式决定强一致边界** —— 涉及的所有数据必须在一个事务内。这是架构师判断"哪些必须强一致，哪些可以最终一致"的黄金标尺。

## 5. 核心流程 —— 画对四条线

每个系统必画四条流程：**① 正常主流程（90% QPS 走这）② 热点路径（最高并发单点，专门优化）③ 异常路径（下游超时/部分失败/MQ 堆积）④ 对账兜底（数据错了怎么发现怎么修，金融系统生死线）。**

### 5.1 核心权衡：同步 vs 异步

| 必须同步 | 可以异步 |
|---------|---------|
| 用户要立即看到结果 | 用户不关心细节（通知/统计） |
| 失败必须回滚 | 失败可以补偿 |
| 强一致要求 | 最终一致可接受 |
| 金额相关关键事务 | 审计、日志、索引构建 |

**原则：核心链路尽量短、尽量同步；非核心全部异步。**

### 5.2 同步链路的"极简主义"

高并发系统核心链路只做三件事：**① 鉴权（能不能做）② 校验（做得合理吗）③ 落地（改状态 + 发事件）**。其他全部异步：发通知 → MQ；更新统计 → MQ；建索引 → Binlog；风控分析 → 日志。这样同步链路 P99 可压到 50ms 以内。

### 5.3 异步链路三铁律
- **幂等**：消息可能重复投递，消费者必须幂等
- **顺序**：同一实体的消息必须同分区同消费者
- **兜底**：消息可能丢，必须有对账机制

## 6. 缓存设计 —— 以空间换时间的艺术

**缓存 = 把高频数据放到更快的存储介质。** 速度层级（每层差 10~100 倍）：

```
L1 CPU Cache 纳秒 → L2 本地内存 微秒 → L3 Redis 毫秒 → L4 SSD 10ms → L5 HDD 100ms → L6 跨区 100ms+
```

**好的缓存架构是多级组合，不是单点。**

### 6.1 三大经典问题

| 问题 | 根因 | 解法 |
|------|------|------|
| **穿透** | 查不存在的 key 次次打 DB | 布隆过滤器 + 空值缓存（短 TTL） |
| **击穿** | 单热点 key 过期瞬间 | singleflight + 互斥锁 + 逻辑过期 |
| **雪崩** | 大量 key 同时过期 / Redis 宕机 | TTL 随机抖动 + 多级缓存 + 熔断降级 |

### 6.2 一致性策略（没有完美方案，只有权衡）

| 策略 | 一致性 | 性能 | 适用 |
|------|-------|------|------|
| Cache Aside | 最终一致 | 高 | 默认选择 |
| Write Through | 强一致 | 中 | 读多写少且要求一致 |
| Write Back | 最终一致 | 最高 | 允许丢数据（计数器） |
| Binlog 订阅 | 最终一致（秒级） | 高 | 跨库同步、CDC |

**真理：缓存一致性本质是"可接受的不一致窗口"问题。** 1 秒容忍 → Cache Aside；毫秒级 → CDC；强一致 → 放弃缓存或分布式锁。

**反模式**：❌ 缓存所有数据 ❌ TTL 全一样（雪崩）❌ 缓存即真相 ❌ 缓存复杂对象 ❌ 缓存未限流

## 7. 消息队列 —— 削峰、解耦、最终一致的三板斧

**所有架构都有 MQ 的原因**：

| 作用 | 说明 |
|------|------|
| **削峰** | 前端 50 万 QPS → Kafka → 后端 5 万 TPS，典型 10:1 |
| **解耦** | 服务间通过事件通信，新增消费者不改上游 |
| **最终一致** | 本地事务 + MQ = 跨服务事务，远比 2PC/XA 实用 |

### 7.1 可靠性三层保障
- **生产者**：acks=all + min.insync.replicas=2 + 同步刷盘（关键链路）+ 本地消息表兜底
- **Broker**：多副本 + 跨可用区 + ISR 防脑裂
- **消费者**：手动提交 offset + 幂等处理 + 重试队列 + 死信

### 7.2 消息三语义

| 语义 | 说明 | 代价 |
|------|------|------|
| At-most-once | 最多一次，可能丢 | 快 |
| At-least-once | 至少一次，可能重（默认） | 中 |
| Exactly-once | 恰好一次 | 贵 |

**工程真理：Exactly-once 不存在，只存在 At-least-once + 幂等 = 等效 Exactly-once。**

### 7.3 堆积应急 & 选型
- **应急优先级**：① 先扩消费者（最快）② 降级非核心 ③ 最后扩分区（rebalance 有成本）。必备监控：lag 按分区、消费速率、重试率、死信数量。
- **选型口诀**：Kafka（日志/大数据/高吞吐）/ RocketMQ（事务/金融/延时）/ RabbitMQ（复杂路由/企业）/ Pulsar（存算分离/多租户）。大厂多用 Kafka + RocketMQ 组合。

## 8. 容错设计 —— 假设一切都会失败

**墨菲定律：凡是可能出错的一定会出错。** 必须假设：网络会抖、机器会挂、磁盘会坏、时钟会漂、依赖会慢、数据会丢。**架构韧性 = 出问题时系统能继续提供（哪怕降级的）服务。**

### 8.1 容错五件套

| 件 | 作用 | 关键点 |
|----|------|-------|
| **限流** | 保护自己不被打死 | 分层（网关/服务/依赖）+ 令牌桶/漏桶/滑动窗口 + QPS/并发/连接 |
| **熔断** | 保护下游不被拖死 | 错误率 > 阈值开启 → 半开探测 → 恢复/继续 |
| **降级** | 优先保核心 | 一级关非核心 → 二级简化响应 → 三级只保核心 |
| **隔离** | 故障不扩散 | 线程池 / 机器 / 租户隔离 |
| **超时** | 不等无意义的等 | 每层都设 + 下层 < 上层 + 整体 < 用户耐心 |

### 8.2 动态开关：ETCD 的真正用途

所有容错参数必须在 ETCD 里（秒级生效）：限流阈值 / 降级开关 / 路由规则 / 熔断参数。**硬编码的容错策略 = 没有容错策略**，故障来时改代码发版已经晚了。

### 8.3 降级的艺术

- **好降级**：用户无感知或可接受（推荐没了还有默认 feed）；数据不脏；可快速恢复（开关立即生效）
- **烂降级**：数据结构不兼容 / 需要手工修数据 / 用户大量报错

### 8.4 灾备层级（SLA 越高成本指数上升，按业务价值决策）

| 层级 | 恢复时间 | 层级 | 恢复时间 |
|------|---------|------|---------|
| 单机故障 | 秒级（LB 摘除） | 同城故障 | 小时级（跨城切换） |
| 单机房故障 | 分钟级（切流） | 异地故障 | 小时~天（异地多活） |

## 9. 可扩展性 —— 让系统"长大"不改造

**扩展性 = 不停服应对 10 倍增长。** 为什么难？单机→集群要架构重写；单库→分库要数据迁移；单协议→多协议要接口重设计。**好架构从 Day 1 就为 10 倍增长预留。**

| 扩展层 | 难度 | 关键手法 |
|-------|------|---------|
| **服务层** | 最容易 | 无状态 = 水平扩展（不存本地数据/会话/强一致本地缓存）+ K8s 一键 |
| **缓存层** | 中等 | Redis Cluster 在线 reshard + MOVED 重定向 + 大 Key 治理。**技巧：Day 1 分 32+ 分片，后续扩容只是调度** |
| **数据库层** | 最难 | 一致性哈希只迁 1/N + 双写迁移 + 按时间分表滚动。**Day 1 分片数要够用 5 年**，迁移成本远大于预留成本 |

### 9.1 冷热分层：对抗数据爆炸

数据 10 年涨 1000 倍，但 90% 访问集中在最近 30 天。**冷热分层是必然选择：**

```
内存（秒级）→ Redis（天级）→ MySQL（月级）→ HBase/CK（年级）→ OSS（永久）
成本差 100~1000 倍，按访问频率放对应层级。
```

## 10. 监控运维 —— 不可观测的系统就是黑盒子

### 10.1 三大支柱（缺一不可）

| 支柱 | 工具 | 告诉你什么 |
|------|------|----------|
| **Metrics 指标** | Prometheus + Grafana | 出问题了 |
| **Logs 日志** | ELK / Loki | 哪里出问题 |
| **Traces 链路** | Jaeger / SkyWalking | 怎么出的问题 |

### 10.2 黄金指标：RED + USE

- **RED（服务视角）**：Rate（QPS）/ Error（错误率）/ Duration（P50/P99/P999）
- **USE（资源视角）**：Utilization（CPU/内存/磁盘/带宽）/ Saturation（队列长度）/ Error（错误数）

### 10.3 告警分级（避免告警疲劳）

| 级别 | 响应时间 | 通知 | 典型场景 |
|------|---------|------|---------|
| P0 | 5 min | 电话 | 核心服务挂、数据错、资损 |
| P1 | 15 min | 企微/钉钉 | 单模块异常、lag > 10 万 |
| P2 | 1 hour | 邮件 | 慢查询、磁盘 80% |
| P3 | 次日 | 报表 | 趋势异常、容量预警 |

**黄金原则：每条 P0 告警必须对应明确的 runbook**，否则就是噪音。

### 10.4 故障响应的"黄金五分钟"

**① 止损（1min，执行预案开关）→ ② 定位（5min，看监控/日志/链路）→ ③ 修复（15min，灰度修复或回滚）→ ④ 验证（5min，确认指标恢复）→ ⑤ 复盘（T+1，根因分析）。先止损再定位，是新手和老手的分水岭。**

## 11. 常见架构范式

| 范式 | 核心 | 适用 |
|------|------|------|
| **分层架构** | 经典四层 | 所有系统默认 |
| **事件驱动（EDA）** | 服务间只用事件不 RPC | 极致解耦，调试难 |
| **微服务** | 按业务能力拆分 | 服务边界/治理/追踪是关键 |
| **CQRS** | 读写模型分离 | 读写比 > 10:1 或读维度多 |
| **Lambda** | 批 + 流并行最终合并 | 大数据（离线 + 实时） |
| **存算分离** | 计算无状态 + 存储独立扩展 | Snowflake / Pulsar / TiDB |
| **Serverless** | 按调用付费 | 低频、突发、事件驱动 |

## 12. 方法论心法

### 12.1 好架构的六个特征
**简单**（复杂度只有业务本身）/ **可演进**（今天不阻碍明天）/ **可观测**（问题都能监控到）/ **可容错**（挂了能降级）/ **可扩展**（10 倍流量不改架构）/ **经济**（成本与价值匹配）。

### 12.2 架构师决策框架（每个选择必问）
① 满足 SLA 吗？② 能扛峰值吗？③ 挂了能救吗？④ 明年还用吗？⑤ 成本合理吗？⑥ 团队能维护吗？**六条都过，方案才成立。**

### 12.3 奥卡姆剃刀：如无必要勿增实体
能单机不集群、能单库不分库、能同步不异步、能 HTTP 不 RPC、能无状态不有状态。**复杂度是债务，能不欠就不欠。** 大部分系统崩溃不是因为简单，而是因为过度设计。

### 12.4 最重要的心法：权衡
架构师没有"正确答案"，只有"当下最优解"：CAP 只能选两个、一致性 vs 性能、成本 vs 体验、复杂度 vs 扩展性、短期 vs 长期。**能清晰说出"放弃了什么换来了什么"的架构师，才是真正的架构师。**

### 12.5 最后的真相
架构设计**不是背模板 / 不是堆中间件 / 不是追时髦 / 不是秀复杂度**，而是：**理解物理约束 → 识别业务本质 → 做出合理权衡 → 保留演进空间**。所有"方法论""模式""最佳实践"都是前人踩坑的经验总结，**知其然更要知其所以然**。当你能回答"为什么这样设计"而不是"应该这样设计"时，你就是架构师了。

## 13. 附录：12 章设计文档模板

```
1. 总览 + 10 个关键技术决策表
```
