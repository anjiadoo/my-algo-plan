> **定位**：这不是教程，是心法 + 落地手册。目标：在任何系统设计场景下，既能快速理解陌生业务（上半场），又能推导出扛得住、救得回、扩得开的方案（下半场），并能直接拿去做评审、写方案、带团队。读完这篇你应该能回答三个问题：**面对陌生业务怎么快速建模？为什么所有高并发系统架构长得都差不多？怎么把方法论真的用起来？**
>

## 核心认知：上下两场，缺一不可
| / | 上半场：业务建模 | 下半场：系统架构 |
| :---: | :---: | :---: |
| **解决的问题** | 做正确的事 | 把事情做正确 |
| **核心命题** | 系统存在的业务价值是什么？ | 如何让系统活下去、跑得稳？ |
| **输出物** | 服务边界、领域模型、核心流程 | 容量评估、架构图、容错方案 |
| **技能本质** | 业务抽象能力 | 工程约束能力 |


**只有系统架构 → 扛住了流量，但做出了错误的东西。**  
**只有业务建模 → 逻辑正确，但系统扛不住。**  
**两者缺一不可，而且必须螺旋迭代。**

## 第一章：业务建模 vs 系统架构
**业务建模在前，系统架构在后，但两者是"螺旋上升"的迭代关系，不是一次性的瀑布。**

更准确的说法：**业务建模先行一步，系统架构紧跟其后，两者在第 2~3 轮开始并行，在落地前必须收敛到一致。**

---

**1、为什么业务建模必须在前？**

因为**没有业务模型，架构就没有服务的对象**。

| 如果颠倒顺序会发生什么 |
| --- |
| 先选了 Kafka → 才发现业务不需要异步 → 白选 |
| 先分了 32 库 → 才发现分片键选错 → 重来 |
| 先上了微服务 → 才发现业务边界模糊 → 服务间疯狂调用变巨石 |
| 先做了缓存 → 才发现读模型根本不该这么建 → 缓存失效策略崩盘 |


**本质：架构决策的输入是业务模型。** 没有模型就去选型，等于没看病人就开药方。

---

**2、为什么不能纯串行？**

因为**业务建模到一半就会撞上物理约束，必须架构提前介入**。

典型案例：

+ 业务说"用户能看到全量历史订单" → 建模时发现要存 100 亿行 → 架构师必须介入说"分片键怎么选？冷热怎么分？" → 反过来逼业务改口"只展示最近 3 年，更早走归档查询"
+ 业务说"排行榜实时更新" → 建模发现 QPS 100 万 → 架构师说"ZSet 扛不住" → 反过来逼业务接受"秒级延迟"

**本质：物理约束会反向塑造业务边界。** 不可能先把业务 100% 定下来再去做架构——业务需求本身就被物理世界约束着。

---

**3、正确的节奏：三轮螺旋：**

```plain
第一轮（粗）：业务建模占 80%，架构占 20%
  ├─ 业务：识别原型、画出实体/事件/读模型
  └─ 架构：只做 Fermi 估算（粗略评估数量级），标出"可能撑爆的点"

第二轮（细）：业务 50% + 架构 50% 并行
  ├─ 业务：细化状态机、一致性约束、对账逻辑
  └─ 架构：精细容量推导、选型、分库分表方案
  ↓
  关键动作：两边互相"掰头"
    - 架构说"这个需求扛不住" → 业务要么加钱要么降需求
    - 业务说"这个点不能让步" → 架构必须想办法扛

第三轮（落地）：架构占 80%，业务占 20%
  ├─ 架构：容错、监控、扩展、运维方案
  └─ 业务：Review 边界是否仍然成立
```

---

**4、三句话记住：**

1. **业务建模先开场**，但不要闷头做完再叫架构 —— 第一轮拿到骨架就喊架构师入场。
2. **两者在中段必须博弈**：业务提出想要什么，架构告诉你代价是什么，**博弈的结果才是最终方案**。没博弈过的方案不是架构，是妄想。
3. **落地前必须收敛到一致**：业务模型里的每个实体/事件/约束，都要在架构里找到对应的表/队列/缓存/兜底。**对不上的地方就是 bug 的温床。**

## 第二章：业务建模（上半场）
**核心认知：业务不是"陌生"的，只是"没抽好象"而已。**

任何业务系统，剥开术语的表皮，本质都是通用模型：

```plain
【谁】  对  【什么】 做了 【什么动作】 产生了 【什么结果】
 Actor     Object       Action         Event/State
```

所有业务都是 `Actor × Object × Action × State` 的排列组合。**陌生领域之所以陌生，是因为你停留在"术语层"，而不是在"抽象层"。**

| 业务 | 本质抽象 | 业务 | 本质抽象 |
| --- | --- | --- | --- |
| 红包 | **库存扣减**（有限池子被多人抢） | 推送 | **消息投递**（长连+厂商+必达） |
| 弹幕 | **发布订阅**（一对多广播+时间对齐） | 视频 | **文件切片 + CDN 分发** |
| 评论 | **层级列表**（树形+分页+计数） | Feed | **推拉模型**（写扩散 vs 读扩散） |
| 短链 | **发号器**（ID 生成+KV 查询） | 排行榜 | **有序集合**（ZSet 业务包装） |
| 延时任务 | **调度器**（时间触发+持久化防丢） | 购物车 | **KV 查询 + 状态机** |


**全世界只有 20~30 种业务原型，剩下的都是组合和变形。**

---

### 2.1 业务原型库（架构师的"元素周期表"）
**方法论：拿到陌生需求先问"它最接近哪几个原型的组合？"**

> 快速匹配示例：
>
> + 电商购物车 ≈ KV 查询 + 状态机
> + 直播打赏 ≈ 库存扣减 + 发布订阅 + 计数器
> + 外卖下单 ≈ 工作流 + 状态机 + 消息投递 + 地理位置
> + 滴滴派单 ≈ 地理位置 + 推荐匹配 + 状态机 + 消息投递
> + 抖音 Feed ≈ 推拉模型 + 推荐匹配 + 上传存储
> + 微信转账 ≈ 账本/账户 + 工作流 + 事务协调
> + 美团附近商家 ≈ 地理位置 + 信息检索 + 推荐匹配
> + 广告投放系统 ≈ 配额/限流 + 有序集合 + 数据聚合/报表
>

| 分类 | 原型 | 核心特征 | 关键难点 | 典型案例 |
| :---: | :---: | :---: | :---: | :---: |
| **存储结构型** | **发号器** | 生成全局唯一 ID | 全局唯一、趋势递增、防枚举 | 短链、订单号、雪花 ID |
| | **KV 查询** | 按唯一 key 查 value | 缓存穿透/击穿/雪崩 | 短链跳转、配置中心、Token 校验 |
| | **层级列表** | 树形 + 分页 + 统计 | 深层展开性能、子树迁移、计数维护一致性 | 评论楼层、目录树、组织架构 |
| | **有序集合** | 全局排序 + 范围查询 | 分片、热点 key、分数精度 | 实时排行榜、积分榜、时间线游标分页 |
| **资源竞争型** | **库存扣减** | 有限资源被多人争抢（只减不增） | 超卖、幂等、热点行锁 | 秒杀、票务/座位预订、优惠券领取 |
| | **计数器** | 高频累加 + 查询（无上限） | 热点写、精度 vs 性能（近似统计用 HyperLogLog） | 点赞、播放量、UV/PV |
| | **账本/账户** | 资金/积分有序进出 + 余额 | 并发扣款、金额精度（必须用分/整数）、对账与审计 | 钱包余额、广告账户预算、电商积分 |
| | **配额/限流** | 计量用量 + 有上限管控（只增不减） | 分布式滑动窗口精度、热点 key、窗口重置原子性 | API 限流、短信验证码频控、用户发帖限额 |
| **消息传递型** | **发布订阅** | 一对多广播 | 扇出风暴、消息时效、顺序保证 | 弹幕、群聊消息、股票行情推送 |
| | **消息投递** | 点对点可靠必达 | 在线/离线状态感知、去重、保序 | 推送通知、IM 私信、站内信 |
| | **推拉模型** | 写扩散 vs 读扩散 | 大 V 热点、存储爆炸、混合模式（大 V 读扩散+普通用户写扩散） | Feed 流、朋友圈、订阅更新 |
| | **调度触发** | 定时 + 延时 + 周期执行 | 精度保证、不丢失、不重复触发、故障后恢复 | 订单超时关闭、到期提醒、每日定时报表 |
| **流程编排型** | **状态机** | 有限状态单向流转 | 并发变更竞争（CAS 乐观锁）、非法状态跳转防护 | 订单状态流转、审批流、工单 |
| | **工作流** | 多步骤有向 DAG 编排 | 长事务（跨服务）、Saga 补偿回滚、幂等重试 | 支付链路、开户流程、跨境退款 |
| | **事务协调** | 跨服务/跨库原子性保障 | 2PC/TCC/Saga 选型权衡、死锁检测、超时回滚 | 分布式事务、跨行转账、库存+订单联动 |
| **查询分析型** | **地理位置** | 坐标范围查询 + 距离排序 | GeoHash vs R-Tree 选型、移动物体高频更新、精度与查询性能平衡 | 附近的人/商家、就近派单、POI 搜索 |
| | **信息检索** | 倒排索引 + 相关性排序 | 分词策略、实时 vs 离线索引延迟、相关性调参 | 商品搜索、全文检索、日志查询 |
| | **推荐匹配** | 用户×物品/供需双边打分 | 冷启动、实时特征更新、多样性 vs 相关性；派单子类还需加 ETA/距离约束 | 信息流推荐、猜你喜欢、滴滴司机派单 |
| | **数据聚合/报表** | 多维度离线/近实时汇总 | 实时性 vs 计算成本、数据倾斜、大查询 OLAP 与 OLTP 隔离 | 商家数据看板、广告效果统计、运营 BI |
| **内容处理型** | **上传存储** | 大文件分块 + CDN 分发 | 秒传（Hash 去重）、断点续传、多地 CDN 分发一致性 | 视频上传、网盘、图床 |
| | **审核过滤** | 内容合规检测 | 实时/异步双轨并行、召回率 vs 误判率取舍 | 评论审核、图片鉴黄、语音违规检测 |

### 2.2 建模四问法：名词/动作/查询/不能错
> 顺序不能乱：名词（实体）→ 动作（事件）→ 查询（读模型）→ 约束（不能错）。
>

**① 核心名词是什么？（找实体）**

+ 让业务讲一遍流程，把**名词**全记下，去重合并，筛出**有独立生命周期、能被 CRUD 的东西**。

🌰 红包场景：用户/红包/订单/账户 → 实体 = { User, RedPacket, Order, Account }

---

**② 核心动作是什么？（找事件）**

+ 所谓**动作**就是事件，每个事件对应一次状态变迁 + 一条流水。

🌰 红包动作：发/抢/拆/退款 → Event = { Created, Grabbed, Opened, Refunded }

每个事件三问：**触发者是谁？改变哪个实体状态？需要记流水吗？**（金融必记）

---

**③ 怎么查？（找读模型）**

问产品"用户会从哪几个入口看到这个数据？" 

+ 每个入口 = 一个读模型 = 一张反范式表/Redis 结构。

🌰 红包入口："我发的" → sender_id 索引表；"我抢的" → receiver_id 索引表；"详情" → 主表；"统计" → 预计算快照

**核心洞察：写模型和读模型必须分开设计。** 写：优化一致性，读：优化性能，两者几乎总是冲突。

---

**④ 什么不能出错？（找一致性约束）**

问产品"什么情况算事故？赔钱吗？"

+ 红包金额不平 = 资损 → 金额守恒强一致
+ 抢两次 = 资损 → user_id + packet_id 唯一索引
+ 抢包延迟 3s = 不算事故 → 异步入账可
+ 查询延迟 1min = 不算事故 → 读缓存足够

**不能错的地方用强一致，能容错的地方用最终一致** —— 这就是 CAP 在业务层的体现。

### 2.3 角色视角法：先识角色，再切原型
-- 复杂业务建议从「角色视角法」开始，简单业务直接从「建模四问开始」。

> **原型库告诉你"系统能做什么"，角色视角告诉你"系统为谁做、多个原型如何协同"。没有角色视角，拿到原型也不知道往哪组合。**（角色视角法，是综合自 Use Case Analysis、Event Storming 等方法，结合工程实践整理，也可理解为：Event Storming 的轻量版）
>



**第一步：角色三分法（枚举参与者）**

| 类型 | 特征 | 外卖示例 |
| :---: | :---: | :---: |
| **主动角色** | 主动发起操作的人 / 外部系统 | 用户（下单）、商家（接单）、骑手（取送） |
| **平台角色** | 协调者 / 仲裁者，视角最全、逻辑最复杂 | 平台自身（派单、资金、兜底） |
| **触发角色** | 时间或外部事件触发，无人直接操作 | 定时器（超时关单）、支付回调、第三方通知 |


> 经验值：主动角色 ≤ 4 个；超过 4 个说明场景边界太大，**先拆场景**再建模。
>

****

**第二步：视角切面表（每个角色 × 四列）**

以外卖下单为例：

| 角色 | 触发条件 | 可见 / 关注的数据 | 可执行的操作 | 关联原型 |
| :---: | :---: | :---: | :---: | :---: |
| **用户** | 主动操作 | 自己的订单状态、骑手实时位置 | 下单、备注、取消、评价 | 状态机 + 地理位置 + 消息投递 |
| **商家** | 新订单推送（被动） | 待接 / 在制订单列表、菜品库存 | 接单、拒单、出餐确认 | 状态机 + 消息投递 + 库存扣减 |
| **骑手** | 派单推送 / 主动抢单 | 任务详情、导航路线、收益 | 接受派单、取餐确认、送达 | 推荐匹配 + 地理位置 + 状态机 |
| **平台** | 超时事件 / 支付回调 | 全局订单 + 资金流 + 异常监控 | 自动派单、超时关单、发起退款 | 工作流 + 事务协调 + 调度触发 |


> **空格即权限边界**：角色对某原型"无感知"，即它不能读写该数据域——这是服务鉴权与数据隔离的设计依据。
>

****

**第三步：找角色交汇点（= 系统难点 = 服务边界主战场）**

**什么叫交汇点？** 一个动作发生后，≥ 2 个角色的状态 / 数据必须同时改变。满足以下任一特征即是：

| 特征 | 判断问题 | 对应原型 |
| --- | --- | --- |
| **① 多状态机联动** | 这个动作会触发几个实体的状态同时跳转？ | 状态机 + 事务协调 |
| **② 多数据域写入** | 这个动作要写几个不同角色"拥有"的数据？ | 事务协调 |
| **③ 多角色即时感知** | 这个动作完成后，哪些角色必须立即知道？ | 消息投递 + 发布订阅 |
| **④ 计时器启停** | 这个动作会启动或终止某个超时计时器？ | 调度触发 |


**怎么找？—— 三步扫描法**

```plain
第一步：扫第二步视角切面表的"可执行操作"列
        → 对每个操作问："做完之后，其他角色的哪些数据 / 状态会变？"

第二步：扫核心实体的状态机节点
        → 对每次状态跳转问："这一跳，几个角色会感知到？几条数据要同时写？"

第三步：重点排查三类高频交汇节点
        → 生命周期节点（创建 / 完成 / 取消）
        → 资金 / 资源变动节点（扣款 / 退款 / 扣库存）
        → 异常 / 超时节点（超时关单 / 退款触发 / 兜底补偿）
```

🌰 外卖：扫"骑手-取餐确认"这个操作：

```plain
骑手状态机：  待取餐 → 配送中          ← 特征①
订单状态机：  已出餐 → 配送中          ← 特征①（两个状态机同时跳）
用户推送：    "骑手已取餐" 通知         ← 特征③
平台计时：    记录取餐时间戳（SLA）     ← 特征④（开始配送超时倒计时）
```

**→ 命中特征 ①③④，需要：事务协调 + 消息投递 + 调度触发 同时出现**

> **交汇点是服务设计的决策点**：强一致要求高（资金、状态）→ 合并进同一事务；可接受最终一致（通知、计时）→ 拆出用 MQ 异步。交汇点越多，说明这个业务的服务边界越复杂。
>

****

**第四步：原型 × 角色完整性矩阵（防漏检）**

| 原型 | 用户 | 商家 | 骑手 | 平台 |
| :---: | :---: | :---: | :---: | :---: |
| **状态机** | ✓ 订单状态 | ✓ 商家端状态 | ✓ 骑手任务 | ✓ 全局监控 |
| **消息投递** | ✓ 收通知 | ✓ 收新订单 | ✓ 收派单 | ✓ 异常告警 |
| **地理位置** | ✓ 看骑手位置 | — | ✓ 导航 | ✓ 派单依据 |
| **推荐匹配** | — | — | ✓ 被派单 | ✓ 执行派单 |
| **工作流** | — | — | — | ✓ 全链路协调 |
| **事务协调** | — | — | — | ✓ 资金 + 状态联动 |
| **库存扣减** | — | ✓ 菜品扣减 | — | ✓ 兜底校验 |
| **调度触发** | — | — | — | ✓ 超时关单 |


**口诀：角色定边界，视角找原型，交汇点是难点。**

---

### 2.4 建模三分法（可选）：物理隔离 写/事件/读
**比建模四问法更深一层：所有数据在物理上必须分成三类，混在一张表是所有性能与一致性问题的根源。**

| 类别 | 本质 | 职责 | 对应物 | 选型 |
| :---: | :---: | :---: | :---: | :---: |
| **聚合根（写模型）** | 有独立生命周期的业务对象 | 负责正确性（状态机、强一致） | MySQL 主表 | OLTP 数据库 |
| **领域事件（流水）** | 发生过的动作（只追加） | 负责可追溯（审计、对账、回放） | Kafka + HBase / 流水表 | 顺序写存储 |
| **读模型（视图）** | 为查询优化的反范式视图 | 负责性能（允许冗余、延迟、可重建） | Redis + ES + ClickHouse | 高性能查询 |


**最大误区**：三者混在一张表 →

+ 读写互相影响（统计 SQL 锁住交易）
+ 一致性和性能二选一（要么慢要么脏）
+ 改需求就要改核心表（改字段=高危操作）

**物理隔离是必须的。** 写库追求"对"，事件库追求"全"，读库追求"快"，三者用不同存储天然解决冲突。

> 这一点也是后面 §3.14 CQRS 的理论基础。**先理解三分法 → 自然理解 CQRS。**
>

### 2.5 拆服务：三条黄金线
**不要按功能模块拆**（会拆出贫血服务），按三条线：

| 线 | 判断方法 | 例子 |
| --- | --- | --- |
| **① 数据所有权** | 两个服务都写同一张表 → 必须合并 | 金额池归"红包服务"，"抢包服务"只调用不直接写 |
| **② 一致性边界** | 失败是否必须一起回滚？是→合；否→拆 | 扣金额+写流水（同事务）/ 入账+发通知（拆+MQ） |
| **③ 变化频率** | 发版节奏差 5x 以上 → 必须拆 | 视频播放（核心）vs 推荐（迭代快）必须分 |


**口诀：同主（主权）同事（事务）同频（频率） → 合；异主异事异频 → 拆。**

**反模式**：❌ 按 CRUD 分（忽略业务流）；❌ 按技术栈分（分层不是拆服务）；❌ 无限微服务化（治理爆炸）；✅ 按业务能力拆（Bounded Context，每个服务独立上线）。

### 2.6 流程设计：四条泳道
每个系统必画四条流程：

**① 正常主流程（90% QPS 走这）**

```plain
  用户  |  网关  | 业务服务 | 缓存 | MQ  | DB  | 下游服务
───────┼───────┼─────────┼──────┼─────┼─────┼──────────
请求 ─▶ │ 鉴权─▶ │ 查缓存─▶ │(miss)────▶│     │
       │       │ 写 DB───────────────▶│     │
       │       │ 发 MQ─────────▶│───────────▶消费处理
       │◀─────返回
```

**② 异常路径（下游超时/部分失败/MQ 堆积）**

+ DB 写成功 + MQ 发失败 → 本地消息表补偿
+ DB 写失败 + 前端重试 → 幂等键保证不重复
+ 下游超时 → 熔断 + 降级

**③ 热点路径（最高并发单点，专门优化）**

+ 红包抢包：Lua 原子 + 预拆金额 + 本地队列
+ 热门视频：CDN + 本地缓存 + singleflight
+ 大 V 发微博：读扩散 + 异步写扩散

**④ 对账兜底（数据错了怎么发现怎么修，金融系统生死线）**

+ 日级全量对账脚本（凌晨低峰）→ 差异 > 阈值 → 自动补偿 or P0 告警 → 人工介入口子

### 2.7 库表：从模型到表的映射
**映射规则**

| 模型元素 | 映射到表 | 分片键 | 索引 |
| --- | --- | --- | --- |
| 实体 | 主表（聚合根） | 业务主键 hash | 主键 |
| 状态 | status 字段（单向流转） | - | 按 status 查询索引 |
| 事件 | 流水表（append only） | 按时间 or 实体 ID | 实体 ID + 时间 |
| 读模型 | 反范式表 / Redis 结构 | 按查询维度 | 查询维度 |
| 一致性约束 | 唯一索引 + 乐观锁 version | - | 业务幂等键 UNIQUE |


**分片键选择铁律**

**看最高频的查询 WHERE 条件是什么，它就是分片键。**

+ 红包 90% 按 packet_id 查 → packet_id 分片
+ 订单 70% 按 user_id 查 → user_id 分片
+ 两个维度都高频（如订单按 user 和 merchant 都要查）→ **建两份数据**（空间换时间）

**字段设计通用规范**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | BIGINT | 雪花 ID，全局唯一 |
| biz_key | VARCHAR + UNIQUE | 业务幂等键 |
| status | TINYINT | 状态机字段 |
| version | INT | 乐观锁 |
| create_time / update_time | DATETIME(3) | 毫秒精度 / ON UPDATE 自动更新 |
| extra | JSON/VARBINARY | 扩展字段（避免频繁加字段） |
| deleted | TINYINT | 软删标记（可选） |


**Schema 演进规范**

+ 加字段：可空 + 默认值，**老代码可读老 Schema**
+ 删字段：先停写 → 观察一周 → 再删
+ 改字段类型：新建字段双写 → 数据迁移 → 切读 → 删旧
+ 改索引：低峰期 + Online DDL（pt-osc / gh-ost）

### 2.8 核心关注点：从"不能出错"反推
| 业务担心的"不能错" | 核心关注点 | 标准解法 |
| --- | --- | --- |
| 钱不能错 | 金额守恒、幂等、对账 | 唯一索引 + 日级对账 + 状态机 |
| 不能重复触发 | 幂等键、状态机、去重 | Redis SETNX + DB UNIQUE |
| 不能丢 | 持久化、至少一次 + 幂等消费 | WAL + MQ ack + 幂等消费 |
| 不能慢 | 缓存、异步、分片 | 多级缓存 + MQ 削峰 + 分库 |
| 不能被刷 | 限流、风控、验证码 | 令牌桶 + 设备指纹 + 人机 |
| 不能泄露 | 鉴权、脱敏、审计 | RBAC + 脱敏字段 + 审计日志 |
| 不能被攻击 | 枚举防护、签名、灰度 | 非连续 ID + HMAC + 灰度开关 |


**幂等实现的四种模式**

| 模式 | 原理 | 适用场景 |
| :---: | :---: | :---: |
| **Token 令牌法** | 请求前先获取 token，服务端一次性消费 | 表单提交、前端重试 |
| **业务唯一键 + UNIQUE 索引** | biz_key 唯一索引，重复写 DB 自动拦截 | 最通用，任何写操作 |
| **状态机控制** | 检查当前状态，相同状态跳转返回成功 | 订单状态流转 |
| **Redis SETNX** | 分布式锁防并发，短时高频去重 | 短时并发写、防重复消费 |


> **<font style="color:#000000;">WAL = Write-Ahead Logging，预写日志（顺序写）</font>**
>
> <font style="color:#000000;">核心原则：</font>**<font style="color:#000000;">写数据前，先写日志；日志落盘成功，才执行实际数据写入</font>**<font style="color:#000000;">。</font>
>
> <font style="color:#000000;background-color:rgba(0, 0, 0, 0);">核心工作流程：</font>
>
> 1. <font style="color:#000000;background-color:rgba(0, 0, 0, 0);">业务执行增删改操作（扣款、发奖、库存扣减等关键写操作）；</font>
> 2. <font style="color:#000000;background-color:rgba(0, 0, 0, 0);">先把本次修改的增量日志（修改前/后值、事务 ID、操作类型）持久化写入WAL文件</font><font style="color:#000000;background-color:rgba(0, 0, 0, 0);">；</font>
> 3. <font style="color:#000000;background-color:rgba(0, 0, 0, 0);">WAL日志</font><font style="color:#000000;background-color:rgba(0, 0, 0, 0);">刷盘落盘成功</font><font style="color:#000000;background-color:rgba(0, 0, 0, 0);">，返回客户端「操作成功」；</font>
> 4. <font style="color:#000000;background-color:rgba(0, 0, 0, 0);">后台进程再</font><font style="color:#000000;background-color:rgba(0, 0, 0, 0);">异步、批量</font><font style="color:#000000;background-color:rgba(0, 0, 0, 0);">把WAL日志合并刷写到真正的数据文件（磁盘页 / 索引文件）。</font>
>
> <font style="color:#000000;background-color:rgba(0, 0, 0, 0);">本质是：</font>**<font style="color:#000000;background-color:rgba(0, 0, 0, 0);">先记流水账，再改账本；机器炸了靠流水账复原，从底层硬件层面杜绝核心数据丢失</font>**<font style="color:#000000;">，是金融、游戏道具、账务这类「绝对不能丢」业务的基础存储底座。</font>
>

### 2.9 进阶建模法（复杂业务专用）
**「角色视角法」**和**「建模四问法」**解决 80% 业务，**剩下 20% 高复杂度业务（金融、保险、银行核心）需要更专业的工具**。

---

**1、Event Storming（事件风暴）**

—— 超复杂业务的终极武器

业务复杂到四问法都拆不清时（保险理赔、银行核心）用它。流程（大白板 + 便签纸，2~4 小时）：

:::tips
1. **混沌风暴**：所有人把「业务事件」写便签贴满墙（订单已创建、支付已完成）；
2. **时间线排序**：按时间顺序排成一条线；
3. **找命令和聚合**：找到每个事件前的命令是什么？作用在哪个聚合根上？
4. **找边界**：哪些事件/命令/聚合属于同一业务能力？圈起来 → 服务边界（Bounded Context）
5. **找热点**：哪些地方有争议？红色便签标记 → 这就是设计难点；

:::

**为什么有效**：**事件**是业务中最稳定的元素（实体会变、流程会变，但"发生过什么"不会变）。以事件为锚点倒推的模型比以实体为中心的更稳定。

---

**2、CQRS（命令查询职责分离）**

:::color2
【Command 侧（写）】              【Query 侧（读）】

1️⃣ 强一致 + 复杂事务          		1️⃣ 最终一致 + 高性能

2️⃣ 严格领域模型                          2️⃣ 反范式 + 物化视图

3️⃣ 写入 MySQL                           3️⃣ 读取 ES/Redis/CK

4️⃣ 保证正确性                             4️⃣ 保证体验

:::

**适用**：读写比 > 10:1；读查询维度多；写一致性高但读可容忍延迟。

**典型实现**：写 MySQL → Binlog → Kafka → 同步到 ES/Redis/CK → 查询侧读。

:::tips
🌟 什么是 反范式+物化视图❓

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">范式设计：写库遵循标准设计，拆分多表、字段精简、减少冗余，保证写入强一致、事务准确。</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">反范式：读侧打破分表原则，做</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">字段冗余、宽表合并、数据嵌套</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，放弃精简，只为查询不用多表联查。</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"></font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">1️⃣</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> 普通视图（虚拟视图）：</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">只存 </font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">SQL 逻辑定义</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，不存真实数据。每次查询都</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">实时关联多表、计算、聚合</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，临时算出结果。</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">2️⃣</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> 物化视图：</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">把视图计算后的最终结果，</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">提前落地成一张真实 物理表 / 数据集，持久化存储下来（宽表数据）</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">。不再实时计算，直接查预计算好的成品数据。核心是：</font><font style="color:#AD1A2B;background-color:rgba(0, 0, 0, 0);">预计算 + 落地存储 + 定时 / 增量刷新</font><font style="background-color:rgba(0, 0, 0, 0);">。</font>



**一句话总结：**写侧用范式拆分保证数据正确，**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">读侧用反范式冗余 + 物化视图预计算</font>**，不再现场多表联查，提前合成成品宽表，以空间换查询速度，牺牲强一致换取高性能。

:::

**3、DDD 真正有用的 3 个概念**

+ **① 限界上下文 = 服务边界**：同一词在不同上下文含义不同（订单系统的"用户"是买家，风控系统的"用户"是风险画像）。不要追求全公司统一 User 模型，那是灾难。
+ **② 聚合根 = 事务边界**：一次事务只能修改一个聚合根；跨聚合 → 最终一致（事件驱动）。例：订单聚合 = 订单主表 + 订单项 + 订单地址（同一事务），但不包括用户账户。
+ **③ 领域事件 = 服务间通信核心**：服务间不直接 RPC，用事件。订单发"OrderCreated"，库存/积分/通知等各自订阅，解耦彻底。

> **其他 DDD 术语（Entity/Value Object/Repository/Factory）可以不管，增加心智负担无收益。**
>

**4、四色建模法（金融/交易/合约业务专用）**

| 颜色 | 核心含义 | 一句话记忆 | 典型例子 |
| :---: | :---: | :---: | :---: |
| **事件（粉）** | 有生命周期的**事件/过程** | 先抓「发生了什么事」，所有东西围它转 | 订单、支付流水、退款单 |
| **角色（黄）** | 事件中的**参与角色** | 谁来参与这件事？ | 买家、卖家、风控审核人、渠道方 |
| **实体（绿）** | 客观存在的**人/地/物** | 实实在在的资源，角色的"真身" | 用户、商品、道具、仓库 |
| **规则（蓝）** | **分类/规格/模板/规则** | 给事件/实体定规则的模板 | 订单类型、商品分类、支付渠道 |


**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">🔗</font>****<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> 建模顺序：</font>****<font style="color:rgb(0, 0, 0);background-color:#F1A2AB;">粉</font>****<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> → </font>****<font style="color:rgb(0, 0, 0);background-color:#FCE75A;">黄</font>****<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> → </font>****<font style="color:rgb(0, 0, 0);background-color:#C1E77E;">绿</font>****<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> → </font>****<font style="color:rgb(0, 0, 0);background-color:#81BBF8;">蓝</font>****<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">（为什么必须按这个顺序？）</font>**

:::tips
1. **<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">先定</font>****<font style="color:rgb(0, 0, 0);">「粉色</font>****<font style="color:#AD1A2B;">事件</font>****<font style="color:rgb(0, 0, 0);">」</font>**<font style="color:rgb(0, 0, 0);">：先抓住</font><font style="color:#DF2A3F;">业务的核心流程</font><font style="color:rgb(0, 0, 0);">（比如 “订单下单→支付→发货”），避免建模时本末倒置。</font>
2. **<font style="color:rgb(0, 0, 0);">再补「黄色</font>****<font style="color:#AD1A2B;">角色</font>****<font style="color:rgb(0, 0, 0);">」</font>**<font style="color:rgb(0, 0, 0);">：明确流程中</font><font style="color:#DF2A3F;">需要哪些身份参与</font><font style="color:rgb(0, 0, 0);">（买家、卖家、快递员）。</font>
3. **<font style="color:rgb(0, 0, 0);">绑定「绿色</font>****<font style="color:#AD1A2B;">实体</font>****<font style="color:rgb(0, 0, 0);">」</font>**<font style="color:rgb(0, 0, 0);">：</font>把角色对应到真实的资源上<font style="color:rgb(0, 0, 0);">（买家 = 用户账号，商品 = 资源库存）。</font>
4. **<font style="color:rgb(0, 0, 0);">最后加「蓝色</font>****<font style="color:#AD1A2B;">规则</font>****<font style="color:rgb(0, 0, 0);">」</font>**<font style="color:rgb(0, 0, 0);">：</font><font style="background-color:rgba(0, 0, 0, 0);">给事件和实体加上分类、规则、模板</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">（订单类型、商品类别）。</font>

:::

**5、业务不变式驱动设计**

**从「永远成立的不变式」倒推模型：**

:::color2
+ 红包：`sum(已抢) + sum(剩余) = 初始总额`
+ 订单：`订单金额 = sum(订单项) + 运费 - 优惠`
+ 账户：`借方总额 = 贷方总额`（复式记账）
+ 库存：`可售 + 锁定 + 已售 = 总库存`

:::

**方法**：先写 3~5 条不变式 → 模型设计必须保证它们**永远成立** → 自然推出强一致边界、事件设计、对账逻辑。**这是金融核心系统架构师的核心能力，很少人明说。**

---

**6、元方法论：组织与维度**

**康威定律的反向利用：**

> **系统架构必然反映组织沟通结构。**
>

**反向利用**：想让系统怎么拆，就先让组织怎么分。两个团队共享一个服务 → 必然变巨石；一个团队维护三个服务 → 必然耦合。

陌生领域建模时先问"**未来由几个团队维护？每个团队管什么？**" 团队边界往往就是服务边界的最优解。



**正交分解原则：**

**复杂业务 = 多个正交维度的组合，找到正交维度就找到模型的基。**

+ 电商 = 用户 × 商品 × 订单 × 营销
+ 广告 = 广告主 × 用户 × 展位 × 计费
+ 直播 = 主播 × 观众 × 房间 × 礼物 × 计费

**如果两个维度可独立变化 → 必须拆成独立模型**，耦合了必后悔。

## 第三章：系统架构（下半场）
**核心认知：架构的本质**

**架构不是"设计出来"的，是"算出来 + 逼出来"的。** 架构的每一个决策，都是被物理约束和业务 SLA 联合逼出来的最优解。三条铁律决定了所有架构的形态：

| 铁律 | 具体含义 |
| --- | --- |
| **① 单机物理极限** | MySQL 写 3k~5k TPS；Redis 单分片 10w QPS；万兆网卡 10 Gbps；SSD 随机写 10w IOPS |
| **② 分布式不可靠** | RPC 会超时/丢包/乱序；节点会突然宕机；时钟有漂移 |
| **③ 业务 SLA 是死命令** | 可用性 99.99%（年停 52min）；P99 < 200ms；峰值 = 平均 × 5~100 倍 |


**架构师的工作，就是在这三条铁律的夹缝中，用有限资源做出满足 SLA 的系统。 **所有高并发架构的"套路"不是巧合，是物理决定论的必然结果。

### 3.1 需求澄清（先划定"战场"再打仗）
**90% 的架构失败在需求阶段。**

**需求三问：**

**① 功能边界**：核心业务流程 / 核心实体和动作 / MVP vs nice-to-have。

**② 非功能 SLA**（必须量化）：

| 维度 | 示例量化 |
| :---: | :---: |
| 可用性 | 99.9%（8.76h）/ 99.99%（52min）/ 99.999%（5min） |
| 延迟 | P99 < 200ms / P999 < 1s |
| 吞吐 | 平均 QPS × 峰值系数（3~100x） |
| 一致性 | 强一致 / 最终一致 / 秒级延迟 |
| 数据规模 | 热 30 天 + 温 1 年 + 冷归档 |


**③ 明确禁行**（最容易被忽略但最关键）：例：延时任务禁止 ORDER BY 实时扫描；红包禁止浮点金额；短链禁止自增 ID 暴露；推送禁止对大 V 写扩散；评论禁止 OFFSET 翻页。

:::tips
**为什么"禁行"比"功能"更重要**

功能可以慢慢加，但错误的技术选型一旦上线就是灾难。禁行本质是"我们已经预判了哪条路是坑，不要走"。好的架构师面对需求时，先亮起的不是"我能怎么做"，而是"我绝对不这么做，因为会死"。

:::

**终极三问：**

1. **这个系统存在的理由是什么？**（别设计没用的东西）
2. **如果系统挂一天，谁会骂街？骂多狠？**（决定 SLA）
3. **业务 3 年后会长什么样？**（决定扩展性预留）

### 3.2 容量评估
**没算容量就画架构 = 没看地形就打仗。** 分库因为单库 TPS 不够；缓存因为 DB QPS 不够；CDN 因为回源带宽不够；异步因为同步链路超时；分片因为单机存储不够。

**容量推导六步公式**

```plain
1. 峰值 QPS       = (DAU × 人均次数 / 86400) × 峰值系数
2. 带宽 (Gbps)    = 峰值 QPS × 单请求字节 × 8bit / 10⁹ × 冗余系数(2x)
3. DB 分库数      = 目标 TPS / 3000 → 向上取 2 的幂次
4. Redis 分片数   = 峰值 QPS × (1 - 本地命中率) / 10w → 2 的幂次
5. 服务节点数      = 峰值 QPS / (单机 QPS × 0.7 水位)
6. 存储 (TB)      = 日新增条数 × 单条字节 × 保留天数 / 10¹²
```

**基于 DAU 的 7 步容量评估**

| 步骤 | 计算 / 规则说明 |
| :---: | --- |
| **① DAU → 日均总请求** | 计算公式：日均总请求 = DAU × 人均日请求系数<br/>1. 普通工具 / 基础 App：系数 20～40<br/>2. 内容 / 社交类：系数 40～70<br/>3. 电商常规场景：系数 40～80<br/>4. 短视频（高频滑动）：系数 80～150<br/>5. 电商秒杀 / 大促：系数 100～200 |
| **② 日均请求 → 峰值 QPS** | 标准算法：平均 QPS = 日均总请求 ÷ 86400<br/>峰值 QPS = 平均 QPS × 峰值系数（通用 3～5 倍，极端流量 5～10 倍）<br/>粗算（面试）简化公式：<br/>1. 常规业务：峰值 QPS ≈ DAU × 0.3%～0.8%<br/>2. 电商大促：峰值 QPS ≈ DAU × 1%～3%<br/>3. 秒杀场景：峰值 QPS ≈ DAU × 3%～8% |
| **③ 读写 QPS 拆分** | 1. 通用业务（读多写少）：读：写 = 95:5 ~ 90:10<br/>2. 电商 / 交易类：读：写 = 80:20 ~ 70:30<br/>3. 秒杀场景：写入受库存、锁机制限制，不适用固定比例，写为核心瓶颈 |
| **④ 缓存层计算** | 1. 缓存命中率：核心业务 95%～98%(健康)，98%～99.9%(优秀)；低于90%触发告警<br/>2. DB 穿透读 QPS = 总读 QPS × (1 - 缓存命中率)，穿透率＞5% 需紧急优化<br/>3. 热点 Key 治理：分片拆分 + 本地缓存兜底 + 独立集群部署 |
| **⑤ 数据库层（InnoDB，16 核 32G SSD）** | 安全水位统一控制≤70%<br/>1. 写 TPS：安全≤1500；警戒 1500～2500；危险＞2500（必须分库）<br/>2. 主库读 QPS：安全≤6000；警戒 6000～8000；危险＞8000<br/>3. 读写分离后：读 QPS 可支撑 10000～50000<br/>4. 存储阈值：单库≤200GB；单表≤5000 万行 /≤20GB，<br/>超量考虑分表优化方案：读压力高做读写分离；写压力高做分库；单表数据量大做分表 |
| **⑥ MQ 使用判断** | 满足任意一条，建议接入 MQ：<br/>1. 写入流量峰值波动大，需要流量削峰<br/>2. 存在通知、日志、异步任务等非实时流程<br/>3. 需解耦核心业务链路，避免非核心流程拖慢主链路<br/>4. 业务要求消息重试、死信队列异常处理 |
| **⑦ 服务机器数计算(8核16G)** | 计算公式：机器数 = 核心接口峰值 QPS ÷ (单机 QPS × 0.7)<br/>单机 QPS 参考：<br/>简单接口 500～2000，<br/>复杂逻辑接口 200～500，0.7 为安全冗余系数，预留性能缓冲 |

### 3.3 架构分层
**分层的本质是"隔离变化"。** 不同层的变化频率不同（接入层几年不动 / 应用层天天改 / 存储层变化最慢但代价最高），不隔离就会互相绑架。

**标准四层架构**

| 层 | 职责 | 特点 |
| :---: | --- | --- |
| **接入层** | LVS/Nginx/Gateway：SSL卸载 / 限流 / 鉴权 / 路由 | 无状态、水平扩展 |
| **应用层** | 业务服务集群：业务逻辑、编排 | 无状态、快速迭代、K8s 弹性 |
| **中间件层** | Redis/Kafka/ETCD：缓存、异步、配置、协调 | 有状态、集群化、稳定 |
| **存储层** | MySQL/HBase/OSS：数据持久化 | 强一致、高可靠、迁移成本极高 |


**分层三铁律**

+ **① 下层不依赖上层**：存储不知道谁在用它
+ **② 同层最小耦合**：两个应用服务最好不直接 RPC，用事件
+ **③ 跨层走标准协议**：应用→DB 必走 ORM/DAO，不裸 SQL 四处散

**分层 ≠ 拆服务**：分层是技术维度（物理布局），拆服务是业务维度（能力独立演进），两者正交。分层错了扩不动，拆服务错了跑不快。

**存储选型决策树**

```plain
需要事务/强一致？
  ├─ 是 → MySQL（OLTP 首选）/ TiDB（需要水平扩展时）
  └─ 否 → 继续判断
         ├─ 海量时序/日志/宽列？    → HBase / Cassandra
         ├─ 全文检索/多维查询？     → Elasticsearch
         ├─ 实时 OLAP/大宽表聚合？  → ClickHouse / Doris / RocksDB
         ├─ 高频 KV / 计数 / 排行？ → Redis
         ├─ 向量检索（RAG/推荐）？   → Milvus / pgvector / Qdrant
         └─ 非结构化文件/对象？      → OSS / S3
```

### 3.4 缓存设计
**缓存 = 把高频数据放到更快的存储介质。** ——以空间换时间的艺术

```plain
L1 CPU Cache   纳秒
→ L2 本地内存  微秒
→ L3 Redis     毫秒
→ L4 SSD       10ms
→ L5 HDD       100ms
→ L6 跨区      100ms+
```

**三大经典问题**

| 问题 | 根因 | 解法 |
| :---: | --- | --- |
| **穿透** | 查**不存在**的 key，次次打到 DB | 布隆过滤器 + 空值缓存（短 TTL） |
| **击穿** | 单热点 key 过期瞬间 | singleflight + 互斥锁 + 逻辑过期 |
| **雪崩** | 大量 key 同时过期 / Redis 宕机 | TTL 随机抖动 + 多级缓存 + 熔断降级 |


**四种缓存策略**

> 核心前提：数据库是**唯一数据源**，缓存是**加速副本**；所有问题根源：**写库和更新缓存谁先谁后、更不更新**。
>

| 策略 | 一致性 | 性能 | 适用场景 |
| :---: | :---: | :---: | --- |
| **旁路缓存 Cache-Aside** | 最终一致（秒级） | 高 | 绝大多数互联网业务（**首选**） |
| **写穿透 Write-Through** | 强一致 | 中 | 读多写少、系统配置、静态数据 |
| **延迟写 Write-Back** | 弱最终一致 | 最高 | 点赞、浏览量、热度计数 |
| **Binlog/CDC 异步同步** | 近实时最终一致 | 极高 | 大促核心、多表关联、不改业务代码 |


**旁路缓存核心要点**（90% 业务首选）：

+ 读：先查缓存 → 命中返回；未命中查 DB → 回填缓存
+ 写：**先更新 DB，再删除缓存**（不更新缓存）
+ 为什么删而不更新？并发场景下更新缓存容易被旧数据覆盖造成永久脏数据；删除后下次查询自动拉新数据，更安全。

---

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">1. 旁路缓存 Cache-Aside（业界主流，90% 业务首选）</font>**

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">流程：</font>**

+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">读：应用先查缓存 → 命中返回；未命中查 DB → 回填缓存再返回</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">写：应用</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">先更新 DB，再删除缓存</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">（不更新缓存）</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">核心特性：</font>**

+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">一致性：</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">秒级最终一致</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，存在短暂不一致窗口；靠缓存过期自动兜底修复</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">性能：读写性能都优秀，缓存承接绝大部分读流量</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">优点：代码自主控制、无中间件强依赖、实现简单</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">风险：极端并发下短时脏数据；会触发</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">缓存三大问题</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">（穿透 / 击穿 / 雪崩）</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">适用：</font>**

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">绝大多数互联网业务：用户数据、订单、商品、普通业务表、高并发读写场景</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">补充考点：为什么删缓存而非更新缓存？</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> 并发场景下，更新缓存容易被旧数据覆盖造成永久脏数据；删除后下次查询自动拉新数据，更安全。</font>

---

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">2. 写穿透 Write-Through（Write 穿透）</font>**

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">流程：</font>**

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">应用只操作缓存，</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">缓存组件同步更新 DB</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，两层都写成功才返回。</font><font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);"> （读逻辑同 Read-Through：应用只访缓存，缓存自动回源 DB）</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">核心特性：</font>**

+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">一致性：</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">强一致性</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，缓存与 DB 实时一致</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">性能：一般，写请求双倍 IO，写放大严重，吞吐低</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">优点：业务层完全不用关心 DB，数据绝对统一</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">缺点：写入延迟高、压力大</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">适用：</font>**

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">读多写少、零容忍脏数据、低频变更数据：系统配置、字典、基础静态数据</font>

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">补充：Read-Through（读穿透）一般和 Write-Through 成对出现，国内普通 Redis 极少原生支持，多见于云托管缓存、自研缓存网关。</font>

---

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">3. 延迟写 Write-Back（Write Behind / 写回）</font>**

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">流程：</font>**

+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">写：仅更新缓存，立即返回；数据</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">异步、批量、延迟</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">刷入 DB</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">读：同读穿透，缓存未命中自动回源 DB</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">核心特性：</font>**

+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">一致性：</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">弱最终一致</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，延迟大</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">性能：写入性能拉满，合并批量刷库大幅降低 DB 压力</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">风险：缓存宕机 / 重启，未落地数据</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">永久丢失</font>**

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">适用：</font>**

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">允许少量数据丢失、纯计数类场景：点赞、浏览量、热度、活动计数器</font>

---

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">4. Binlog/CDC 异步同步（工程解耦方案，独立于经典四模式）</font>**

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">流程：</font>**

1. <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">业务代码</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">只读写 DB，完全不操作缓存</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，链路无缓存逻辑</font>
2. <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">中间件（Canal、Debezium 等）监听 DB Binlog，捕获数据增删改</font>
3. <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">异步触发：更新 / 删除缓存</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">核心特性：</font>**

+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">一致性：</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">近实时最终一致</font>**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">，延迟通常毫秒～秒级</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">性能：极高，业务主链路零侵入、无一致性开销</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">优点：业务与缓存解耦，适合多表、多数据源联动</font>
+ <font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">缺点：依赖额外中间件，有一定运维成本；存在异步延迟</font>

**<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">适用：</font>**

<font style="color:rgb(0, 0, 0);background-color:rgba(0, 0, 0, 0);">大促核心数据、多表关联数据、跨库同步、复杂业务场景、不希望改业务代码维护缓存的架构</font>

> **CDC**（Change Data Capture ）：变更数据捕获
>
> **<font style="color:#AD1A2B;background-color:rgba(0, 0, 0, 0);">简单理解：不侵入业务代码，实时监听数据库的数据「增、删、改」，异步同步到别的地方（如ES、Redis）。</font>**
>

**热点 Key 的系统性解法**

| 方案 | 原理 | 适用 |
| --- | --- | --- |
| **本地缓存** | 减少 Redis 请求，热点数据 JVM 缓存 | 读多写少的极端热点 |
| **Key 分片（副本复制）** | 同一 key 复制为 key_0~key_N，分散到多 slot | 读热点 |
| **Lua 原子操作** | check + set 原子化，防超卖 | 写热点/库存扣减 |
| **本地 buffer + 批量写** | 写先聚合到本地，周期性批量写 Redis | 计数类写热点 |
| **独立集群隔离** | 热点 key 单独部署 Redis 实例 | 必须完全隔离的场景 |


**真理：缓存一致性本质是"可接受的不一致窗口"问题。**

**反模式（5 条务必避免）**：

+ ❌ 缓存所有数据（命中率低，浪费内存）
+ ❌ TTL 全一样（同时过期 = 雪崩）
+ ❌ 把缓存当真相（缓存丢了等于数据丢了）
+ ❌ 缓存复杂对象（大 Key + 反序列化慢）
+ ❌ 缓存层未限流（穿透打死 DB）

### 3.5 消息队列
**所有架构都有 MQ 的原因**：

| 作用 | 说明 |
| :---: | --- |
| **削峰** | 前端 50 万 QPS → Kafka → 后端 5 万 TPS，典型 10:1 |
| **解耦** | 服务间通过事件通信，新增消费者不改上游 |
| **最终一致** | 本地事务 + MQ = 跨服务事务，远比 2PC/XA 实用 |


**可靠性三层保障**

+ **生产者**：acks=all + min.insync.replicas=2 + 同步刷盘（关键链路）+ 本地消息表兜底
+ **Broker**：多副本 + 跨可用区 + ISR 防脑裂
+ **消费者**：手动提交 offset + 幂等处理 + 重试队列 + 死信

**消息三语义**

| 语义 | 说明 | 代价 |
| :---: | :---: | :---: |
| At-most-once | 最多一次，可能丢 | 快 |
| At-least-once | 至少一次，可能重（默认） | 中 |
| Exactly-once | 恰好一次 | 贵 |


**工程真理："恰好一次"不存在，只存在 "至少一次" + 幂等 = 等效 "恰好一次"。**

****

**异步链路三铁律（高并发系统必背）**

| 铁律 | 含义 | 后果 |
| --- | --- | --- |
| **① 幂等** | 消息可能重复投递，消费者必须幂等 | 不幂等 = 资损/数据错乱 |
| **② 顺序** | 同一实体的消息必须同分区同消费者 | 乱序 = 状态机崩盘（先收到"已支付"后收到"已创建"） |
| **③ 兜底** | 消息可能丢，必须有对账机制 | 没兜底 = 数据永久不一致 |


**记不住三铁律的工程师，做出来的异步系统都会在某天某夜某次故障中爆炸。**

**堆积应急 & 选型口诀**

+ **应急优先级**：① 先扩消费者（最快）→ ② 降级非核心 → ③ 最后扩分区（rebalance 有成本）
+ **选型**：Kafka（日志/大数据/高吞吐）/ RocketMQ（事务/金融/延时）/ Pulsar（存算分离/多租户）

---

### 3.6 同步 vs 异步：核心权衡
| 必须同步 | 可以异步 |
| :---: | :---: |
| 用户要立即看到结果 | 用户不关心细节（通知/统计） |
| 失败必须回滚 | 失败可以补偿 |
| 强一致要求 | 最终一致可接受 |
| 金额相关关键事务 | 审计、日志、索引构建 |


**原则：核心链路尽量短、尽量同步；非核心全部异步。**

**同步链路的"极简主义"**：高并发系统核心链路只做三件事：

+ **① 鉴权（能不能做）**
+ **② 校验（做得合理吗）**
+ **③ 落地（改状态 + 发事件）**

其他全部异步：发通知 → MQ；更新统计 → MQ；建索引 → Binlog；风控分析 → 日志。这样同步链路 P99 可压到 50ms 以内。

### 3.7 分布式事务选型
跨服务写操作的一致性是架构师绕不开的难题，按业务对一致性的容忍度选：

| 方案 | 一致性 | 性能 | 业务侵入 | 适用 |
| :---: | :---: | :---: | :---: | :---: |
| **本地消息表** | 最终一致 | 高 | 低 | **互联网 99% 首选** |
| **TCC** | 最终强一致 | 中 | 高（需实现 Try/Confirm/Cancel） | 金融、支付、对性能有要求 |
| **Saga（补偿）** | 最终一致 | 中 | 中（需实现补偿操作） | 长事务、跨多系统流程 |
| **2PC/XA** | 强一致 | 低 | 低 | 传统企业内网，互联网**极少用** |


> **明确推荐**：**互联网业务直接选本地消息表**，除非有金融级强一致要求才考虑 TCC。Saga 用于步骤特别多（>5 步）的长事务编排。2PC 在互联网场景几乎没有正确使用的理由。
>

**本地消息表实现要点**：

:::tips
1. 写业务 DB + 写消息表 → **同一本地事务**（关键）
2. 后台轮询任务扫描消息表 `status=pending` 的记录
3. 投递 MQ → 投递成功标记 `status=sended`
4. 消费侧幂等处理（biz_key + UNIQUE）
5. 长时间未投递成功 → 告警人工介入

:::

### 3.8 容错设计
> 假设一切都会失败：网络会抖、机器会挂、磁盘会坏、时钟会漂、依赖会慢、数据会丢。
>

**容错五件套**

| 方案 | 作用 | 关键点 |
| :---: | --- | --- |
| **限流** | 保护自己不被打死 | 分层（网关/服务/依赖）+ 令牌桶/漏桶/滑动窗口 + QPS/并发/连接 |
| **熔断** | 保护下游不被拖死 | 错误率 > 阈值 → 开启 → 半开探测 → 恢复/继续 |
| **降级** | 优先保核心 | 一级关非核心 → 二级简化响应 → 三级只保核心 |
| **隔离** | 故障不扩散 | 线程池 / 机器 / 租户隔离 |
| **超时** | 不等无意义的等待 | 每层都设 + 下层 < 上层 + 整体 < 用户耐心 |


**限流算法对比**

| 算法 | 原理 | 优点 | 缺点 | 适用 |
| --- | --- | --- | --- | --- |
| **固定窗口** | 单位时间内计数 | 简单 | 窗口边界突刺 | 粗粒度场景 |
| **滑动窗口** | 滑动计数，消除边界 | 平滑 | 内存占用稍高 | 通用限流 |
| **漏桶** | 固定速率流出 | 绝对平滑 | 突发流量被削平，不灵活 | 恒定速率输出 |
| **令牌桶** | 恒速生产令牌，可突发消费 | 允许合理突发 | 实现稍复杂 | **互联网首选** |


**降级的艺术：好降级 vs 烂降级**

| 维度 | 好降级 | 烂降级 |
| --- | --- | --- |
| **用户感知** | 无感或可接受（推荐没了还有默认 feed） | 大量报错、白屏 |
| **数据状态** | 不脏（只读降级、跳过非核心） | 数据错乱、需手工修数据 |
| **恢复方式** | 开关立即生效 | 需要发版才能恢复 |
| **兼容性** | 数据结构兼容旧逻辑 | 数据结构破坏，无法回滚 |


**判断标准三句话**：

1. **降级开启时，用户能否完成最核心动作？**（电商：还能下单；视频：还能播放）
2. **降级开启时，数据会不会错？**（不能错就是好降级）
3. **降级关闭时，能否秒级恢复？**（动态开关 + 不重启）

**动态开关（ETCD 的真正用途）**：所有容错参数必须在 ETCD/Apollo/Nacos 里（秒级生效）：限流阈值 / 降级开关 / 路由规则 / 熔断参数。**硬编码的容错策略 = 没有容错策略**，故障来时改代码发版已经晚了。

**灾备层级**

| 层级 | 恢复时间 | 层级 | 恢复时间 |
| --- | --- | --- | --- |
| 单机故障 | 秒级（LB 摘除） | 同城故障 | 小时级（跨城切换） |
| 单机房故障 | 分钟级（切流） | 异地故障 | 小时~天（异地多活） |

### 3.9 可扩展性 & 数据迁移
**扩展性 = 不停服应对 10 倍增长。**

| 扩展层 | 难度 | 关键手法 |
| :---: | --- | --- |
| **服务层** | 最容易 | 无状态 = 水平扩展 + K8s 一键 |
| **缓存层** | 中等 | Redis Cluster 在线 reshard + MOVED 重定向。**Day1 分 32+ 分片** |
| **数据库层** | 最难 | 一致性哈希只迁 1/N + 双写迁移 + 按时间分表滚动。**Day1 分片数要够用 5 年** |


**冷热分层：对抗数据爆炸**

数据 10 年涨 1000 倍，但 90% 访问集中在最近 30 天。**冷热分层是必然选择**：

```plain
内存（秒级）→ Redis（天级）→ MySQL（月级）→ HBase/CK（年级）→ OSS（永久）
成本差 100~1000 倍，按访问频率放对应层级。
```

****

**数据迁移：最难的工程问题**

**双写迁移五步法**（不停服换库/换表/换分片键的标准范式）：

```plain
Step 1：建新库/新表/新分片，影子表先跑通
Step 2：开启双写（新库 + 老库都写），新库写失败不阻塞业务
        → 监控双写一致性，差异 > 阈值告警
Step 3：存量数据迁移（按时间分批/按 ID 分批）
        → 迁移完做全量校验（diff 工具）
Step 4：灰度切读（1% → 10% → 50% → 100%）
        → 每一档观察 24h，数据/性能/错误率全绿才升档
Step 5：切写（双写 → 单写新库）
        → 老库保留一周作为回滚兜底 → 一周后下线
```

**迁移期间必备**：

+ **双写一致性校验脚本**（实时 diff）
+ **全量数据 diff 工具**（按主键对比）
+ **回滚开关**（任何阶段一键回到上一档）
+ **数据修复脚本**（差异自动补偿）

**迁移踩坑清单**：

+ ❌ 双写时只写不读校验 → 迁移完才发现数据全错
+ ❌ 没有灰度切读 → 一刀切炸雷
+ ❌ 老库立刻下线 → 出事无法回滚
+ ❌ 迁移期间业务变更 Schema → 双写直接挂
+ ✅ 迁移期间冻结相关业务的 Schema 变更

### 3.10 监控运维
**不可观测的系统就是个"黑盒子"。**

**三大支柱（缺一不可）**

| 支柱 | 工具 | 告诉你什么 |
| :---: | :---: | :---: |
| **Metrics 指标** | Prometheus + Grafana | 出问题了 |
| **Logs 日志** | ELK / Loki | 哪里出问题 |
| **Traces 链路** | Jaeger / SkyWalking / OpenTelemetry | 怎么出的问题 |


**黄金指标：RED + USE**

+ **RED（服务视角）**：Rate（QPS）/ Error（错误率）/ Duration（P50/P99/P999）
+ **USE（资源视角）**：Utilization（CPU/内存/磁盘/带宽）/ Saturation（队列长度）/ Error（错误数）

**TraceId 透传：日常排障的生命线**

每个请求必须有唯一 TraceId，**全链路透传**才能拼出完整调用链。透传规则：

| 通道 | 透传方式 | 工具 |
| --- | --- | --- |
| **HTTP/RPC** | Header（如 `X-Trace-Id`） | OpenTelemetry SDK 自动注入 |
| **MQ 消息** | 消息属性/Header 字段 | 生产者塞入，消费者取出后注入本地上下文 |
| **异步线程池** | 通过 ThreadLocal/Context 拷贝 | 装饰 Runnable，线程切换时复制 TraceId |
| **跨进程协程** | 显式传参 or Context 包装 | 不能依赖 ThreadLocal，必须显式 |
| **定时任务** | 任务启动时生成新 TraceId | 加上 task_id 标识 |


**实战要求**：

+ 日志必须带 TraceId（`logger.info("xxx", traceId)`）
+ 错误堆栈必须带 TraceId（方便从告警一键跳转链路）
+ 慢查询/慢接口告警必须带 TraceId（直接定位）
+ **没有 TraceId 透传的系统 = 出事故只能盲猜**

**告警分级**

| 级别 | 响应时间 | 通知 | 典型场景 |
| --- | --- | --- | --- |
| P0 | 5 min | 电话 | 核心服务挂、数据错、资损 |
| P1 | 15 min | 企微/钉钉 | 单模块异常、lag > 10 万 |
| P2 | 1 hour | 邮件 | 慢查询、磁盘 80% |
| P3 | 次日 | 报表 | 趋势异常、容量预警 |


**黄金原则：每条 P0 告警必须对应明确的 runbook**，否则就是噪音，要避免告警疲劳。

**故障响应的"黄金五分钟"**

| 阶段 | 时长 | 动作 | 责任人 |
| --- | --- | --- | --- |
| **① 止损** | 1 min | 执行预案开关（限流/降级/切流） | 值班 SRE |

### 3.11 多活架构（高可用的天花板）
**单机房再可靠，也扛不住光缆挖断 / 机房断电 / 区域网络故障。** 业务要求 99.99% 以上可用性，必须多活。

**多活演进路径**

```plain
单机房                      → 单点风险，99.9%
  ↓
同城双活（同城两机房）         → 抗机房故障，99.95%
  ↓
两地三中心（同城双活+异地灾备） → 抗城市故障，99.99%
  ↓
异地多活（三地多机房）         → 抗区域故障，99.999%
  ↓
单元化架构（按用户维度拆单元）  → 大厂终极形态
```

**多活的核心难题：写冲突**

读多活简单（任何机房都能读）；**写多活难**——同一个用户在两个机房同时下单怎么办？

**主流方案**

| 方案 | 核心思想 | 适用 |
| --- | --- | --- |
| **主备切换** | 一个写中心，故障切换 | 中小业务，RTO 分钟级 |
| **按用户维度路由（单元化架构）** | 用户被路由到固定单元写入 | 大厂主流（阿里、字节） |
| **按业务维度拆分** | 不同业务在不同机房写 | 业务能切分干净的场景 |
| **多写 + CRDT/最终一致** | 允许冲突，数据自动合并 | IM、协同编辑等 |


**单元化架构核心要点**

```plain
路由层（接入层）：
  按 user_id % N → 路由到对应单元
  → 用户的所有请求都在同一单元闭环

单元内部：
  独立的 应用 + 缓存 + 数据库
  → 单元内强一致

跨单元同步：
  通过 Binlog / 消息 异步同步全量数据
  → 故障时可以快速接管
```

**多活避坑**：

+ ❌ 全局序列号：单元化后会冲突，必须改雪花 ID 等分布式 ID
+ ❌ 跨单元强一致：单元化的本质就是"放弃跨单元强一致换可用性"
+ ❌ 单元间互相 RPC：性能崩盘，必须单元闭环
+ ✅ 数据同步延迟监控：跨单元同步慢于阈值即告警

### 3.12 接口设计与版本兼容
**接口是服务的契约，契约一变全员遭殃。**

**RPC vs HTTP 选型**

| 场景 | 推荐 |
| --- | --- |
| 内部服务间高频调用 | gRPC / Thrift / Dubbo |
| 对外开放/三方集成 | RESTful HTTP + JSON |
| 流式/双向通信 | gRPC streaming / WebSocket |
| 浏览器直连 | HTTP / GraphQL |


**接口设计五原则**

1. **幂等**：同一请求重复调用结果一致（带 biz_key）
2. **向后兼容**：加字段允许，删字段/改语义禁止
3. **版本化**：URL 带版本（`/v1/orders`）或 Header 带版本
4. **错误码标准化**：`{code, message, data, traceId}` 统一格式
5. **超时/重试约定**：服务端声明最大耗时，客户端按此设超时

**字段演进规范**

| 操作 | 是否兼容 | 注意 |
| --- | --- | --- |
| 加字段（可空） | ✅ 兼容 | 老客户端忽略，无影响 |
| 加字段（必填） | ❌ 不兼容 | 必须升版本 |
| 删字段 | ❌ 不兼容 | 必须先 deprecated 一个版本周期 |
| 改字段类型 | ❌ 不兼容 | 必须新增字段，老字段保留 |
| 改字段语义 | ❌ 灾难性 | 必须新建字段，绝不复用 |
| 加枚举值 | ⚠️ 慎用 | 老客户端可能 case 不到，需默认分支兜底 |


**API 治理的 3 个工具**：

+ **OpenAPI/Swagger**：契约即文档，前后端基于契约并行开发
+ **gRPC Protobuf**：编译期检查兼容性
+ **API 网关**：统一鉴权/限流/版本路由/灰度

### 3.13 AI 时代架构补充（2026 必备）
大模型应用已经成为主流业务，**架构师必须掌握 AI 推理服务的特殊性**。

**AI 应用的架构差异**

| 维度 | 传统业务 | AI 应用 |
| --- | --- | --- |
| **延迟** | P99 < 200ms | 首 token < 1s，全程 5~30s |
| **吞吐瓶颈** | CPU/IO | GPU 显存 + 算力 |
| **状态** | 多无状态 | 长上下文 / 多轮会话有状态 |
| **成本** | 接近线性 | Token 计费，单请求成本可能 $ 0.01~ $1 |
| **失败率** | < 0.1% | 模型幻觉 / 超长上下文截断 / 限流 5%~10% |


**核心组件清单**

| 组件 | 职责 | 选型 |
| --- | --- | --- |
| **推理服务** | 模型推理执行 | vLLM / TGI / TensorRT-LLM |
| **网关层** | 路由、限流、计费、鉴权 | LiteLLM / 自研 |
| **向量数据库** | Embedding 检索（RAG） | Milvus / pgvector / Qdrant |
| **会话存储** | 多轮上下文 | Redis（短期）+ MySQL（长期） |
| **Prompt 管理** | 模板版本化、AB | 自研模板系统 |
| **可观测** | Token 用量、推理耗时、命中率 | Langfuse / 自研 |


**AI 架构 6 个特殊关注点**

1. **流式输出（SSE/WebSocket）**：必须流式返回首 token，否则用户等不住
2. **Prompt Cache（提示缓存）**：长 system prompt 必须开启缓存，成本降 50%~90%
3. **限流维度多样**：QPS / 并发 / Token/min / GPU 占用 / 单用户配额
4. **异步任务化**：耗时 >10s 的请求改成"任务 ID + 轮询/Webhook"模式
5. **降级策略**：大模型挂了 → 切小模型；小模型挂了 → 切规则/缓存
6. **多模型路由**：根据复杂度路由到不同模型（简单问题用 Haiku，复杂用 Opus）

**RAG 架构标准范式**

```plain
用户问题
  ↓
Embedding 模型 → 向量
  ↓
向量数据库 检索 Top-K（语义相似）
  ↓
Rerank 重排（可选，提升相关性）
  ↓
拼接 Prompt（system + 检索结果 + 用户问题）
  ↓
LLM 推理（开启 Prompt Cache）
  ↓
流式返回
```

**AI 时代的"幂等"问题**：模型输出本身有随机性，**业务侧必须保证调用幂等而非输出幂等**——同一请求多次调用扣一次 Token、产出可缓存复用。

### 3.14 CQRS（命令查询职责分离）
> 这是 §2.4 建模三分法在架构层面的落地形态。
>

```plain
【Command 侧（写）】              【Query 侧（读）】
① 强一致 + 复杂事务               ① 最终一致 + 高性能
② 严格领域模型                   ② 反范式 + 物化视图
③ 写入 MySQL                    ③ 读取 ES/Redis/CK
④ 保证正确性                     ④ 保证体验
```

**适用**：读写比 > 10:1；读查询维度多；写一致性高但读可容忍延迟。

**典型实现**：写 MySQL → Binlog → Kafka → 同步到 ES/Redis/CK → 查询侧读。

> **反范式 + 物化视图**：写库遵循范式拆分保证正确性；读侧打破分表原则，字段冗余宽表合并，把计算结果提前落地成真实物理表（预计算 + 落地存储 + 定时/增量刷新）。以空间换查询速度，牺牲强一致换取高性能。
>

### 3.15 常见架构范式
| 范式 | 核心 | 适用 |
| :---: | --- | --- |
| **分层架构** | 经典四层 | 所有系统默认 |
| **事件驱动（EDA）** | 服务间只用事件不 RPC | 极致解耦，调试难 |
| **微服务** | 按业务能力拆分 | 服务边界/治理/追踪是关键 |
| **CQRS** | 读写模型分离 | 读写比 > 10:1 或读维度多 |
| **Lambda** | 批 + 流并行最终合并 | 大数据（离线 + 实时） |
| **Kappa** | 全流式（流即批） | 实时优先的大数据 |
| **存算分离** | 计算无状态 + 存储独立扩展 | Snowflake / Pulsar / TiDB |
| **Serverless** | 按调用付费 | 低频、突发、事件驱动 |
| **Service Mesh** | 治理能力下沉到 Sidecar | 大规模微服务 |

### 4.2 下半场：系统架构
**Step 8 容量评估**

```plain
日订单 1000 万，午晚高峰 4h 承载 700 万 → 平均 486 单/秒
峰值系数 5x → 峰值下单 ≈ 2500 单/秒
读：写 = 20:1（用户刷状态、商家看单、骑手看单）→ 峰值读 5 万 QPS

DB 写：2500 TPS → 单库勉强扛，分 4 库留余量
DB 读：5 万 QPS → 读写分离 + 8 个只读从库
Redis：5 万 QPS × (1-95%) 缓存命中 = 2500 穿透 → 单分片够，开 4 分片高可用
应用节点：5 万 / (2000 × 0.7) ≈ 36 台
存储：3 年 1000万×3×365×2KB ≈ 22TB → 90 天热表 + 历史归档 OSS
```

**Step 9 架构分层 + 关键中间件**

+ 接入层：网关（鉴权、限流 5 万 QPS）
+ 应用层：8 个微服务，K8s 部署
+ 中间件：Redis 4 分片 / **RocketMQ**（事务消息）/ Apollo 配置中心 / Nacos 服务发现
+ 存储：MySQL 4 分库 + ES（订单搜索）+ HBase（流水归档）+ OSS（图片）

**Step 10 缓存设计**

+ 订单详情：Cache-Aside，TTL 10min ± 随机抖动
+ 骑手位置：Redis GEO，5 秒上报一次
+ 商家菜单：Cache-Aside + 本地缓存（极少变更）
+ 配送状态：Write-Through（保证骑手看到最新）

---

**Step 11 消息队列（RocketMQ）**

**为什么用事务消息？** 

订单状态变更要求"DB 写入"与"消息发布"原子——DB 提交但消息没发 = 下游永远不知道；消息发了但 DB 回滚 = 下游误触发。Kafka 不支持事务消息，RocketMQ 原生支持。

**事务消息两阶段流程（以支付回调触发 **`order.paid`** 为例）：**

```plain
订单服务
  ① 发"半消息"（Prepare）→ RocketMQ Broker  ← 消费者此时不可见
  ② 执行本地事务：DB 更新订单状态 = 已支付
  ③ 成功 → Commit（消息对消费者可见）
     失败 → Rollback（消息丢弃）
  ④ 若宕机未 Commit/Rollback：
     RocketMQ 定时回查 → 订单服务查 DB 状态 → 补偿 Commit/Rollback
```

**Topic 设计（统一 **`order_lifecycle`** Topic + Tag 过滤）：**

| Tag | 生产者 | 消费者组 | 是否事务消息 |
| :---: | :---: | :---: | :---: |
| `PAID` | 订单服务 | merchant_group / dispatch_group / notify_group | ✓ |
| `ACCEPTED` | 商家服务 | dispatch_group / notify_group | ✓ |
| `DISPATCHED` | 配送服务 | notify_group | ✓ |
| `DELIVERED` | 配送服务 | order_group（触发结算）/ notify_group | ✓ |


**可靠性保障：**

+ 顺序消息：同一 order_id hash 到同一 Queue，保证单订单消费有序
+ 消费失败：自动重试 3 次 → 进死信 Topic（DLQ）→ 告警人工介入
+ 幂等消费：消费者按 `msgId + biz_key` 去重，防止重试重复写库

---

**Step 12 容错**

+ 限流：网关 5 万 QPS + 单接口令牌桶
+ 熔断：商家服务挂 → 订单进"待商家接单"队列，定时重试
+ 降级：营销服务挂 → 跳过优惠，订单按原价创建（不阻塞下单）
+ 隔离：支付服务独立机器池（资源隔离）
+ 多活：同城双活 + 用户按 user_id 路由到固定机房

+ TraceId 全链路（网关注入，RocketMQ/异步线程透传）
+ RED 指标：每个服务的 QPS / 错误率 / P99
+ 关键告警：P0 = 支付失败率 > 1% / 派单延迟 > 30s

                              [Redis × 4]          [RocketMQ（事务消息）]
                                  ↓                       ↓
                              [MySQL × 4分库]       [HBase / ES / 风控]
                                  ↓
                              [OSS 冷数据 / 历史归档]

## 终极心法
**好架构的六个特征：**

> **简单**（复杂度只有业务本身）/ **可演进**（今天不阻碍明天）/ **可观测**（问题都能监控到）/ **可容错**（挂了能降级）/ **可扩展**（10 倍流量不改架构）/ **经济**（成本与价值匹配）。
>

**架构师决策框架（每个选择必问）：**

> ① 满足 SLA 吗？② 能扛峰值吗？③ 挂了能救吗？④ 明年还用吗？⑤ 成本合理吗？⑥ 团队能维护吗？**六条都过，方案才成立。**
>

**奥卡姆剃刀：如无必要勿增实体**

> 能单机不集群、能单库不分库、能同步不异步、能 HTTP 不 RPC、能无状态不有状态。**复杂度是债务，能不欠就不欠。** 大部分系统崩溃不是因为简单，而是因为过度设计。
>

**最重要的心法：权衡**

> 架构师没有"正确答案"，只有"当下最优解"：CAP 只能选两个、一致性 vs 性能、成本 vs 体验、复杂度 vs 扩展性、短期 vs 长期。**能清晰说出"放弃了什么换来了什么"的架构师，才是真正的架构师。**
>

**三句话总结两章心法**

> **上半场**：业务本质是【谁】对【什么】做了【什么动作】产生了【什么结果】。术语会变，结构不变。陌生业务：先匹配原型 → 按四问法建模 → 30 分钟出框架。
>
> **下半场**：架构不是"设计出来"的，是"算出来 + 逼出来"的。所有高并发架构的"套路"不是巧合，是物理决定论的必然结果。
>
> **两者关系**：业务建模定"做什么"，系统架构定"能不能做、怎么做、挂了怎么办"。先有骨（业务），再有肉（架构），中间必须反复对齐。谁先谁后不重要，**谁都不能单飞才重要**。
>

## 附录 B：架构反模式 30 条速查
> 反模式比正模式更值钱——做对的收益是线性的，犯错的代价是指数的。
>

**B.1 数据存储类（10 条）**

1. ❌ **浮点存金额** → 必须用 BIGINT（分）或 DECIMAL
2. ❌ **自增 ID 暴露** → 用雪花 ID/UUID，否则被枚举爬取
3. ❌ **OFFSET 翻页** → 深翻页性能崩盘，必须游标分页（WHERE id > last_id）
4. ❌ **大字段塞主表** → 大字段（detail/content）单独建表
5. ❌ **TEXT 字段建索引** → 用前缀索引或全文索引
6. ❌ **跨库 JOIN** → 分库后必须冗余字段或多次查询
7. ❌ **分片键选错** → Day1 选最高频查询条件，错了重做
8. ❌ **单表过亿不分** → 1 亿是红线，索引/查询/迁移全崩盘
9. ❌ **删字段直接 DDL** → 必须先停写观察一周
10. ❌ **没有软删** → 误删无法恢复，金融业务必须软删 + 审计

****

**B.2 缓存类（5 条）**

11. ❌ **缓存所有数据** → 命中率低反而拖慢，只缓存热数据
12. ❌ **TTL 全一样** → 雪崩根源，必须随机抖动 ±10~30%
13. ❌ **把缓存当真相** → 缓存丢了等于数据丢了
14. ❌ **缓存复杂对象** → 大 Key（>10MB）+ 反序列化慢
15. ❌ **更新缓存而非删除** → 并发下脏数据

****

**B.3 异步与一致性类（5 条）**

16. ❌ **没有幂等** → MQ 重复投递必然出现，没幂等就是数据错乱
17. ❌ **跨分区指望顺序** → 同一实体必须同分区
18. ❌ **本地事务 + 远程调用混用** → 用本地消息表
19. ❌ **2PC/XA 在互联网** → 性能崩盘，几乎没有正确使用场景
20. ❌ **没有对账兜底** → 异步系统必有数据不一致，没对账就是定时炸弹

****

**B.4 服务设计类（5 条）**

21. ❌ **按 CRUD 拆服务** → 拆出贫血服务，按业务能力拆
22. ❌ **同层服务互相 RPC** → 用事件驱动
23. ❌ **共享 DB 多服务写** → 数据所有权混乱，必合并
24. ❌ **无限微服务化** → 治理爆炸，3~5 个服务先做对再说
25. ❌ **大 V 写扩散** → Feed 系统对大 V 必须读扩散或分桶

****

**B.5 容错与运维类（5 条）**

26. ❌ **硬编码容错策略** → 必须 ETCD/Apollo 动态开关
27. ❌ **没有限流就上线** → 流量异常瞬间打死
28. ❌ **超时配置上层 < 下层** → 下层算完了上层已超时，浪费资源
29. ❌ **告警没有 runbook** → 告警疲劳 + 定位慢
30. ❌ **故障不复盘** → 同一个坑踩第二次就是工程团队的羞耻

## 附录 C：建模四问法 vs 四色建模法 选型表
| 维度 | 四问法 | 四色建模法 |
| --- | --- | --- |
| **核心顺序** | 名词→动作→查询→约束 | 事件→角色→实体→规则 |
| **适用业务** | 全业务通用 | 金融、交易、合约、理赔 |
| **设计耗时** | 30 分钟出框架 | 半天到一天 |
| **严谨程度** | 中等 | 高 |
| **学习成本** | 低 | 中 |
| **何时切换** | 有资损风险时切到四色 | 业务不涉及金融可用四问 |
| **典型应用** | 评论、IM、订单、内容平台 | 红包、账务、保险、借贷 |


**实战建议**：默认用四问法，遇到金融/资金/合约场景叠加四色建模法。两者不冲突，可以组合使用。

## 最后的真相
> 架构设计**不是背模板 / 不是堆中间件 / 不是追时髦 / 不是秀复杂度**，而是：
>
> **理解物理约束 → 识别业务本质 → 做出合理权衡 → 保留演进空间**。
>
> 所有"方法论"、"模式"、"最佳实践"都是前人踩坑的经验总结，**知其然更要知其所以然**。当你能回答"为什么这样设计"而不是"应该这样设计"时，你就是架构师了。
>

> **系统架构设计 = 用有限资源，对抗无限复杂度，在物理约束和业务 SLA 之间找到那个"活下来且长得大"的解。**
>
