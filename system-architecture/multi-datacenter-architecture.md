## 多地多中心部署架构详解

### 一、为什么需要多数据中心？

```
单数据中心的致命问题：

1. 自然灾害（地震/洪水/火灾）→ 整个服务不可用
2. 市政事故（光缆被挖断/停电）→ 几小时甚至几天中断
3. 物理距离（北京机房服务广州用户）→ 光速限制，延迟 30~50ms
4. 容量天花板（单机房电力/制冷/带宽有限）→ 无法承载亿级用户

国民级应用的 SLA 要求：99.99%+ = 年宕机 < 52 分钟
单数据中心无论怎么做冗余，都无法满足这个要求
```

### 二、几种主流架构模式

```
┌─────────────────────────────────────────────────────────────────┐
│ 模式                     容灾级别     RPO      RTO        成本    │
├─────────────────────────────────────────────────────────────────┤
│ 同城双活                  机房级       0        秒级        ★★★    │
│ 两地三中心                城市级       分钟级   分钟级       ★★★★   │
│ 三地五中心(异地多活)        区域级       0        秒级       ★★★★★  │
├─────────────────────────────────────────────────────────────────┤
│ RPO = Recovery Point Objective (能丢多少数据)                     │
│ RTO = Recovery Time Objective (多久恢复服务)                      │
└─────────────────────────────────────────────────────────────────┘
```

### 三、同城双活（基础款）

#### 原理

两个机房在同一城市（距离 30~80km），通过**专线光纤**互联，延迟 < 2ms，可以做同步复制。

```
                    同城专线（裸光纤，RTT < 2ms）
      ┌──────────────────────────────────────────┐
      │                                          │
┌─────┴─────┐                              ┌─────┴─────┐
│  IDC-A    │                              │  IDC-B     │
│  上海浦东   │                             │  上海嘉定  │
│           │                              │            │
│ App × 200 │                              │ App × 200  │
│ DB Master  │───── 同步复制 ──────────────→ │ DB Slave   │
│ Redis 主   │───── 同步复制 ──────────────→ │ Redis 从   │
│ MQ Broker  │───── 镜像队列 ──────────────→ │ MQ Broker  │
└────────────┘                             └────────────┘
      ↑                                          ↑
      │          ┌──────────────┐                │
      └──────────│ GSLB/DNS智能 │────────────────┘
                 │   解析调度    │
                 └──────────────┘
                       ↑
                    用户请求
```

#### 实施细节

**1. 网络层：**
```
• 两机房之间租用运营商裸光纤或波分复用（DWDM），至少两路冗余
• 带宽：通常 100G~400G 专线
• 延迟：同城 < 2ms（光纤中光速约 200km/ms，60km ≈ 0.6ms 往返）
• 为什么 < 2ms 很关键？→ 数据库同步复制可接受（不影响写入性能）
```

**2. 数据层（最核心的难点）：**
```
MySQL：
  • 主库在 IDC-A，半同步复制（semi-sync）到 IDC-B
  • semi-sync：主库等至少一个从库 ACK 才返回成功
  • 同城 < 2ms 延迟，semi-sync 对写入性能影响 < 5%
  • 故障切换：MHA/Orchestrator 自动 failover，RTO < 30s

Redis：
  • 主从复制（异步），但同城延迟极低，实际 lag < 1ms
  • 或用 Redis Cluster 跨机房分片（部分分片主在 A，部分在 B）

MQ（RocketMQ）：
  • Broker 主在 A，从在 B（同步刷盘 + 同步复制）
  • 或 Dledger 模式，3 节点 Raft（2A + 1B），多数派提交
```

**3. 流量调度：**
```
正常态：
  • 两个机房都接流量，按权重 50:50（或 60:40 主次）
  • GSLB（Global Server Load Balancing）按健康检查动态调整

故障态（IDC-A 整体不可用）：
  ① 健康检查探测到 A 不可达（3s 超时 × 3 次 = 9s）
  ② GSLB/DNS 自动摘除 A 的解析（DNS TTL 设 30s~60s）
  ③ B 提升 DB Slave 为 Master（MHA，< 30s）
  ④ B 承载 100% 流量
  ⑤ 总 RTO ≈ 30s~60s
```

**4. 局限性：**
```
• 只能抗机房级故障，不能抗城市级灾难（地震/洪水整个上海不可用）
• 写入路径有跨机房延迟（虽然同城很小，但高频写仍受影响）
• 不能解决用户就近接入问题（广州用户访问上海仍有 30ms 延迟）
```

### 四、两地三中心（金融行业标配）

#### 原理

```
"两地"：同城 + 异地
"三中心"：同城两个（生产+同城灾备）+ 异地一个（异地灾备）

            同城（< 2ms）                    异地（20~50ms）
┌────────────┐      ┌────────────┐      ┌────────────┐
│ 生产中心    │      │ 同城灾备    │       │ 异地灾备    │
│ 上海浦东    │      │ 上海嘉定    │       │ 深圳       │
│ (Active)   │      │ (Standby)  │      │ (Standby)  │
│            │      │            │      │            │
│ 读写主力    │──同步→│ 热备,可切换 │──异步→ │ 冷/温备    │
│ 100% 写    │      │ 0% 写      │       │ 0% 写      │
│ 60% 读     │      │ 40% 读     │       │ 0% 读(平时) │
└────────────┘      └────────────┘      └────────────┘
```

#### 三层容灾能力

```
Level 1：生产中心单机/单架故障
  → 机房内冗余自动处理，用户无感

Level 2：生产中心整体不可用
  → 切换到同城灾备，RTO < 1min，RPO = 0（同步复制，不丢数据）

Level 3：整个城市不可用（地震等）
  → 切换到异地灾备，RTO < 30min，RPO = 分钟级（异步复制，可能丢几分钟数据）
```

#### 实施细节

**1. 数据复制拓扑：**
```
                    生产中心 (上海浦东)
                    MySQL Master
                         │
          ┌──────────────┼──────────────┐
          │ 同步复制      │ 异步复制      │
          ↓              ↓              ↓
    同城灾备Slave    本地Slave×2     异地灾备Slave
    (上海嘉定)      (读分担)         (深圳)
    
  同步复制：semi-sync，延迟 < 2ms，保证 RPO=0
  异步复制：延迟 20~50ms，高峰期 lag 可能到秒级
```

**2. 异地灾备为什么不能同步？**
```
上海→深圳 物理距离 ≈ 1400km
光纤中光速 ≈ 200,000 km/s
单程延迟 ≈ 1400/200000 = 7ms
往返 RTT ≈ 14ms（理论最小值）
实际经路由器/交换机 ≈ 20~30ms

如果数据库做同步复制：
  每次写入都等 30ms ACK → 写 TPS 从 5万降到 < 3000
  → 完全不可接受

所以异地只能异步复制，代价是灾难时可能丢几秒~几分钟数据
```

**3. 切换流程（以城市灾难为例）：**
```
① 监控系统多维度判定（网络/应用/数据库都不可达，持续 5 分钟）
② 人工决策确认（通常需要 VP 级审批，因为可能丢数据）
③ 异地灾备 DB 提升为 Master
④ 检查异步复制 lag，评估丢失数据量
⑤ DNS/GSLB 全量切换到异地
⑥ 通知业务方进行数据一致性校验
⑦ 恢复服务

总 RTO：20~30 分钟（大量人工介入）
RPO：取决于最后一次同步成功的时间点
```

**4. 局限性：**
```
• 异地灾备平时"闲置浪费"（冷备不跑流量，几千万硬件投入无产出）
• 切换需人工决策（不敢自动切，怕脑裂/误切丢数据）
• 异地 RPO > 0（业务不可接受丢数据时需要额外方案）
• 不能解决用户就近接入（所有写流量必须到生产中心）
```

### 五、异地多活（微信/抖音/支付宝的做法）

#### 核心思想

**不再有"主/备"的概念，每个数据中心都同时对外提供完整服务。**

```
关键突破：
  两地三中心 = "一个写，其他备着" → 资源浪费 + 切换慢
  异地多活   = "每个中心都能读写" → 充分利用 + 无需切换
  
但代价是：数据分片 + 路由规则 + 一致性妥协
```

#### 架构全貌（以三地五中心为例）

```
          北京（华北用户）         上海（华东用户）         广州（华南用户）
      ┌──────┐ ┌──────┐      ┌──────┐ ┌──────┐      ┌──────┐
      │ BJ-1 │ │ BJ-2 │      │ SH-1 │ │ SH-2 │      │ GZ-1 │
      │      │ │      │      │      │ │      │      │      │
      │同城双活│ │同城双活│    │同城双活│ │同城双活│     │同城双活│
      └──┬───┘ └──┬───┘      └──┬───┘ └──┬───┘      └──┬───┘
         │        │             │        │             │
         └───┬────┘             └───┬────┘             │
             │                      │                  │
             ├───── 异步复制 ────────┼───── 异步复制 ─────┤
             │                       │                 │
        ┌────┴────┐             ┌────┴────┐        ┌────┴────┐
        │北京单元  │              │上海单元  │        │广州单元  │
        │UID%3==0 │             │UID%3==1 │        │UID%3==2 │
        └─────────┘             └─────────┘        └─────────┘
```

#### 核心机制1：数据分片与单元化（Set 化）

```
最关键的问题：多个数据中心同时写同一条数据 → 冲突！

解决方案：按用户维度分片，每条数据只在一个中心写（单点写）

┌─────────────────────────────────────────────────────┐
│                    分片规则                          │
│                                                     │
│  user_id % 3 == 0 → 归属北京单元（北京中心负责写）       │
│  user_id % 3 == 1 → 归属上海单元（上海中心负责写）       │
│  user_id % 3 == 2 → 归属广州单元（广州中心负责写）       │
│                                                     │
│  "归属"意味着：                                       │
│  • 该用户的所有写操作必须路由到归属中心                   │
│  • 该用户的数据以归属中心为权威                         │
│  • 其他中心异步复制该用户数据（只读副本）                 │
└─────────────────────────────────────────────────────┘
```

**微信的实际做法（按 Region 划分）：**
```
微信把用户分成若干"Set"：
  Set-深圳：广东/华南用户
  Set-上海：华东用户
  Set-天津：华北用户
  Set-成都：西南用户
  ...

每个 Set 是一个完整的服务闭环：
  自己的 MySQL、Redis、MQ、微服务 全套
  用户的消息、朋友圈、支付等数据都在 Set 内闭环处理
```

#### 核心机制2：路由层（最关键的组件）

```
用户请求进来后，必须路由到正确的单元：

┌──────────┐     ┌─────────────────┐     ┌──────────────┐
│ 用户请求  │ ──→  │  全局路由服务     │ ──→ │ 目标单元      │
│UID=12345 │     │ (Global Router) │     │ 上海单元      │
└──────────┘     └─────────────────┘     └──────────────┘

路由决策过程：
  ① 从请求中提取 UID（JWT/Cookie/参数）
  ② 查路由表：UID 12345 → 归属上海单元
  ③ 如果用户当前请求已在上海 IDC → 本地处理
  ④ 如果用户当前在北京 IDC → 转发到上海单元（跨单元调用）
```

**路由表实现：**
```
方案1：取模法
  unit = uid % N（简单但不灵活，扩缩容需要全量迁移）

方案2：虚拟分片（推荐）
  uid → hash → 虚拟桶(0~1023) → 物理单元映射表
  
  映射表存储在全局配置中心（ETCD/ZooKeeper）：
  {
    bucket_0~341:  "unit-beijing",
    bucket_342~683: "unit-shanghai", 
    bucket_684~1023: "unit-guangzhou"
  }
  
  扩容时只需修改映射表 + 迁移对应桶的数据
```

#### 核心机制3：跨单元访问

不是所有请求都能在单元内闭环，例如：

```
场景：北京用户 A (归属北京) 给上海用户 B (归属上海) 发消息

处理流程：
  ① A 的发送请求进入北京单元
  ② 北京单元发现收件人 B 归属上海
  ③ 北京单元通过"跨单元 MQ"将消息投递到上海单元
  ④ 上海单元将消息写入 B 的收件箱
  ⑤ 上海单元推送给 B

设计原则：
  • 写操作必须在归属单元执行（不能在北京写 B 的数据）
  • 跨单元调用走异步 MQ（解耦 + 容忍延迟）
  • 允许短暂不一致（A 发完消息后 B 可能延迟 100ms~1s 才收到）
```

#### 核心机制4：数据同步

```
┌──────────┐         ┌──────────┐         ┌──────────┐
│ 北京单元  │          │ 上海单元  │         │ 广州单元  │
│          │         │          │         │          │
│ 写入数据  │──DRC──→ │ 只读副本  │          │ 只读副本  │
│ (UID%3=0)│         │ (异步同步)│         │ (异步同步)│
│          │←──DRC── │ 写入数据  │──DRC──→ │ 只读副本  │
│ 只读副本  │         │ (UID%3=1)│         │ (异步同步)│
│          │         │          │←──DRC── │ 写入数据  │
│ 只读副本  │←──DRC── │ 只读副本  │          │ (UID%3=2)│
└──────────┘         └──────────┘         └──────────┘

DRC (Data Replication Center) 数据复制中心：
  • 监听 MySQL Binlog / Redis AOF
  • 跨机房传输（压缩 + 批量）
  • 目标端回放（幂等写入）
  • 冲突检测与解决
```

**数据同步组件（DRC）实现：**
```
                源端                               目标端
┌────────────────────────────┐    ┌────────────────────────────┐
│  MySQL Binlog              │    │                            │
│       ↓                    │    │                            │
│  Binlog Parser             │    │                            │
│  (解析为行变更事件)           │    │                            │
│       ↓                    │    │                            │
│  Filter                    │    │        Applier             │
│  (按库表/UID过滤)            │──→│  (幂等回放,INSERT改REPLACE)  │
│       ↓                    │    │        ↓                   │
│  Batch + Compress          │    │     目标 MySQL             │
│  (攒批压缩,降低带宽)          │    │                           │
│       ↓                    │    │                            │
│  Kafka/MQ 传输              │    │                           │
└────────────────────────────┘    └────────────────────────────┘

关键指标：
  • 延迟：正常 < 1s，峰值 < 5s
  • 带宽：每条变更 ~200B，百万 TPS × 200B = 200MB/s
  • 幂等：所有写入必须幂等（DRC 可能重传）
```

#### 核心机制5：容灾切换（和两地三中心最大的区别）

```
两地三中心切换：人工决策 + 手动操作 + 20~30 分钟
异地多活切换：自动化 + 秒级~分钟级

切换场景：上海单元整体不可用

  ① 全局监控检测到上海不可达（连续 10s 健康检查失败）
  ② 全局路由服务自动更新路由表：
     原：bucket_342~683 → unit-shanghai
     新：bucket_342~683 → unit-beijing (或 unit-guangzhou)
  ③ 后续请求自动路由到北京/广州
  ④ 北京/广州的只读副本提升为可写（因为已有异步同步的数据）
  ⑤ 用户体验：短暂 1~5s 错误后恢复

为什么能自动切？
  • 每个单元已有其他单元数据的只读副本（DRC 同步）
  • 切换只是改路由表 + 副本提升为主
  • 不存在"丢不丢数据的决策"——异步复制本来就接受最终一致
```

#### 核心机制6：流量调度层

```
┌─────────────────────────────────────────────────────────────┐
│                     流量调度全景                              │
│                                                             │
│  Layer 1: DNS 智能解析 (HTTPDNS / 运营商 LocalDNS)           │
│    → 按用户 IP 地理位置解析到最近的 IDC                        │
│    → 北京用户 → 北京 IDC IP                                  │
│    → 上海用户 → 上海 IDC IP                                  │
│                                                            │
│  Layer 2: 接入层 (Nginx/OpenResty/Envoy)                    │
│    → 从请求中提取 UID                                        │
│    → 查本地缓存的路由表                                       │
│    → UID 归属本单元？→ 本地处理                                │
│    → UID 归属其他单元？→ 302 重定向 或 反向代理转发              │
│                                                             │
│  Layer 3: RPC 框架层 (基于 gRPC/Dubbo 扩展)                   │
│    → 服务间调用自动识别目标数据归属                              │
│    → 单元内调用：直接走本地服务发现                              │
│    → 跨单元调用：走专线转发到目标单元                            │
└─────────────────────────────────────────────────────────────┘
```

**HTTPDNS（客户端调度优化）：**
```
传统 DNS 的问题：
  • 运营商 LocalDNS 可能代理导致解析不准
  • TTL 缓存导致切换慢（几分钟才生效）

HTTPDNS：
  客户端启动时直接 HTTP 请求调度服务：
  GET https://httpdns.example.com/resolve?uid=12345&ip=1.2.3.4
  
  Response: {
    "unit": "shanghai",
    "endpoints": ["ws://sh-1.example.com", "ws://sh-2.example.com"],
    "ttl": 300
  }
  
  优势：
  • 精确控制（按 UID + IP 双维度调度）
  • 秒级生效（客户端短 TTL 轮询 或 长连接推送）
  • 绕过运营商 DNS 劫持
```

### 六、实际案例：微信的 Set 化架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         微信架构简化版                            │
│                                                                 │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐          │
│  │ Set-深圳    │   │  Set-上海    │   │ Set-天津     │          │
│  │ (华南用户)   │   │ (华东用户)   │    │ (华北用户)   │          │
│  │            │    │             │   │             │           │
│  │ 完整微服务栈 │    │ 完整微服务栈 │    │ 完整微服务栈 │            │
│  │ 自己的MySQL │    │ 自己的MySQL │    │ 自己的MySQL │           │
│  │ 自己的Redis │    │ 自己的Redis │    │ 自己的Redis │           │
│  │ 自己的MQ    │    │ 自己的MQ    │    │ 自己的MQ    │           │
│  └──────┬──────┘    └──────┬──────┘  └──────┬──────┘         │
│         │                  │                │                 │
│         └─────── 跨Set MQ 总线 ──────────────┘                 │
│                  (跨单元消息投递)                                │
│                                                                │
│  全局服务（不分 Set）：                                          │
│  • 账号注册/登录（全局唯一性校验）                                 │
│  • 路由表管理                                                    │
│  • 全局搜索                                                      │
│  • 数据复制中心 (DRC)                                             │
└─────────────────────────────────────────────────────────────────┘

微信消息发送流程：
  ① 用户 A（深圳Set）发消息给用户 B（上海Set）
  ② 深圳Set 接收发送请求，写入 A 的"发件箱"（本Set闭环）
  ③ 深圳Set 通过跨Set MQ 投递到上海Set
  ④ 上海Set 消费消息，写入 B 的"收件箱"
  ⑤ 上海Set 通过长连接推送给 B
  ⑥ B 收到消息，ACK 回写已读状态（上海Set闭环）
```

### 七、实际案例：抖音/字节的异地多活

```
字节的方案更激进：不按 UID 固定归属，而是"就近写入 + 异步同步 + 冲突解决"

设计哲学：
  微信：强一致性优先（消息不能丢、不能乱序）→ 单点写 + 转发
  字节：可用性优先（刷视频允许短暂不一致）→ 就近写 + CRDT/Last-Write-Win

┌─────────────────────────────────────────────────────┐
│  字节 ByteMesh 多活架构核心                           │
│                                                     │
│  1. 数据分类：                                       │
│     强一致数据（支付/账户）→ 仍然单点写                  │
│     最终一致数据（点赞/评论/播放）→ 就近写多活            │
│                                                     │
│  2. 冲突解决：                                       │
│     • 计数类：CRDT Counter（多端 increment 可合并）    │
│     • 状态类：Last-Write-Win（以时间戳大的为准）        │
│     • 列表类：合并去重                                │
│                                                     │
│  3. 好处：                                           │
│     • 所有用户就近读写，延迟最低                        │
│     • 不需要跨机房转发（写路径无长距离 RTT）             │
│     • 单个数据中心故障，其他中心完全不受影响              │
└─────────────────────────────────────────────────────┘
```

### 八、关键挑战与解决方案汇总

| 挑战 | 解决方案 | 实施细节 |
|------|---------|---------|
| 数据一致性 | 单元化分片（单点写）或 CRDT（多点写） | 按 UID hash 分片；计数用 CRDT |
| 跨单元延迟 | 异步 MQ + 允许短暂不一致 | 跨单元调用走 MQ 解耦，P99 < 1s |
| 路由准确性 | HTTPDNS + 接入层二次路由 | 客户端+服务端双重保证 |
| 数据迁移（扩容） | 虚拟桶 + 双写过渡 | 新桶双写→追平→切流→停老写 |
| 全局唯一性（如用户名） | 全局服务（不分单元） | 注册走全局中心，强一致 |
| 切换数据丢失 | 异步复制 lag 兜底 + 业务补偿 | DRC lag < 1s；切换后对账补发 |

### 九、一句话总结各模式

```
同城双活   = 同一城市两个机房互为备份，抗机房故障，用户无感
两地三中心 = 同城双活 + 异地冷备，抗城市灾难，但切换慢且浪费资源
异地多活   = 每个数据中心都活着干活，按用户分片各管各的，秒级容灾
```

> 国民级应用的终极形态是**异地多活**：不是"一个干活其他看着"，而是"大家都在干活，谁挂了其他人接管它那份"。代价是需要一套完善的路由、分片、同步、冲突解决机制——这也是为什么只有微信/支付宝/抖音这种体量才做得起。

## 异地多活路由的"脏活"：如何提取路由键？

### 一、先承认现实：完全零侵入是幻想

```
理想状态：
  架构层完全透明，业务无感知，请求自动路由到正确单元

现实：
  路由的前提是"知道这个请求属于哪个用户/哪个单元"
  但请求格式千差万别，架构层不可能猜出来
  
  所以一定有某种程度的"约定"或"轻侵入"
  问题不是"要不要侵入"，而是"侵入到什么程度"
```

### 三、方案1：统一染色（Header 透传）— 最主流

#### 核心思想

**不在每个请求体里解析 ID，而是在入口处一次性确定路由标，全链路透传。**

```
┌─────────────────────────────────────────────────────────────────┐
│                        请求生命周期                               │
│                                                                 │
│  ① 客户端登录时，服务端返回路由信息（种在 Cookie/Token 里）         │
│     Login Response: {                                            │
│       token: "xxx",                                             │
│       route_key: "unit-shanghai",  ← 告诉客户端你归属哪个单元      │
│       // 或编码在 JWT claims 里                                   │
│     }                                                           │
│                                                                 │
│  ② 客户端后续所有请求自动携带路由标                                │
│     HTTP Header: X-Route-Key: unit-shanghai                      │
│     或 Cookie: _route=sh                                        │
│     或 JWT payload: {"uid": 12345, "unit": "sh"}                │
│                                                                 │
│  ③ 接入层网关只看 Header/Cookie，不解析 Body                      │
│     Nginx/Envoy 匹配 Header → 转发到对应单元                     │
│                                                                 │
│  ④ 服务间调用通过 RPC Context 透传，业务代码无需感知               │
│     gRPC metadata / Dubbo attachment 自动携带 route_key           │
└─────────────────────────────────────────────────────────────────┘
```

#### 具体实现

**入口网关（Go 自研云原生网关）：**
```go
// 大厂通常自研 Go 网关（类似字节 CloudWeGo/Hertz、腾讯 tRPC-Gateway）
// 核心：从 Header/Cookie/JWT 中提取路由标，反向代理到目标单元

package gateway

import (
    "net/http"
    "net/http/httputil"
    "net/url"
    "sync"

    "github.com/golang-jwt/jwt/v5"
)

// 单元路由表（从 ETCD 动态加载，热更新）
type UnitRouter struct {
    mu        sync.RWMutex
    unitAddrs map[string]*url.URL // unit_name → backend 地址
    // 示例: {"bj": "http://10.0.1.0:8080", "sh": "http://10.0.2.0:8080", "gz": "http://10.0.3.0:8080"}
}

// 路由中间件：从请求中提取路由标，决定转发到哪个单元
func (r *UnitRouter) RouteMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, req *http.Request) {
        unit := r.extractRouteKey(req)
        
        if unit == "" {
            // 兜底策略：无法识别路由标的请求，本地处理
            next.ServeHTTP(w, req)
            return
        }

        r.mu.RLock()
        target, ok := r.unitAddrs[unit]
        r.mu.RUnlock()

        if !ok {
            // 未知单元，本地降级处理
            next.ServeHTTP(w, req)
            return
        }

        // 反向代理到目标单元
        proxy := httputil.NewSingleHostReverseProxy(target)
        proxy.ServeHTTP(w, req)
    })
}

// 三级 fallback 提取路由标
func (r *UnitRouter) extractRouteKey(req *http.Request) string {
    // 优先级1：从 Header 直接取（客户端 SDK 主动设置）
    if unit := req.Header.Get("X-Route-Key"); unit != "" {
        return unit
    }

    // 优先级2：从 Cookie 取（登录时服务端种下）
    if cookie, err := req.Cookie("_route"); err == nil && cookie.Value != "" {
        return cookie.Value
    }

    // 优先级3：从 JWT Token 解析
    if unit := r.extractFromJWT(req); unit != "" {
        return unit
    }

    return "" // 无法确定路由，走降级
}

// 从 JWT claims 中解析 unit 字段
func (r *UnitRouter) extractFromJWT(req *http.Request) string {
    tokenStr := req.Header.Get("Authorization")
    if len(tokenStr) <= 7 { // "Bearer " 前缀
        return ""
    }
    tokenStr = tokenStr[7:]

    // 注意：网关层只需要解析 claims 中的 unit 字段
    // 不需要验签（验签由下游业务服务负责，网关追求极致性能）
    // 如果安全要求高，可以用公钥验签
    token, _, err := jwt.NewParser().ParseUnverified(tokenStr, jwt.MapClaims{})
    if err != nil {
        return ""
    }

    claims, ok := token.Claims.(jwt.MapClaims)
    if !ok {
        return ""
    }

    unit, _ := claims["unit"].(string)
    return unit
}

// --- 启动网关 ---
func main() {
    router := &UnitRouter{
        unitAddrs: map[string]*url.URL{
            "bj": mustParseURL("http://unit-beijing.internal:8080"),
            "sh": mustParseURL("http://unit-shanghai.internal:8080"),
            "gz": mustParseURL("http://unit-guangzhou.internal:8080"),
        },
    }

    // 启动 ETCD watcher，动态更新路由表（支持运行时切换单元）
    go router.watchETCD("etcd://config-center:2379/route_table")

    mux := http.NewServeMux()
    mux.Handle("/", router.RouteMiddleware(localHandler()))

    // 高性能: 通常配合 epoll + goroutine pool + 连接池
    http.ListenAndServe(":80", mux)
}
```

**RPC 框架透传（以 gRPC 为例）：**
```go
// 框架拦截器自动注入，业务代码完全不感知

// --- 客户端拦截器 ---
func routeInterceptor(ctx context.Context, ...) {
    // 从当前请求上下文取出路由标
    routeKey := ctx.Value("X-Route-Key")
    // 自动注入到 gRPC metadata
    md := metadata.Pairs("x-route-key", routeKey)
    ctx = metadata.NewOutgoingContext(ctx, md)
    // 继续调用
    return invoker(ctx, ...)
}

// --- 服务端拦截器 ---
func routeServerInterceptor(ctx context.Context, ...) {
    // 从 metadata 取出路由标，放入 context
    md, _ := metadata.FromIncomingContext(ctx)
    routeKey := md.Get("x-route-key")[0]
    ctx = context.WithValue(ctx, "X-Route-Key", routeKey)
    // 继续处理
    return handler(ctx, req)
}

// --- 业务代码完全不知道路由的存在 ---
func (s *Service) GetUserProfile(ctx context.Context, req *pb.Request) {

    return user
```

```
时机：用户登录/注册时确定

① 用户注册：

   unit = routeTable.Lookup(uid)  // 查全局路由表
   // 把 unit 信息编码到 JWT/Session
   
② 用户登录：
```

### 四、方案2：SDK/框架内置路由（微信/字节的做法）

#### 核心思想

不依赖 Header，而是**在 RPC 框架层内置路由能力**，框架知道每个接口的"路由键字段"是什么。

```
┌─────────────────────────────────────────────────────────────────┐
│  不同服务的路由键不同，但框架统一管理                               │
│                                                                 │
│  消息服务：路由键 = receiver_uid（收件人决定在哪个单元写）           │
│  朋友圈服务：路由键 = author_uid（作者决定在哪个单元写）            │
│  支付服务：路由键 = payer_uid                                     │
│  群聊服务：路由键 = group_id（群维度分片，不按用户）               │
│                                                                 │
│  这些"路由键是请求中的哪个字段"需要业务声明，但只声明一次            │
└─────────────────────────────────────────────────────────────────┘
```

#### 具体实现：IDL 注解 + 代码生成

```protobuf
// Proto 定义中，通过注解声明路由键
// 这是"轻侵入"：只需要在 IDL 里加一个注解

service MessageService {
  rpc SendMessage(SendReq) returns (SendResp) {
    option (route_key) = "receiver_uid";  // ← 声明路由键字段
  }
  
  rpc GetInbox(InboxReq) returns (InboxResp) {
    option (route_key) = "owner_uid";
  }
}

message SendReq {
  int64 sender_uid = 1;
  int64 receiver_uid = 2;   // ← 框架知道用这个字段做路由
  string content = 3;
}

message InboxReq {
  int64 owner_uid = 1;      // ← 框架知道用这个字段做路由
  int64 cursor = 2;
}
```

#### 微信 WXG 的实际做法（Svrkit 框架）

```
微信内部 RPC 框架 Svrkit 的路由机制：

① 每个服务在"服务配置"中声明自己的路由键类型：
   service_config.yaml:
     route_type: "uid"           # 按 UID 路由
     route_field: "to_uid"       # 请求结构体中的字段名
     
② Svrkit 框架在序列化请求时，自动提取 route_field 的值
③ 查本地缓存的路由表（ETCD 同步），得到目标 Set
④ 如果目标 Set == 本 Set → 本地调用
   如果目标 Set != 本 Set → 走跨 Set 专线转发
⑤ 业务代码完全不变，换 Set 只需改路由表配置
```

### 五、没有 ID 的请求怎么办？

这是最棘手的场景，实际有好几种情况：

```
┌──────────────────────────────────────────────────────────────┐
│  场景                        解决方案                          │
├──────────────────────────────────────────────────────────────┤
│ 未登录用户浏览（无UID）       → 本地处理，读就近缓存副本         │
│ 注册请求（还没有UID）         → 全局服务处理，分配后写入路由表   │
│ 搜索/推荐（跨用户查询）       → 全局服务，不走单元化            │
│ 批量接口（涉及多个UID）       → 拆分+扇出，各回各单元           │
│ Webhook/回调（第三方调入）     → 网关解析回调参数中的ID          │
└──────────────────────────────────────────────────────────────┘
```

#### 场景1：未登录用户

```
未登录 = 没有 UID = 不知道归属哪个单元

处理策略：就近处理 + 只读

  ① 浏览内容（视频/文章/商品）→ 任何单元都有只读副本，本地直接读
  ② 发起写操作 → 强制要求登录（登录后有了 UID，有了路由标）
  
  实际上：
    抖音刷视频不登录也行 → 推荐服务是全局服务，不分单元
    微信所有操作都要登录 → 不存在无 UID 的写请求
```

#### 场景2：注册请求

```
注册时用户还没有 UID，怎么决定归属？

┌─────────────────────────────────────────────┐
│  注册流程（全局服务处理）                       │
│                                             │
│  ① 用户发起注册（手机号/邮箱）                 │
│  ② 请求到达"全局注册服务"（不分单元的独立服务）  │
│  ③ 全局服务检查唯一性（用户名/手机号全局唯一）   │
│  ④ 分配 UID（雪花算法）                       │
│  ⑤ 决定归属单元：                             │
│     策略A：按手机号归属地（广东号→广州单元）      │
│     策略B：按当前接入 IDC 就近分配               │
│     策略C：按各单元当前负载均衡分配              │
│  ⑥ 写入全局路由表：UID 12345 → unit-shanghai  │
│  ⑦ 在目标单元创建用户数据                      │
│  ⑧ 返回 Token（含 unit 信息）                 │
└─────────────────────────────────────────────┘

"全局服务"是异地多活中少数不走单元化的组件：
  • 全局唯一性校验（用户名/手机号不能重复）
  • 全局路由表管理
  • 全局搜索索引
  
  这些服务仍然用"两地三中心"模式（单点写），不做多活
  因为注册是低频操作（QPS 远低于消息/视频等），单点写能扛住
```

#### 场景3：涉及多个用户的请求

```
场景：拉取群聊消息列表（群里有来自不同单元的用户）

方案A：群维度分片（微信的做法）
  群聊数据按 group_id 分片，不按 user_id
  群消息统一存在群归属的单元
  → 拉群消息只需路由到群所在单元，一次搞定
  → 但群成员的"未读数"等用户维度数据仍在各自单元

方案B：请求拆分 + 扇出 + 聚合
  GET /friends?uid=123 (获取好友列表，好友散布在各单元)
  
  ① 网关识别这是"多单元查询"
  ② 先查本单元 UID=123 的好友 ID 列表
  ③ 按好友 ID 分组：哪些在北京/上海/广州
  ④ 并行请求各单元获取好友资料
  ⑤ 聚合后返回客户端
  
  注意：这种扇出查询性能差，实际通过"数据冗余"规避：
    好友资料异步同步到所有单元（只读副本）
    拉好友列表 → 本单元直接读只读副本 → 无需跨单元
```

#### 场景4：OpenID / 非 UID 的路由键

```
问题：不同业务用不同 ID 体系
  • 用户服务用 uid
  • 微信开放平台用 openid
  • 支付用 merchant_id + order_id
  • 设备推送用 device_id

解决方案：统一映射层

┌─────────────────────────────────────────────────┐
│            ID 映射服务（全局）                      │
│                                                 │
│  openid  → uid → unit                           │
│  phone   → uid → unit                           │
│  device_id → uid → unit                         │
│  order_id → uid (下单用户) → unit               │
│                                                 │
│  本质上所有 ID 最终都能映射回一个 uid              │
│  路由键永远是 uid，只是"从请求中提取 uid"的方式不同 │
└─────────────────────────────────────────────────┘

实现：
  // 网关层配置不同接口的 ID 提取规则
  route_rules:
    /api/user/*:
      extract: header.X-UID           # 直接从 Header 取
    /api/wx/callback:
      extract: body.openid            # 从 Body JSON 取 openid
      mapping: openid_to_uid          # 再查映射表得到 uid
    /api/pay/notify:
      extract: body.order_id
```

### 七、蚂蚁金服 CRG（Cell Route Gateway）实战案例

蚂蚁是业界异地多活做得最成熟的，它的方案最能回答你的问题：

```
蚂蚁的三层路由体系：

Layer 1: 接入层（Spanner 网关）
  ├─ 从 Cookie 中取 _zone_uid_ 字段
  ├─ 没有 Cookie？从 URL 参数取 uid
  ├─ 都没有？看请求路径匹配规则（如 /api/register → 全局服务）
  └─ 以上都失败？→ 本地处理（降级策略）

Layer 2: 服务层（SOFARPC 框架）
  ├─ 框架拦截器自动从请求对象中按配置提取路由键
  ├─ 配置中心下发每个接口的路由键字段名
  ├─ 支持多级 fallback：优先取 uid，没有取 order_id 再映射
  └─ 跨单元调用自动走专线，对业务透明

Layer 3: 数据层（OceanBase / 自研中间件）
  ├─ DB Proxy 层根据 SQL 中的分片键自动路由
  ├─ 分片键 = uid（所有表必须包含 uid 列 — 这是唯一的硬性约束）
  └─ 跨单元 JOIN 直接禁止（规范约束）

蚂蚁对业务的"硬性要求"（最小侵入）：
  ① 所有核心表必须有 uid 列（分片键）— 建表规范
  ② 所有写接口必须能提取到 uid — 接口规范
  ③ 禁止跨分片键 JOIN — SQL 规范
  ④ 其余业务逻辑不需要任何改动
```

### 八、总结：路由问题的本质

```
Q: 架构层怎么知道请求该路由到哪个单元？
A: 分三步走——

  ① 登录时确定归属，编码到 Token/Cookie（一次性成本）
  ② 后续请求通过 Header/Cookie 透传（零侵入）
  ③ 框架层自动传播（RPC Context 透传，业务无感知）

Q: 不同业务用不同 ID 怎么办？
A: 所有 ID 最终映射回 uid，路由键永远是 uid

Q: 没有 ID 的请求怎么办？
A: 分类处理——
   • 读请求 → 本地读副本（不需要路由）
   • 注册 → 走全局服务
   • 搜索/推荐 → 全局服务
   • 写请求 → 强制要求携带 uid（接口规范）

Q: 对业务的真实侵入是什么？
A: 三条规范：
   ① 所有表有 uid 列
   ② 所有写接口可提取 uid  
   ③ 禁止跨用户 JOIN
   
   代码逻辑本身不需要改，只是数据模型和接口设计要遵守规范
```

> 一句话：**异地多活对业务的侵入不在"代码逻辑"，而在"数据模型设计"和"接口规范"。** 只要你的表有 uid、接口能提取 uid、不跨用户 JOIN，剩下的路由/同步/切换全部由中间件搞定。这也是为什么异地多活必须从项目初期就规划——后期改造数据模型的成本远大于代码改造。
