## BGP 多线是什么？

**BGP（Border Gateway Protocol）多线**是中国 IDC 机房的一种网络接入方式，用来解决**跨运营商访问慢**的问题。

### 背景：中国运营商互联互通问题

中国有多个运营商（电信、联通、移动、教育网等），它们之间互联带宽有限、延迟高：

```
电信用户 → 访问联通服务器 → 跨网交换节点 → 延迟 30~100ms，丢包高
电信用户 → 访问电信服务器 → 同网直连 → 延迟 5~10ms，质量好
```

### 传统方案 vs BGP 多线

| 方案 | 做法 | 缺点 |
|------|------|------|
| **单线机房** | 只接一家运营商（如电信） | 其他运营商用户体验差 |
| **双线/多线（DNS 分流）** | 每个运营商一个 IP，DNS 按来源解析 | 需要多个 IP，DNS 切换慢，维护复杂 |
| **BGP 多线** | 机房同时接入多家运营商，通过 BGP 协议**用同一个 IP** 自动选择最优路径 | ✓ 最优方案 |

### BGP 多线的工作原理

```
                    ┌─ 电信骨干网 ─┐
                    │              │
电信用户 ──────────→┤              ├──→ ┌──────────────┐
                    │              │    │              │
联通用户 ──────────→┤ BGP 路由选路 ├──→ │  IDC 机房    │
                    │              │    │  (同一个IP)  │
移动用户 ──────────→┤              ├──→ │              │
                    │              │    └──────────────┘
                    └─ 移动骨干网 ─┘
```

BGP 路由器会向所有运营商**广播同一段 IP**，各运营商路由器根据 BGP 路由表自动选择到这个 IP 的最短路径（通常就是直连该运营商的链路）。

### 为什么弹幕系统需要它？

结合文档的场景：

1. **DAU 4 亿**，覆盖所有运营商用户，不能只服务电信用户
2. **直播推送 P99 ≤ 1s**，跨运营商绕路的额外延迟不可接受
3. **WebSocket 长连接**对网络质量敏感，丢包/高延迟会导致频繁断连重连
4. **出口带宽 200 Gbps**，需要多运营商链路叠加容量

文档写的是：

> **BGP 多线 + 边缘节点 L4 负载均衡，单 IDC 50 Gbps × 4 IDC = 200 Gbps**

所以实际部署是 **4 个 IDC（多地域）× 每个 IDC BGP 多线接入**，保证：
- 任何运营商的用户都能以最短路径到达最近的 IDC
- 单 IDC 故障时 BGP 撤路由，流量秒级切换到其他 IDC

### 一句话总结

> BGP 多线 = **用一个 IP 同时接入所有运营商，让每个用户自动走最快的路**，是大流量实时系统在中国部署的标配。

## BGP 具体工作原理

### 一、BGP 是什么

BGP 是**自治系统（AS）之间**的路由协议，是互联网的"导航系统"。整个互联网由几万个 AS 组成，每个 AS 是一个独立管理的网络（如电信 AS4134、联通 AS4837、阿里云 AS37963）。

```
互联网 = 一堆 AS 互相连接
BGP = 告诉每个 AS "去某个 IP 段该走哪条路"
```

### 二、核心概念

| 概念 | 说明 |
|------|------|
| **AS（自治系统）** | 一个独立运营的网络，有全球唯一编号（ASN），如电信 AS4134 |
| **BGP Peer（邻居）** | 两个 AS 之间建立 TCP 179 端口的 BGP 会话 |
| **Prefix（前缀）** | 一段 IP 地址块，如 `203.0.113.0/24` |
| **AS-PATH** | 到达某个 Prefix 需要经过哪些 AS，如 `[AS4837, AS4134, AS37963]` |
| **NEXT-HOP** | 下一跳的路由器 IP |
| **LOCAL-PREF** | 本地优先级，数值越大越优先（用于出站选路） |
| **MED** | 多出口鉴别符，告诉邻居"从我这个口进来更好"（用于入站选路） |

### 三、BGP 建立连接过程

```
Router A (AS 100)                    Router B (AS 200)
     │                                    │
     │──── TCP 3-way handshake ──────────→│  (TCP 179端口)
     │                                    │
     │──── OPEN (ASN, Router-ID, Hold) ──→│  ① 交换身份
     │←─── OPEN ─────────────────────────│
     │                                    │
     │──── UPDATE (路由通告) ─────────────→│  ② 交换路由表
     │←─── UPDATE ───────────────────────│
     │                                    │
     │──── KEEPALIVE (每60s) ────────────→│  ③ 保活
     │←─── KEEPALIVE ───────────────────│
     │                                    │
     │         [Established 状态]          │
```

### 四、路由通告（核心机制）

BGP 的核心工作就是**通告和传播路由信息**：

```
假设：阿里云（AS37963）拥有 IP 段 47.96.0.0/16

Step 1: 阿里云向邻居（电信、联通）发 UPDATE 消息
        "47.96.0.0/16 可以通过我到达，AS-PATH = [AS37963]"

Step 2: 电信收到后，转发给它的邻居
        "47.96.0.0/16，AS-PATH = [AS4134, AS37963]"
        
Step 3: 联通收到后，转发给它的邻居
        "47.96.0.0/16，AS-PATH = [AS4837, AS37963]"

Step 4: 某个 ISP 同时从电信和联通收到这条路由，比较后选最优
```

**UPDATE 消息结构：**
```
UPDATE {
  NLRI: 47.96.0.0/16          // 可达的网络前缀
  AS-PATH: [AS4134, AS37963]  // 经过的AS列表
  NEXT-HOP: 202.97.33.1       // 下一跳地址
  MED: 100                     // 多出口度量
  Community: [13335:10000]     // 社区属性(策略标签)
}
```

### 五、选路算法（Best Path Selection）

当一个路由器收到多条去同一个目标的路由时，按以下顺序选最优：

```
优先级从高到低：

① LOCAL-PREF 最大                 ← 本地策略（"我想走哪条"）
② AS-PATH 最短                    ← 经过的AS越少越好
③ Origin 类型 (IGP > EGP > ?)    ← 路由来源
④ MED 最小                        ← 对端建议（"从这个口进来最近"）
⑤ eBGP > iBGP                    ← 外部学到的优于内部
⑥ IGP metric 到 NEXT-HOP 最小    ← 内部到下一跳的物理距离
⑦ Router-ID 最小                  ← 兜底 tiebreaker
```

**实际例子：**
```
目标：47.96.0.0/16（阿里云）

路由A: AS-PATH = [AS4134, AS37963]       经过2个AS
路由B: AS-PATH = [AS4837, AS9929, AS37963] 经过3个AS

→ 选路由A（AS-PATH更短）
```

### 六、BGP 多线 IDC 如何工作

回到弹幕系统的场景：

```
                    AS4134 (电信)
                   /     ↑
                  /      │ BGP Peer
                 /       │
电信用户 ────→ 电信路由器  │
                         │
              ┌──────────┴──────────────┐
              │   IDC 的 BGP 路由器       │
              │   AS 65000 (私有ASN)      │
              │                          │
              │   向所有 Peer 广播：       │
              │   "203.0.113.0/24        │
              │    通过我可达"            │
              │                          │
              └──────────┬──────────────┘
                         │ BGP Peer
联通用户 ────→ 联通路由器  │
                 \       │
                  \      ↓
                   AS4837 (联通)
```

**关键操作：**

```
1. IDC 路由器与电信、联通、移动分别建立 BGP Peer

2. IDC 向所有 Peer 广播同一段 IP：
   "203.0.113.0/24, AS-PATH=[AS65000], NEXT-HOP=我"

3. 电信路由器收到后写入路由表：
   "去 203.0.113.0/24 → 走我和 IDC 的直连链路"

4. 联通路由器收到后也写入路由表：
   "去 203.0.113.0/24 → 走我和 IDC 的直连链路"

5. 结果：
   电信用户访问 203.0.113.1 → 电信路由器查表 → 直连到IDC（不绕联通）
   联通用户访问 203.0.113.1 → 联通路由器查表 → 直连到IDC（不绕电信）
```

### 七、故障切换（BGP 撤路由）

```
正常状态：
  IDC-北京 广播 203.0.113.0/24  ← 北京用户走这里
  IDC-上海 广播 203.0.113.0/24  ← 上海用户走这里（MED/AS-PATH调优）

IDC-北京故障：
  ① BGP KEEPALIVE 超时（默认180s，可调到几秒）
  ② 或主动发 WITHDRAW 消息撤回路由
  ③ 所有运营商路由器删除"去北京"的路由条目
  ④ 重新选路 → 流量自动切到 IDC-上海
  ⑤ 收敛时间：秒级（配合 BFD 检测可做到亚秒）
```

### 八、BGP 与弹幕系统的关系总结

```
┌─────────────────────────────────────────────────────┐
│                  弹幕系统部署视角                      │
├─────────────────────────────────────────────────────┤
│                                                     │
│  4 个 IDC，每个 IDC：                                │
│  ┌─────────────────────────────────────┐            │
│  │ BGP 路由器                           │            │
│  │  ├─ Peer → 电信（AS4134）           │            │
│  │  ├─ Peer → 联通（AS4837）           │            │
│  │  ├─ Peer → 移动（AS9808）           │            │
│  │  └─ 广播同一个 Anycast IP 段         │            │
│  ├─────────────────────────────────────┤            │
│  │ LVS (L4 负载均衡)                    │            │
│  ├─────────────────────────────────────┤            │
│  │ WS 长链接服务 × N 台                  │            │
│  └─────────────────────────────────────┘            │
│                                                     │
│  效果：                                             │
│  - 用户无感知，访问同一个 IP                          │
│  - 路由器自动选最近 IDC + 同运营商链路                │
│  - 单 IDC 故障，BGP 撤路由，秒级切换                 │
│  - 多运营商带宽叠加，单 IDC 50Gbps 出口              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 一句话

> BGP 的本质就是：**每个网络不断告诉邻居"这些 IP 通过我可以到达"，所有路由器汇总后选最短路径转发数据包**。BGP 多线就是让 IDC 同时告诉所有运营商"我这里有你要的 IP"，各运营商用户自然走各自的直连链路。

## AS（Autonomous System，自治系统）

### 一句话定义

> AS = 互联网上**由同一个组织统一管理、统一路由策略**的一个网络集合，对外表现为一个整体。

### 类比理解

把互联网想象成全球物流网络：

```
AS ≈ 一个快递公司的内部网络

顺丰内部：有自己的分拣中心、干线、末端网点，内部怎么运转自己说了算
中通内部：同理，自成体系

AS 之间的 BGP ≈ 快递公司之间的转运协议
  "你那边有个包裹要到我管辖的区域？从这个口给我，我内部搞定"
```

### 具体来说，一个 AS 包含什么

```
┌─────────────────────────────────────────────┐
│           AS4134（中国电信骨干网）              │
│                                             │
│  • 几千台核心/汇聚/接入路由器                  │
│  • 几百万公里光纤                             │
│  • 几亿个 IP 地址                            │
│  • 统一的内部路由协议（OSPF/IS-IS）            │
│  • 统一的对外路由策略（BGP）                   │
│                                             │
│  对外只暴露：                                 │
│    "我是 AS4134，我能到达这些 IP 段"           │
└─────────────────────────────────────────────┘
```

### 谁有 AS？

| 类型 | 举例 | ASN |
|------|------|-----|
| 电信运营商 | 中国电信 | AS4134 |
| 电信运营商 | 中国联通 | AS4837 |
| 电信运营商 | 中国移动 | AS9808 |
| 云厂商 | 阿里云 | AS37963 |
| 云厂商 | 腾讯云 | AS45090 |
| CDN | Cloudflare | AS13335 |
| 大企业 | Apple | AS714 |
| 大学/教育网 | CERNET | AS4538 |
| 小型 IDC | 某机房 | AS65000（私有） |

> 基本规律：**只要你的网络大到需要和多个运营商/网络直接互联，你就需要一个 AS 号**。

### ASN（AS Number）

每个 AS 有一个全球唯一编号，由 IANA 及其下属机构（APNIC、ARIN 等）分配：

```
公有 ASN：1 ~ 64511，64512 ~ 4199999999（32位扩展）
  → 全球唯一，在互联网上可见

私有 ASN：64512 ~ 65534（16位），4200000000 ~ 4294967294（32位）
  → 仅内部使用，不对外广播（类似私有IP 192.168.x.x）
```

### AS 内部 vs AS 之间

```
┌──────────────┐          BGP           ┌──────────────┐
│   AS4134     │◄──────────────────────►│   AS4837     │
│  (电信)      │    "我能到这些IP"        │  (联通)      │
│              │    AS之间用 eBGP         │              │
│  内部用      │                         │  内部用      │
│  OSPF/IS-IS  │                         │  OSPF/IS-IS  │
│  (几千台路由  │                         │  (自己管自己) │
│   器自己玩)  │                         │              │
└──────────────┘                         └──────────────┘

AS 内部：OSPF/IS-IS（快速收敛，信任所有节点，追求最短路径）
AS 之间：BGP（策略路由，不信任对方，追求可控+可商业化）
```

### 为什么需要 AS 这个概念？

互联网有 **90万+** 台路由器，如果没有层级划分：

```
无 AS 的世界：
  每台路由器要知道去所有 IP 的完整路径 → 路由表爆炸
  任何一个网络变动 → 全球所有路由器重新计算 → 永远收敛不了

有 AS 的世界：
  AS 内部：自己管自己的几千台路由器（OSPF，收敛快）
  AS 之间：只交换"IP 段级别"的汇总信息（BGP，约100万条路由）
  → 分层管理，各管各的，只在边界交换汇总信息
```

这和计算机网络分层、DNS 分级是同一个思想：**大系统必须分治**。

### 一句话总结

> AS = 互联网的"国家"，有自己的领土（IP 段）、内政（OSPF）和外交（BGP）。BGP 就是 AS 之间的外交协议。

## OSPF（Open Shortest Path First，开放最短路径优先）

### 一句话定义

> OSPF = **一个 AS 内部**使用的路由协议，让 AS 内所有路由器自动学习网络拓扑，算出到任意目的地的最短路径。

### 和 BGP 的关系

```
BGP  = "国家之间"的外交协议（AS 之间，跨组织）
OSPF = "国家内部"的交通规划（AS 内部，同一组织）

电信内部几千台路由器之间怎么互相转发？→ OSPF
电信和联通之间怎么交换路由？→ BGP
```

### 为什么 AS 内部不用 BGP？

| 维度 | BGP（AS 间） | OSPF（AS 内） |
|------|-------------|--------------|
| 设计目标 | 策略可控、商业利益 | 快速收敛、最优路径 |
| 信任模型 | 不信任对方（可能骗你） | 完全信任（都是自家设备） |
| 选路依据 | 策略（LOCAL-PREF、AS-PATH...） | 链路开销（带宽越大开销越小） |
| 收敛速度 | 慢（分钟级，需要策略协商） | 快（亚秒~秒级） |
| 规模 | 全球 100 万条路由 | 单 AS 内几千~几万条 |

### OSPF 的核心思想

**每台路由器都拥有完整的网络地图，自己用 Dijkstra 算法算最短路。**

```
传统"距离向量"路由（如 RIP）：
  路由器只知道"邻居告诉我去 X 要 3 跳" → 信息不完整，容易环路

OSPF"链路状态"路由：
  每台路由器广播"我和谁相连，链路带宽多少"
  → 所有路由器拼出完整拓扑图
  → 各自跑 Dijkstra 算最短路径树
  → 无环路，收敛快
```

### OSPF 工作流程

```
Step 1: 邻居发现
  路由器启动后，向组播地址 224.0.0.5 发 Hello 包（每10s）
  "我是 Router-A，我在 10.0.1.0/24 这个网段"
  
Step 2: 邻接关系建立
  两台路由器互相收到 Hello → 确认邻居
  选举 DR/BDR（减少广播量） → 建立邻接（Full 状态）

Step 3: 链路状态通告（LSA）
  每台路由器生成自己的 LSA：
  "我是 Router-A，我连着：
    → Router-B，链路开销 10（1Gbps）
    → Router-C，链路开销 1（10Gbps）
    → 网段 192.168.1.0/24"

Step 4: LSA 洪泛（Flooding）
  所有 LSA 在整个区域内洪泛传播
  → 每台路由器都收集到完整的 LSA 数据库（LSDB）

Step 5: SPF 计算
  每台路由器用 Dijkstra 算法：
  以自己为根，算出到所有目的网段的最短路径树
  → 生成路由表
```

### 图解：Dijkstra 选路

```
网络拓扑（数字=链路开销，开销越小=带宽越大）：

        10          1
  A ─────────── B ─────────── D
  │                           │
  │ 1                         │ 10
  │                           │
  C ─────────────────────────E
            5

A 要去 D，两条路径：
  路径1: A → B → D，开销 = 10 + 1 = 11
  路径2: A → C → E → D，开销 = 1 + 5 + 10 = 16

→ 选路径1（开销更小）
```

### 链路开销怎么算？

```
开销 = 参考带宽 / 接口带宽

默认参考带宽 = 100 Mbps

  10 Gbps 链路 → 开销 = 100M / 10G = 0.01 → 取1（最小）
  1 Gbps 链路  → 开销 = 100M / 1G = 0.1 → 取1
  100 Mbps 链路 → 开销 = 100M / 100M = 1
  10 Mbps 链路  → 开销 = 100M / 10M = 10

→ 带宽越大，开销越小，越会被选为最短路径
```

### OSPF 区域划分（大规模网络）

单个 AS 内如果有几千台路由器，所有 LSA 全洪泛 → LSDB 太大、SPF 计算太慢。解决方案：**分区域**。

```
┌─────────────────────────────────────────────────┐
│                  AS4134（电信）                    │
│                                                 │
│         ┌──────── Area 0 (骨干区) ────────┐      │
│         │  核心路由器 CR1 ─── CR2 ─── CR3 │      │
│         └──┬──────────┬──────────┬───────┘      │
│            │          │          │              │
│      ┌─────┴────┐ ┌───┴────┐ ┌──┴─────┐        │
│      │ Area 1   │ │ Area 2 │ │ Area 3 │        │
│      │ 华北区域  │ │ 华东   │ │ 华南   │        │
│      │ 200台路由 │ │ 300台  │ │ 250台  │        │
│      └──────────┘ └────────┘ └────────┘        │
│                                                 │
│  规则：                                          │
│  • 所有区域必须连接到 Area 0（骨干区）              │
│  • 区域内 LSA 只在区域内洪泛                       │
│  • 区域间通过 ABR（边界路由器）汇总通告             │
│  • 大幅减少每台路由器的 LSDB 规模和 SPF 计算量     │
└─────────────────────────────────────────────────┘
```

### OSPF 收敛过程（故障恢复）

```
正常状态：A → B → D 是最短路径

B-D 链路断了：
  ① B 检测到链路 Down（毫秒级，物理层中断信号）
  ② B 生成新的 LSA："我和 D 之间的链路没了"
  ③ B 洪泛新 LSA 到整个区域（毫秒级）
  ④ 所有路由器收到后重新跑 Dijkstra
  ⑤ A 重新计算：去 D 改走 A → C → E → D
  ⑥ 全过程：亚秒 ~ 1秒（对比 BGP 的分钟级）
```

### OSPF vs IS-IS（另一种 AS 内协议）

文档中提到 "OSPF/IS-IS"，它们是同类竞品：

| 维度 | OSPF | IS-IS |
|------|------|-------|
| 标准 | IETF（IP 世界） | ISO（电信世界） |
| 运行层次 | IP 层之上（协议号 89） | 直接在二层之上 |
| 扩展性 | 适合中等规模（几百~千台） | 适合超大规模（万台+） |
| 使用者 | 企业网、中小 ISP | 大型运营商骨干网（电信/联通） |
| IPv6 支持 | 需要 OSPFv3（新协议） | 原生 TLV 扩展即可 |

> 实际上中国三大运营商骨干网大多用 IS-IS，但原理和 OSPF 一样（链路状态 + Dijkstra）。文档里写 "OSPF/IS-IS" 就是泛指"AS 内部的链路状态路由协议"。

### 回到弹幕系统

```
用户数据包的完整旅程：

用户手机
  → 基站/家庭宽带
  → 运营商接入层
  → 运营商内部（OSPF/IS-IS 选路，自动走最快链路）  ← AS 内
  → 运营商边界路由器
  → BGP 选路，决定走哪条线到 IDC                   ← AS 间
  → IDC 的 BGP 路由器
  → LVS (L4 负载均衡)
  → WS 长链接服务
```

### 一句话总结

> OSPF = AS 内部的"自动导航"，每台路由器共享完整地图，用 Dijkstra 算最短路，链路故障亚秒级自愈。BGP 管"去哪个网络"，OSPF 管"在自家网络里怎么走最快"。
