# 业务建模通用方法论

> **前置阅读**：`系统架构设计通用方法论` 解决"怎么扛住流量"的非功能问题。本文解决更难的问题：**面对陌生业务，如何在 30 分钟内建出模型、拆好服务、画出流程、识别风险** —— 这是架构师的真正门槛。

---

 **核心认知：业务不是"陌生"的，是"没抽象"的**

任何业务系统，剥开术语的表皮，本质都是通用模型：

```
【谁】对【什么】做了【什么动作】产生【什么结果】
 Actor    Object     Action        Event/State
```

所有业务都是 `Actor × Object × Action × State` 的排列组合。**陌生领域之所以陌生，是因为你停留在"术语层"，而不是在"抽象层"。**

| 业务 | 本质抽象 | 业务 | 本质抽象 |
|------|---------|------|---------|
| 红包 | 库存扣减（有限池子被多人抢） | 推送 | 消息投递（长连+厂商+必达） |
| 弹幕 | 发布订阅（一对多广播+时间对齐） | 视频 | 文件切片 + CDN 分发 |
| 评论 | 层级列表（树形+分页+计数） | Feed | 推拉模型（写扩散 vs 读扩散） |
| 短链 | 发号器（ID 生成+KV 查询） | 排行榜 | 有序集合（ZSet 业务包装） |
| 延时任务 | 调度器（时间触发+持久化防丢） |  |  |

**全世界只有 20~30 种业务原型，剩下的都是组合和变形。**

## 1. 业务原型库（架构师的"元素周期表"）

**方法论：拿到陌生需求先问"它最接近哪几个原型的组合？"**
- 电商购物车 ≈ KV 查询 + 状态机
- 直播打赏 ≈ 库存扣减 + 发布订阅
- 外卖下单 ≈ 工作流 + 状态机 + 消息投递
- 滴滴派单 ≈ 推荐匹配 + 状态机 + 消息投递

### 1.1 数据操作型

| 原型 | 核心特征 | 典型案例 | 关键难点 |
|------|---------|---------|---------|
| **发号器** | 生成唯一 ID | 短链、订单号、雪花 ID | 全局唯一、趋势递增、防枚举 |
| **库存扣减** | 有限资源被多人争抢 | 红包、秒杀、优惠券领取 | 超卖、幂等、热点 |
| **计数器** | 高频累加查询 | 点赞、播放量、粉丝数 | 热点、精度 vs 性能 |
| **KV 查询** | 按 key 查 value | 短链跳转、配置中心 | 缓存、穿透 |
| **层级列表** | 树形+分页+计数 | 评论、目录、组织架构 | 深度、展开、计数 |
| **有序集合** | 排序+范围查询 | 排行榜、Feed、评论按时间 | 分片、热点、精度 |

### 1.2 消息传递型

| 原型 | 核心特征 | 典型案例 | 关键难点 |
|------|---------|---------|---------|
| **发布订阅** | 一对多广播 | 弹幕、直播间消息、群聊 | 扇出、时效、顺序 |
| **消息投递** | 点对点必达 | 推送、IM、通知 | 在线/离线、去重、保序 |
| **推拉模型** | 写扩散 vs 读扩散 | Feed、朋友圈、订阅 | 大 V 热点、存储爆炸 |
| **调度触发** | 定时/延时执行 | 延时任务、Cron、到期提醒 | 精度、不丢、不重 |

### 1.3 流程编排型

| 原型 | 核心特征 | 典型案例 | 关键难点 |
|------|---------|---------|---------|
| **状态机** | 有限状态单向流转 | 订单、审批、工单 | 并发变更、回滚 |
| **工作流** | 多步骤有向编排 | 支付、开户、退款 | 长事务、补偿、幂等 |
| **事务协调** | 跨服务一致性 | 分布式事务、跨行转账 | 2PC/TCC/Saga 选型 |

### 1.4 内容处理型

| 原型 | 核心特征 | 典型案例 | 关键难点 |
|------|---------|---------|---------|
| **上传存储** | 大文件分块+CDN | 视频、网盘、图床 | 秒传、断点续传、分发 |
| **搜索检索** | 倒排索引+排序 | 搜索、商品筛选 | 相关性、实时性 |
| **推荐匹配** | 用户×物品打分 | 信息流、猜你喜欢 | 冷启动、多样性 |
| **审核过滤** | 内容合规检查 | 评论审核、图片鉴黄 | 实时/异步、召回率 |

## 2. 拆服务：三条黄金线

**不要按功能模块拆**（会拆出贫血服务），按三条线：

| 线 | 判断方法 | 例子 |
|----|---------|------|
| **① 数据所有权（最核心）** | 两个服务都写同一张表 → 必须合并 | 红包金额池归 RedPacketService，GrabService 只调用不直接写 |
| **② 一致性边界** | 失败是否必须一起回滚？是→合；否→拆 | 扣金额+写流水（同事务）/ 入账+发通知（拆 + MQ） |
| **③ 变化频率/SLA** | 发版节奏差 5x 以上 → 必须拆 | 视频播放（核心）vs 推荐（算法迭代快）必须分 |

**口诀：同主同事同频 → 合；异主异事异频 → 拆。**

**反模式**：❌ 按 CRUD 分（忽略业务流）❌ 按技术栈分（分层不是拆服务）❌ 无限微服务化（治理爆炸）✅ 按业务能力拆（Bounded Context，每个服务独立上线）

## 3. 建模：四问法

### 问 1：核心名词是什么？→ 找实体
让业务讲一遍流程，把**名词**全记下，去重合并，筛出**有独立生命周期、能被 CRUD 的东西**。
- 红包场景：用户/红包/金额/订单/流水/账户 → 实体 = { User, RedPacket, Order, Account }（金额、流水是属性或事件，不是实体）

### 问 2：核心动作是什么？→ 找事件
所有**动词**就是事件，每事件对应一次状态变迁 + 一条流水。
- 红包动作：发/抢/拆/退款 → Event = { Created, Grabbed, Opened, Refunded }
- 每事件三问：**触发者是谁？改变哪个实体状态？需要记流水吗？**（金融必记）

### 问 3：怎么查？→ 找读模型
问产品"用户会从哪几个入口看到这个数据？" 每个入口 = 一个读模型 = 一张反范式表/Redis 结构。
- 红包入口："我发的" → sender_id 索引表；"我抢的" → receiver_id 索引表；"详情" → 主表；"统计" → 预计算快照

**核心洞察：写模型和读模型必须分开设计。** 写优化一致性，读优化性能，两者几乎总是冲突。

### 问 4：什么不能错？→ 找一致性约束
问产品"什么情况算事故？赔钱吗？"
- 红包金额不平=资损→金额守恒强一致 / 抢两次=资损→ user_id+packet_id 唯一索引 / 抢包延迟 3s=不算事故→异步入账可 / 查询延迟 1min=不算事故→读缓存足够

**不能错的地方用强一致，能容错的地方用最终一致**——这就是 CAP 在业务层的体现。

### 四问顺序不能乱
**名词（实体）→ 动词（事件）→ 查询（读模型）→ 约束（不能错）**。逆序会越来越混乱，正序自然收敛。

## 4. 库表：从模型到表的机械映射

### 映射规则

| 模型元素 | 映射到表 | 分片键 | 索引 |
|---------|---------|-------|------|
| 实体 | 主表（聚合根） | 业务主键 hash | 主键 |
| 状态 | status 字段（单向流转） | - | 按 status 查询索引 |
| 事件 | 流水表（append only） | 按时间 or 实体 ID | 实体 ID + 时间 |
| 读模型 | 反范式表 / Redis 结构 | 按查询维度 | 查询维度 |
| 一致性约束 | 唯一索引 + 乐观锁 version | - | 业务幂等键 UNIQUE |

### 分片键选择铁律
**看最高频的查询 WHERE 条件是什么，它就是分片键。**
- 红包 90% 按 packet_id 查 → packet_id 分片 / 订单 70% 按 user_id 查 → user_id 分片
- 两个维度都高频（如订单按 user 和 merchant 都要查）→ **建两份数据**（空间换时间）

### 字段设计通用规范

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 雪花 ID，全局唯一 |
| biz_key | VARCHAR + UNIQUE | 业务幂等键 |
| status | TINYINT | 状态机字段 |
| version | INT | 乐观锁 |
| create_time / update_time | DATETIME(3) | 毫秒精度 / ON UPDATE 自动更新 |
| extra | JSON/VARBINARY | 扩展字段（避免频繁加字段） |
| deleted | TINYINT | 软删标记（可选） |

## 5. 理清流程：一张泳道图打穿

陌生业务流程复杂时，画**四条泳道图**就够。

### 5.1 正常主流程（happy path）

```
  用户 |  网关  | 业务服务 | 缓存 | MQ  | DB  | 下游服务
──────┼───────┼─────────┼──────┼─────┼─────┼──────────
请求 ─▶│ 鉴权─▶ │ 查缓存─▶ │(miss)────▶│     │
      │       │ 写 DB───────────────▶│     │
      │       │ 发 MQ─────────▶│───────────▶消费处理
      │◀─────返回
```



### 5.2 核心异常路径
- DB 写成功 + MQ 发失败 → 本地消息表补偿
- DB 写失败 + 前端重试 → 幂等键保证不重复
- 下游超时 → 熔断 + 降级

### 5.3 热点路径（最高 QPS 专门设计）
- 红包抢包：Lua 原子 + 预拆金额 + 本地队列
- 热门视频：CDN + 本地缓存 + singleflight
- 大 V 发微博：读扩散 + 异步写扩散

### 5.4 对账/兜底路径
- 日级全量对账脚本（凌晨低峰）→ 差异 > 阈值 → 自动补偿 or P0 告警 → 人工介入口子（运营后台）

## 6. 核心关注点：从"不能错"反推

**核心关注点不是背的，是从四问法的问题 4 反推的：**

| 业务担心的"不能错" | 核心关注点 | 标准解法 |
|------------------|-----------|---------|
| 钱不能错 | 金额守恒、幂等、对账 | 唯一索引 + 日级对账 + 状态机 |
| 不能重复触发 | 幂等键、状态机、去重 | Redis SETNX + DB UNIQUE |
| 不能丢 | 持久化、至少一次 + 幂等消费 | WAL + MQ ack + 幂等消费 |
| 不能慢 | 缓存、异步、分片 | 多级缓存 + MQ 削峰 + 分库 |
| 不能被刷 | 限流、风控、验证码 | 令牌桶 + 设备指纹 + 人机 |
| 不能泄露 | 鉴权、脱敏、审计 | RBAC + 脱敏字段 + 审计日志 |
| 不能被攻击 | 枚举防护、签名、灰度 | 非连续 ID + HMAC + 灰度开关 |

**每个"不能错"都对应一套标准解法**——这就是跨业务场景解法能复用的原因。

## 7. 进阶方法论：被省略的"高深"部分

**前面六节是战术层，下面是战略层** —— 大厂架构师隐式在用但很少明说的东西。

### 7.1 Event Storming（事件风暴）—— 超复杂业务的终极武器

业务复杂到**四问法**都拆不清时（保险理赔、银行核心）用它。**流程**（大白板 + 便签纸，2~4 小时）：

1. **混沌风暴**：所有人把业务事件（过去式动词）写便签贴满墙。"订单已创建"、"支付已完成"、"库存已扣减"
2. **时间线排序**：按时间顺序排成一条线
3. **找命令和聚合**：每事件前的命令是什么？作用在哪个聚合根上？
4. **找边界**：哪些事件/命令/聚合属于同一业务能力？圈起来 → 服务边界（Bounded Context）
5. **找热点**：哪些地方有争议？红色便签标记 → 设计难点

**为什么有效**：事件是业务中最稳定的元素（实体会变、流程会变，但"发生过什么"不会变）。以事件为锚点倒推的模型比以实体为中心的更稳定。

### 7.2 CQRS（命令查询职责分离）—— 读写模型彻底分离

```
Command Side（写）              Query Side（读）
强一致 + 复杂事务       ───▶    最终一致 + 高性能
严格领域模型                    反范式 + 物化视图
写入 MySQL                      读取 ES / Redis / CK
保证正确性                      保证体验
```

**适用**：读写比 > 10:1；读查询维度多；写一致性高但读可容忍延迟。**典型实现**：写 MySQL → Binlog → Kafka → 同步到 ES/Redis/CK → 查询侧读。

### 7.3 DDD 真正有用的 3 个概念（其他都是花架子）

- **① Bounded Context（限界上下文）= 服务边界**：同一词在不同上下文含义不同（订单系统的"用户"是买家，风控系统的"用户"是风险画像）。不要追求全公司统一 User 模型，那是灾难。
- **② Aggregate（聚合根）= 事务边界**：一次事务只能修改一个聚合根；跨聚合 → 最终一致（事件驱动）。例：订单聚合 = 订单主表 + 订单项 + 订单地址（一个事务），但不包括用户账户。
- **③ Domain Event（领域事件）= 服务间通信核心**：服务间不直接 RPC，用事件。订单发"OrderCreated"，库存/积分/通知各自订阅，解耦彻底。

**其他 DDD 术语（Entity/Value Object/Repository/Factory）可以不管，增加心智负担无收益。**

### 7.4 四色建模法（UML Color Modeling）—— 金融/交易类万能模板

| 颜色 | 含义 | 例子 |
|------|------|------|
| **粉（Moment-Interval）** | 时刻/时段，有生命周期的事件 | 订单、支付流水、合同 |
| **黄（Role）** | 参与方角色 | 买家、卖家、担保人 |
| **绿（Party/Place/Thing）** | 人、地、物 | 用户、商品、仓库 |
| **蓝（Description）** | 分类/规格/模板 | 商品类别、合同模板、规格 |

**建模顺序**：粉（核心事件）→ 黄（谁参与）→ 绿（哪个人/物）→ 蓝（什么类型）。**适用**：有"交易/合约/流水"的业务，10 分钟搭骨架。

### 7.5 业务不变式驱动设计（最高段位）

**从"永远成立的不变式"倒推模型：**
- 红包：`sum(已抢) + sum(剩余) = 初始总额`
- 订单：`订单金额 = sum(订单项) + 运费 - 优惠`
- 账户：`借方总额 = 贷方总额`（复式记账）
- 库存：`可售 + 锁定 + 已售 = 总库存`

**方法**：先写 3~5 条不变式 → 模型设计必须保证它们永远成立 → 自然推出强一致边界、事件设计、对账逻辑。**这是金融核心系统架构师的核心能力，很少人明说。**

### 7.6 康威定律的反向利用

> **系统架构必然反映组织沟通结构。**

**反向利用：想让系统怎么拆，就先让组织怎么分。** 两个团队共享一个服务 → 必然变巨石 / 一个团队维护三个服务 → 必然耦合。陌生领域建模时先问"**未来由几个团队维护？每个团队管什么？**" 团队边界往往就是服务边界的最优解。

### 7.7 正交分解原则

**复杂业务 = 多个正交维度的组合，找到正交维度就找到模型的基。**
- 电商 = 用户 × 商品 × 订单 × 营销
- 广告 = 广告主 × 用户 × 展位 × 计费

**如果两个维度可独立变化 → 必须拆成独立模型**，耦合了必后悔。

## 8. 陌生领域 30 分钟出方案的实战流程

```
① 5min 原型匹配：让业务方讲一遍 → 过"原型库" → 识别组合
② 5min 四问建模：名词（实体）→ 动词（事件）→ 查询（读模型）→ 约束（不能错）
③ 5min 拆服务：三条黄金线 → 画出 3~5 个服务及职责
④ 5min 画主流程：泳道图（用户/网关/服务/缓存/MQ/DB/下游），只画正常主流程
⑤ 5min 库表设计：实体→主表，事件→流水，读模型→反范式；选分片键
⑥ 5min 核心关注点：从"不能错"反推 → 套标准解法（幂等/对账/缓存/限流）
```

## 9. 终极心法

1. **业务本质是 `Actor × Object × Action × State`。** 术语会变，结构不变。陌生业务：先匹配原型 → 按四问法建模 → 30 分钟出框架。
2. **建模用四问法**（名词→动词→查询→约束）、**拆服务看三条线**（所有权/一致性/频率）、**核心关注点从"不能错"反推**。这三招覆盖 90% 的业务建模需求。
3. **陌生领域 ≠ 没见过，而是没抽象**。世上只有 20~30 种业务原型，剩下都是组合。**把业务翻译成抽象，把抽象匹配到原型，把原型套用到方法论** —— 这就是架构师跨领域的底层能力。

## 10. 进阶工具箱

| 场景 | 推荐方法 | 场景 | 推荐方法 |
|------|---------|------|---------|
| 流程特别复杂（保险/银行核心） | Event Storming | 金融核心系统 | 业务不变式驱动 |
| 读写比失衡严重 | CQRS | 组织架构复杂 | 康威定律反向设计 |
| 跨服务一致性 | 聚合根 + 领域事件 | 业务多个独立维度 | 正交分解 |
| 金融/交易业务 | 四色建模法 | 一般互联网业务（默认） | 四问法 + 三条线 |
