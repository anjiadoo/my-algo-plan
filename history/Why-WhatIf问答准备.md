# Why / What if 问答准备

> 面试官不只问"你做了什么"，更会追问"为什么这样做"和"如果换一种方式呢"。
> 本文档按简历每个技术点，准备对应的设计权衡、替代方案分析、失败场景应对。

---

## 一、互娱通用抽奖系统

### 1. 插件化引擎

| 问题 | 回答 |
|------|------|
| **Why 插件化而不是 if-else 分支？** | 10+款游戏玩法差异大（LOL要定制道具响应格式、剑侠情缘要按服务器等级选奖池），if-else 维护成本爆炸，一改全动。插件后注册覆盖先注册，新游戏只需实现差异插件，核心流程零改动。 |
| **Why 后注册覆盖而不是优先级链？** | 简单。优先级链需要管理多个插件的执行顺序和冲突，后注册覆盖是最简模型——同一 Hook 点只有一个生效实现，语义明确无歧义。 |
| **What if 插件执行出错？** | 每个 Hook 点有默认实现（空操作或返回默认值），插件 panic 被 recover 捕获后降级到默认行为 + 告警，不影响主流程。 |
| **What if 新游戏需要的 Hook 点不存在？** | 15+ Hook 点已覆盖主流程所有环节。极端情况下新增 Hook 点成本很低（在主流程对应位置加一行调用），但需要评审避免 Hook 泛滥。 |

### 2. 分层概率模型与采样机制

| 问题 | 回答 |
|------|------|
| **Why 轮盘赌而不是 Alias Method？** | 简历里实际场景是过滤后概率组 per-user 不同且 per-request 可变（每个用户已抽道具不同→过滤后概率组不同），Alias 表需要 O(n) 预构建，如果每次请求都重建就失去了 O(1) 优势。轮盘赌虽然是 O(n)，但零预处理、天然适配动态过滤场景，n 通常<100，实际耗时可忽略。 |
| **Why 系统设计文档里写了 Alias Method？** | 系统设计文档是理想化的通用设计（假设奖池固定、道具数量可能达数百）。实际落地中，因为过滤链的存在导致每次概率组都是动态的，所以选了轮盘赌。这是"设计 vs 落地"的常见 trade-off。 |
| **Why Fixed/Unfixed 双轨？** | 保证"保底道具/兜底道具"概率恒定不受过滤影响。比如金币作为 Fixed 道具永远占 30% 概率，其他 Unfixed 道具被过滤后在剩余 70% 中按比例重分配，无需手动调权重。 |
| **Why CSPRNG + xoshiro256 而不是纯 math/rand？** | 抽奖涉及付费，需要防止玩家/内部人员通过种子预测结果。CSPRNG（crypto/rand）产生安全种子，xoshiro256 在安全种子基础上快速生成后续随机数，兼顾安全性和性能。纯 math/rand 种子可被逆推，有合规风险。 |
| **What if 概率组嵌套层数过深？** | 限制最大10层递归。实际业务中2-3层就够了（层级→稀有度→道具），10层是防御性上限。超过直接报错拒绝配置发布。 |

### 3. 三级幂等保障机制

| 问题 | 回答 |
|------|------|
| **Why 三级而不是只用分布式锁？** | 分层防御，每层解决不同场景：① 响应缓存解决网络重试（最高频，不需要加锁直接返回）；② 状态回退/回放解决中间链路异常（锁已释放但结果未正确持久化）；③ 分布式锁解决真正的并发请求。如果只有锁，每次重试都要竞争锁再走完整流程，性能差且无法处理"锁已释放但状态不一致"的场景。 |
| **Why 响应缓存放在加锁之前？** | 重试请求是最高频场景（客户端超时重发、动画重播触发）。如果先加锁再查缓存，重试请求也要竞争锁，白白增加 Redis 竞争。缓存前置可以在无锁状态下直接返回，大幅减少锁竞争。 |
| **Why 双缓存回退 vs 回放模式互斥？** | 两种模式语义不同：回退模式适合"结果可以不同但状态必须一致"（重新随机也行）；回放模式适合"结果必须完全一致"（活动侧已展示过结果）。业务方按需选择，不能同时启用否则语义冲突。 |
| **What if Redis 响应缓存过期了？** | 走第二层：检查 UserRealTimeState 中的 drawRound 是否已执行过，若已执行则进入回退/回放逻辑重建结果。缓存只是加速层，不是唯一保障。 |
| **What if Redis 分布式锁超时未释放？** | SETNX 设置了 TTL（通常3-5秒），到期自动释放。业务逻辑正常执行时间 P99<50ms，TTL 远大于正常执行时间，不会误释放。极端情况下 TTL 释放后另一个请求进来，会被第一层缓存或第二层状态校验拦住。 |

### 4. 四层限量管控机制

| 问题 | 回答 |
|------|------|
| **Why Redis 原子递增而不是分布式锁？** | 限量道具可能被高并发命中（爆款皮肤），分布式锁会让所有请求排队等待，吞吐量急剧下降。Redis INCRBY 是原子操作，单分片 10万+ QPS，不存在锁等待。超限时 DECRBY 回补即可。 |
| **Why 采样后检查而不是采样前过滤？** | 限量道具通常概率很低（SSR），绝大多数抽奖不会命中。如果采样前先查库存再决定是否放入池中，每次请求都多一次 Redis 查询；采样后检查只在命中时才查，99%的请求零额外开销。 |
| **Why 最多重抽3次？** | 防止极端场景下（池中只剩限量道具且全部耗尽）无限循环。3次重抽失败后走兜底概率组返回金币等保底道具，保证每次抽奖必有结果。 |
| **What if Redis 主从切换导致库存计数不准？** | MySQL 对账兜底。定时任务对比 Redis 库存和 MySQL 已发放记录，发现差异自动告警。极端情况下（Redis 数据丢失），从 MySQL 恢复 Redis 计数。"Redis 权威 + MySQL 兜底"是经典双写模式。 |
| **What if 高并发下多个请求同时 INCRBY 超过库存？** | 不会。Lua 脚本里是 `INCRBY → 判断是否 > total → 超了就 DECRBY 回补`，整个操作原子执行。多个请求顺序执行Lua，只有先到的成功，后到的在INCRBY后发现超限立即回补。 |

### 5. 里程碑保底系统

| 问题 | 回答 |
|------|------|
| **Why 幸运值累加+阈值切换而不是直接修改概率？** | 解耦。幸运值是通用计数器，阈值和目标概率组通过配置绑定。这样同一套幸运值机制可以支持硬保底（阈值=90次→100%SSR）、软保底（阈值=74次→概率递增）、大小保底（阈值50→50/50切换）等所有玩法，玩法变更只改配置不改代码。 |
| **Why 多条规则多优先级？** | 一个奖池可能同时有多种保底（如：90次硬保底 + 10次保底必出SR + 大小保底）。多条规则按优先级排序，匹配第一条命中的规则。优先级保证高稀有度保底优先触发。 |
| **Why 幸运值存在 UserRealTimeState 中而不是独立存储？** | 减少 Redis 访问次数。每次抽奖必读 UserRealTimeState，幸运值内嵌其中不需要额外一次 GET。状态原子更新时幸运值一起更新，保证一致性。 |
| **What if 玩家在保底前弃坑，回来后保底还在吗？** | 在。幸运值持久化在 Redis + MySQL 异步落盘，不过期（除非活动下线清理）。跨池继承场景下，即使换了新池，旧池的幸运值可以配置继承到新池。 |

### 6. Layer 多层隔离

| 问题 | 回答 |
|------|------|
| **Why Layer 隔离而不是清理旧状态重用？** | 清理旧状态有风险——万一清理不干净就数据污染。Layer 隔离是"加法"思维：新 Layer 天然干净，旧 Layer 保留可回溯。切换 Layer 等于开新局，零清理成本。 |
| **Why 每个 Layer 独立 Redis 状态/奖池/概率表/缓存？** | 完全隔离保证不同 Layer 之间互不影响。比如赛季1的已抽道具不会影响赛季2的过滤；轮次1的限量道具耗尽不影响轮次2。如果共享状态就需要大量"归属Layer"的判断逻辑，复杂度爆炸。 |
| **What if Layer 数量越来越多，Redis 内存爆炸？** | 过期 Layer 的 Redis Key 设置 TTL 自动清理。活动下线后触发异步清理任务，删除该活动所有 Layer 的 Redis 数据。长期数据在 MySQL 有持久化，需要恢复时从 MySQL 重建。 |
| **What if 玩家在 Layer 切换时正好在抽奖？** | Layer 切换由客户端请求参数中的 Layer 标识决定，不是服务端主动切换。客户端先收到"进入下一轮"的通知，再发起新 Layer 的抽奖请求。不存在"服务端切了Layer但客户端还在用旧Layer"的竞态。 |

### 7. 多维过滤链 + 兜底概率组

| 问题 | 回答 |
|------|------|
| **Why 串行过滤链而不是合并为一次查询？** | 每个过滤器数据源不同：有效期查配置、已有查状态、背包查外部接口、限量查 Redis。合并为一次查询不现实。串行链的好处是每个过滤器独立可配置（业务A只需有效期+已有，业务B需要全部6层），灵活组合。 |
| **Why 过滤后缩放而不是过滤前重新计算？** | 性能。过滤是"移除不符合的道具"，缩放是"剩余道具概率按比例放大到总和=1"。如果过滤前重新计算，意味着每个用户每次请求都要重建完整概率组，开销大。过滤+缩放只需要简单的除法。 |
| **Why 需要兜底概率组？** | 极端场景：无放回抽奖抽完了、所有限量道具都耗尽了、过滤后池为空。没有兜底就返回空结果，客户端崩溃。兜底保证"每次抽奖必有产出"，最差情况返回金币。 |
| **What if 兜底概率组也为空？** | 返回配置的 fallback 道具（如金币/碎片），这是最终兜底，不可能为空（系统初始化时必须配置 fallback）。 |
| **What if 过滤链执行耗时过长？** | 6层过滤都是内存操作或单次 Redis 查询，总耗时<5ms。背包过滤（外部接口）是最慢的一环，做了本地缓存（TTL 5min），首次穿透后续走缓存。 |

## 二、QQ 音乐直播系统

### 1. 连麦/PK 状态机

| 问题 | 回答 |
|------|------|
| **Why CAS 乐观锁而不是分布式锁？** | PK 状态变更由多种事件触发（心跳超时检测、客户端操作、定时器回调），频率高。分布式锁每次变更都要竞争锁，开销大。CAS 乐观锁只在写入时校验版本号，大多数情况下无冲突直接成功，冲突时重试即可。 |
| **Why 断线保护 15s 而不是立即判负？** | 网络抖动是直播场景的常态（移动网络切换、弱信号区），3-5秒的短暂断连非常高频。立即判负会导致大量"冤枉判负"投诉。15s 覆盖了绝大多数短暂断连恢复时间，超过15s大概率是真断线。 |
| **Why 邀请超时30s？** | 平衡等待体验和资源占用。<15s 主播可能没看到通知就超时了；>60s 等待方会失去耐心。30s 是经验值（友谊PK实际用60s因为要切场景，随机PK匹配超时120s因为可能等待配对）。 |
| **Why 定时器用 Redis ZSET + 500ms 轮询而不是延迟队列？** | 实现简单可控。Redis ZSET 天然支持按 score 范围查询到期项；500ms 轮询精度足够（PK 是分钟级活动，500ms 误差不影响体验）。延迟队列（如 RocketMQ 延迟消息）引入额外中间件依赖，且精度不一定更好。 |
| **What if 两个信令同时到达（如"接受"和"超时取消"）？** | CAS 校验。两个操作都带当前版本号，只有版本匹配的那个能写成功。比如超时取消操作的版本号是 v3，但"接受"操作先执行将版本号更新为 v4，超时取消操作发现版本号不匹配就放弃。 |
| **What if 匹配池中的主播状态已变（匹配到时已不空闲）？** | 匹配成功后有"二次状态校验"：锁定双方 showID → 重新读取状态 → 确认双方仍空闲 → 才修改为 Accepting。不空闲则放弃本对配对，等待下一轮。 |

### 2. PK 积分系统与实时结算

| 问题 | 回答 |
|------|------|
| **Why Lua 脚本而不是分布式锁？** | 加分是高频操作（每秒可能多笔礼物），分布式锁会让所有加分请求排队等待，延迟不可接受。Lua 脚本在 Redis 单线程中原子执行，一次网络往返完成"去重+加总分+加榜单"三步，无等待开销。 |
| **Why 用 pk_status 标志位互斥而不是同一把锁？** | 结算和加分是不同入口、不同频率的操作。加分高频（每秒多次），结算低频（PK结束时一次）。如果共享一把锁，高频加分会阻塞结算，结算也会阻塞加分。标志位互斥是"写后读"模型：结算置 finished 后，加分检测到就拒绝，无竞争无等待。 |
| **Why 订单号去重而不是"加分前先查状态"？** | 支付回调可能重试（网络超时后运营商重发），同一笔礼物可能触发多次加分回调。如果只查 pk_status，重试的回调仍然会重复加分。订单号去重（SISMEMBER）保证同一笔礼物只计一次分。 |
| **Why 500ms 轮询结算而不是精确延迟触发？** | PK 是分钟级活动（通常3-5分钟），500ms 误差对用户体验无感知。精确延迟触发（如 Redis Key 过期事件）不可靠（Redis 过期是惰性的，不保证准时触发），反而不如轮询稳定。 |
| **What if 结算瞬间和加分同时执行 Lua？** | Redis 单线程，两个 Lua 脚本不可能真正"同时"。必然有先后：如果加分先执行完→结算读到最新积分→正确判定；如果结算先执行完→加分检测到 finished→被拒绝。两种情况都正确。 |
| **What if 加分成功但 IM 广播失败？** | 积分数据以 Redis 为准（加分 Lua 已成功），IM 广播是"尽力而为"。客户端通过心跳对账（10s一次）或进房全量同步获取最新积分，即使广播丢了也会自动修复。 |

### 3. 麦位管理与信令可靠性

| 问题 | 回答 |
|------|------|
| **Why 以 TRTC 回调为最终确认而不是客户端上报？** | 客户端不可信。客户端可能伪造"上麦成功"跳过 TRTC 鉴权；也可能网络问题导致客户端以为成功但实际 TRTC 没收到。TRTC 回调是第三方可信源——TRTC 说进房了才算真进房。 |
| **Why ACK 重试而不是"推失败就算了"？** | 连麦信令（上麦通知、PK 邀请）是关键业务操作，丢失会导致状态不一致（服务端已变更状态，客户端不知道）。ACK 保证关键信令必达。普通弹幕可以丢，但信令不行。 |
| **Why 锁定麦位的设计？** | 主播场控需求。比如预留某个麦位给特定嘉宾，不希望被随机观众抢占。锁定是业务需求驱动的功能，不是纯技术设计。 |
| **What if 用户进了 TRTC 房间但服务端没收到回调？** | TRTC 回调有重试机制。如果确实丢了，客户端会发现自己在推流但麦位状态未更新，通过心跳对账触发重新确认。极端情况下运维巡检服务（MliveConnMonitorSvr）定期比对 TRTC 房间成员和麦位状态。 |
| **What if 主播断线，麦位上的连麦者怎么办？** | 连麦者不受主播断线影响（都在同一 TRTC 房间，房间不会因主播断线立即销毁）。服务端通过断流保护窗口等待主播重连；超时后触发关播流程→所有连麦者收到退出通知→释放麦位。 |

### 4. 直播弹幕系统

| 问题 | 回答 |
|------|------|
| **Why 拉模型而不是推模型（WebSocket Push）？** | ① 拉模型实现更简单——客户端定时拉取，服务端无需维护每个连接的推送状态；② 更可控——过载时客户端自动降速（拉取间隔可调），不会因推送风暴压垮网关；③ 空查询率仅 15%（大房间几乎每秒都有新消息），拉模型的"浪费"很小；④ 成本在带宽不在 CPU，推拉结合的切换逻辑反而增加 CPU 开销。 |
| **Why 大小房间隔离？** | 80/20 原则：少数热门房间（签约主播）贡献大部分流量。如果不隔离，一个热门直播间的流量洪峰会影响同集群所有小房间的延迟。隔离后热门房间单独集群，互不影响。 |
| **Why 小房间用一致性 Hash 而大房间用无状态 L5？** | 小房间需要"状态亲和"——同一房间的长轮询请求路由到同一台机器，该机器本地缓存该房间的最新消息，减少存储穿透。大房间请求量大到单机无法承载，必须无状态打散到多台。 |
| **Why 合并写（100ms/10条）？** | 热门房间消息密度高（>200条/s），逐条写 CKV 会产生大量 CAS 冲突（多个写请求同时修改同一 Key 的消息列表）。合并写减少写入次数 = 减少 CAS 冲突 = 减少重试 = 降低延迟。 |
| **What if 合并写窗口内服务重启？** | 重启时在 handle_fini 中将未写入的消息强制落盘。另外定时 200ms 检查未写入消息兜底。即使极端丢失，也只影响几十毫秒窗口内的普通弹幕（非关键消息）。 |
| **What if 单房间消息量超过 200条/s 限流？** | 超出的消息被丢弃（仅普通弹幕）。客户端显示极限本身就是 70-100 条/s，超过人眼看不清。信令类消息（PK/连麦状态）走独立优先级通道，永不被限流丢弃。 |

### 5. 开关播流程与并发控制

| 问题 | 回答 |
|------|------|
| **Why Lua 限制 3s 去重而不是幂等 Token？** | 幂等 Token 需要客户端配合（携带 Token），但频繁创建的根因是客户端 Bug 或网络抖动——客户端可能不会正确携带 Token。Lua 在服务端强制保证"3s 内同一用户只能创建一个 ShowInfo"，不依赖客户端行为。 |
| **Why 3s 而不是更长/更短？** | 太短（<1s）：正常重试也会被拦截；太长（>10s）：主播真的想重新开播（比如手动关播后立即重开）会被误拦。3s 是经验值，覆盖网络重试（通常<2s）但不影响正常操作。 |
| **Why UDID 而不是 IP 判断跨设备？** | 同一 WiFi 下多台手机 IP 相同，无法区分设备。UDID 是设备级唯一标识，精确到每一台手机。 |
| **Why "UDID一致恢复，不一致拒绝" 而不是都拒绝？** | UDID 一致说明是同一台手机（APP 崩溃重启、切后台回来），恢复直播是更好的用户体验。UDID 不一致说明是另一台设备，可能是账号被盗或误操作，拒绝并提示更安全。 |
| **What if UDID 被伪造？** | UDID 由客户端 SDK 生成并加密上报，普通用户无法伪造。即使被伪造，最坏结果是"恢复了别人的直播"——但直播间内容是实时推流的，伪造者还需要拿到 TRTC 密钥才能真正推流，实际安全风险可控。 |

### 6. 进退房流程与在线观众管理

| 问题 | 回答 |
|------|------|
| **Why TRTC/CDN 双链路而不是全部走 CDN？** | TRTC 支持实时互动（连麦、PK 需要<300ms 延迟），CDN 只能看不能互动（1-3s 延迟）。小房间用 TRTC 保留互动能力；超大房间（>10万人）超出 TRTC 承载能力才降级到 CDN。 |
| **Why 快速进房（并行）而不是串行？** | 串行进房 = 业务接口耗时 + TRTC进房耗时，用户感知到明显的"加载中"。并行后用户先看到画面（TRTC 进房快），再渐进式加载房间装扮，体感"秒进"。 |
| **Why 在线观众组件用 CKV 而不是 Redis？** | 腾讯内部 CKV 是专门优化的分布式 KV 系统，比 Redis 更适合大规模在线状态管理（内存成本更低、集群管理更成熟）。CKV+ 类似 Redis（支持数据结构），CKV 类似 Memcache（纯 KV）。 |
| **Why 心跳 30s 而不是更短/更长？** | 太短（<10s）：心跳请求量 = 在线人数/间隔，间隔越短 QPS 越高，百万用户 × 1s心跳 = 100万 QPS 心跳，不可接受。太长（>60s）：用户退出后在线列表延迟清除过慢，影响在线人数准确性。30s 是平衡点。 |
| **Why MQ 解耦进退房上报而不是同步写？** | 进退房是高频操作（热门直播间每秒数百人进出），同步写在线列表会拖慢 JoinShow 接口延迟。MQ 解耦后 JoinShow 立即返回，在线列表异步更新，用户体验不受影响（在线人数不需要实时精确）。 |
| **Why 统一排行榜按 Key 聚合批量并行计算？** | 热门主播的贡献流水集中在同一个 Key，逐条写入 CAS 冲突严重。聚合后同一 Key 的多笔流水合并为一次写入，大幅降低 CAS 冲突和 Redo 次数。 |
| **What if TRTC 房间到达 10万人上限？** | JoinShow 接口判断房间在线人数，超过阈值后新用户自动切换为 CDN 进房。CDN 进房只能拉流观看，不能连麦互动——但超大房间（演唱会）本身也不需要连麦。 |
| **What if 在线观众组件和排行榜数据不一致？** | 对账进程定期消费"对账任务"（按时间周期固化上一个周期的数据），对比在线组件数据和流水记录，发现差异自动告警修正。因为每笔流水都落库，所以最终一致性有保障。 |

## 三、通用高频 Why 问题

| 问题 | 适用场景 | 通用回答模板 |
|------|---------|-------------|
| **Why Redis 而不是 DB？** | 限量库存、在线状态、PK积分 | 高并发场景 DB 无法承载（单主库3000TPS），Redis 单分片10万QPS，且支持 Lua 原子操作。DB 做兜底对账。 |
| **Why 异步（MQ）而不是同步？** | 发货、进退房上报、开播事件 | 非关键路径同步执行会拖慢主接口 RT。MQ 解耦后主接口立即返回，下游异步处理，通过重试保证最终一致。 |
| **Why Lua 脚本而不是事务？** | 库存扣减、积分加减、去重 | Redis 不支持真事务（MULTI/EXEC 不支持条件判断），Lua 脚本在单线程中原子执行复合逻辑（if-then-else），一次 RTT 完成。 |
| **Why 最终一致而不是强一致？** | 发货、在线人数、统计数据 | 强一致要求分布式事务（2PC/TCC），性能开销 10 倍以上。业务容忍秒级延迟（用户看到"发货中"可接受），最终一致 + 对账兜底足够。 |
| **Why 本地缓存 + Redis 而不是只用 Redis？** | 配置数据、周榜数据 | 配置/榜单数据变更低频但读取极高频。本地缓存（atomic.Value）零网络开销，3s 更新一次覆盖 99% 读请求，减少 Redis 压力。 |

## 四、面试应对技巧

**当面试官问 Why 时：**
1. 先说**选择了什么**（1句话）
2. 再说**核心理由**（性能/简单性/可靠性，挑最关键的1个）
3. 最后说**对比方案的缺点**（为什么不选另一个）

**模板：** "我们选了X方案，核心原因是[具体量化的理由]。另一个选择是Y，但Y的问题是[具体缺点]。"

**示例：** "积分加减用 Lua 脚本而不是分布式锁，核心原因是加分是高频操作每秒可能几十次，分布式锁每次都要两次网络往返（SETNX + DEL），延迟不可接受。Lua 一次 RTT 就完成去重+加分+更新榜单，性能好一个数量级。"
