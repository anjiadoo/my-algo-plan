## 1. 连麦/PK 状态机设计与实现

### 1.1 核心流程图：随机 PK 完整生命周期

```
┌───────────────────────────────────────────────────────────────────────┐
│                        PK 状态机 (CAS 版本号防并发跳变)                   │
│                                                                       │
│   空闲 ──→ 邀请中 ──→  连接中  ──→   PK中  ──→ 结算中 ──→ 空闲    │        │
│    ↑         │              │           │                    │        │
│    │         │超时/拒绝      │超时        │一方断线>15s         │        │
│    └─────────┴──────────────┴───────────┴────────────────────┘        │
│                                                                       │
│   每次状态变更: CasPK++ (CAS乐观锁, 防止并发信令覆盖)                       │
└───────────────────────────────────────────────────────────────────────┘
```

### 1.3 跨房 PK 音视频链路

```
┌────────────┐                                        ┌────────────┐
│  主播 A    │                                        │  主播 B    │
│ (TRTC房间) │                                        │ (TRTC房间) │
└─────┬──────┘                                        └──────┬─────┘
      │                                                      │
      │   ┌──────────────────────────────────────────────┐   │
      └─► │          同一个 TRTC 房间 (端到端<300ms)       │◄───┘
          └────────────────────┬─────────────────────────┘
                               │
                    ┌──────────▼──────────┐
                    │    云端 MCU 混流      │
                    │  (左右分屏 360x640x2) │
                    └───┬────────────┬────┘
                        │            │
              ┌─────────▼──┐     ┌───▼──────────┐
              │ CDN流(A视角) │    │ CDN流(B视角)  │
              │ A在左,B在右  │    │ B在左,A在右   │
              └─────┬──────┘     └──────┬───────┘
                    │                  │
            ┌───────▼───────┐  ┌───────▼───────┐
            │ A的观众 (拉流)  │  │ B的观众 (拉流)  │
            │ 延迟 1~3s      │  │ 延迟 1~3s      │
            └───────────────┘  └───────────────┘

连麦者链路: TRTC实时通道, 端到端 < 300ms
观众链路:   MCU混流 → CDN旁路推流, 延迟 1~3s, 成本低
```

### 1.5 面试记忆要点

| 关键词 | 说明 |
|--------|------|
| 状态机6态 | idle→inviting→matching→pk_ing→punishing→finished |
| CAS防跳变 | 每次变更CasPK++，操作前校验版本号 |

## 2. PK 积分系统与实时结算机制

### 2.1 核心流程图：积分写入（Lua原子去重+加分）

```
观众送礼
    │
    ▼
支付服务确认 → 生成唯一 order_id
    │
    ▼
支付回调到达 PK 服务
    │
    ▼
┌─────────────────────────────────────────────────────┐
│            加分 Lua 脚本 (原子执行)                    │
│                                                     │
│  ① GET pk_status                                   │
│     └─→ == "finished"? → return 0 (拒绝, PK已结束)    │
│                                                     │
│  ② SISMEMBER order_set, order_id  (幂等检查)         │
│     └─→ 已存在? → return 0 (重复回调, 跳过)            │
│                                                     │
│  ③ SADD order_set, order_id  (标记已处理)            │
│                                                     │
│  ④ INCRBY pkTotalGrant_{showID}_{session}, score   │
│     (总积分递增, 用于最终判定胜负)                      │
│                                                     │
│  ⑤ ZINCRBY pkContribution_{showID}_{session},       │
│     score, user_id  (贡献榜递增, 展示TopN)            │
│                                                     │
│  → return 1 (成功)                                  │
└─────────────────────────────────────────────────────┘
    │
    ▼
IM群聊广播积分变更 → 双方房间观众实时看到积分条更新
```

### 2.2 核心流程图：PK 结算（Lua原子读+判+写）

```
┌──────────────────────────────────────────────────────────────────┐
│                    PK 结算定时器体系                                │
│                                                                  │
│  pkEndingPool (Redis ZSET, score=PK结束时间)                      │
│       │                                                          │
│       ▼                                                          │
│  timerPKEnding (500ms轮询)                                       │
│       │                                                          │
│       ▼                                                          │
│  ZRANGEBYSCORE pkEndingPool 0 now  → 取出到期PK对                 │
└──────────────────────────────────────────────────────────────────┘
         │
         ▼
lockConnectStatus(showID_A, showID_B)  // 分布式锁
         │
         ▼
┌─────────────────────────────────────────────────────┐
│           结算 Lua 脚本 (原子执行)                     │
│                                                     │
│  ① GET pkTotalGrant_A, pkTotalGrant_B  (读双方积分)  │
│                                                     │
│  ② 比较大小 → 确定胜/负/平                            │
│                                                     │
│  ③ SET result_key = {winner, score_a, score_b}      │
│                                                     │
│  ④ SET pk_status = "finished"  ★关键: 阻止后续加分   │
│                                                     │
└─────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│  后续处理:                               │
│  ✓ 设置惩罚阶段结束时间                   │
│  ✓ ZADD pkPunishingPool (score=惩罚结束) │
│  ✓ 发送PK结果IM → 观众看到胜负动画        │
│  ✓ 发送Pulsar事件 → 下游数据处理          │
└─────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│  惩罚阶段结束 (timerPKPunishingEnding):   │
│  ✓ 重置双方PK状态为Idle                   │
│  ✓ 结束TRTC混流                          │
│  ✓ 发送连麦结束IM                         │
│  ✓ 保存排位赛积分结果(如果是排位赛)        │
│  ✓ 取消玩法互斥                           │
└─────────────────────────────────────────┘
```

### 2.3 结算与加分的互斥关系

```
时间线 ──────────────────────────────────────────────────────────►

     PK进行中                    结算点                  结算后
  ┌─────────────────────────┐     │      ┌────────────────────┐
  │ 加分Lua: 正常执行         │     │      │ 加分Lua: 检测到     │
  │  ① check pk_status     │     │      │  pk_status=finished │
  │     → 不是finished       │     │      │  → return 0 拒绝    │
  │  ② 去重+加分             │     │      │                    │
  └─────────────────────────┘     │      └────────────────────┘
                                  │
                           ┌──────┴──────┐
                           │ 结算Lua:    │
                           │ ① 读积分    │
                           │ ② 比大小    │
                           │ ③ 写结果    │
                           │ ④ 置finished│
                           └─────────────┘

关键: 两个Lua脚本各自原子, 通过pk_status标志位逻辑互斥
     结算后的加分一律被拒绝 → 不存在"判完胜负又被改分"
```

### 2.4 面试记忆要点

| 关键词 | 说明 |
|--------|------|
| 加分原子 | Lua: pk_status检查→去重(order_id)→INCRBY总分→ZINCRBY榜单 |
| 结算原子 | Lua: 读双方积分→比大小→写结果→置finished |
| 互斥机制 | pk_status标志位, 结算置finished后加分被拒绝 |
| 定时器 | pkEndingPool ZSET + 500ms轮询 + 分布式锁 |
| 惩罚阶段 | pkPunishingPool → 定时器到期 → 重置状态 |

### 3.3 信令可靠性三层保障

```
┌─────────────────────────────────────────────────────────────────┐
│                    信令可靠性保障机制                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  第一层: 关键信令 ACK 重试                                        │
│  ┌─────────────────────────────────────────────────────┐        │
│  │  服务端推送 → 等待ACK(超时) → 未收到 → 重试           │        │
│  │  保证关键信令(上麦通知/PK邀请)尽量不丢                  │        │
│  └─────────────────────────────────────────────────────┘        │
│                                                                 │
│  第二层: 状态以服务端为准                                          │
│  ┌─────────────────────────────────────────────────────┐        │
│  │  客户端状态展示由服务端推送驱动                          │        │
│  │  不依赖客户端本地计算                                   │        │
│  └─────────────────────────────────────────────────────┘        │
│                                                                 │
│  第三层: TRTC回调作为最终确认                                      │
│  ┌─────────────────────────────────────────────────────┐        │
│  │  上麦成功 ≠ 客户端说成功                               │        │
│  │  上麦成功 = 服务端收到TRTC进房回调                      │        │
│  │  → 然后才更新席位状态                                  │        │
│  └─────────────────────────────────────────────────────┘        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 面试记忆要点

| 关键词 | 说明 |
|--------|------|
| 麦位4态 | 空闲→申请中→已上麦→已锁定 |
| 上麦流程 | 申请→审批→签发UserSig→TRTC进房→回调确认→混流重算→广播 |
| 信令可靠 | ACK重试 + 服务端为准 + TRTC回调最终确认 |
| 下麦3种 | 主动下麦、被踢、断线超时 |
| 核心原则 | 以服务端TRTC回调为准，不信任客户端 |

```

│                                                                 │
```

```
│  │       ▼                                          │            │
│  │  是否"直播中"?                                    │            │
│  │    │           │                                 │            │
│  │    否          是                                 │            │
│  │    │           │                                 │            │
│  │    ▼           ▼                                 │            │
│  │  正常开播    比较UDID                              │            │
│  │              │         │                         │            │
```
