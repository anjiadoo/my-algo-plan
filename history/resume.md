### 腾讯游戏 | 后端开发工程师 | [起止时间请填写]

#### 项目：互娱通用抽奖系统

**项目背景**：深度参与腾讯游戏旗下通用抽奖系统的架构演进与核心优化。系统包含抽奖框架、运营配置后台和独立部署管理，以 HTTP 接口对外提供服务，每个业务每个活动独立部署实例、活动下线后自动回收资源。通过插件化架构被王者荣耀、英雄联盟、英雄联盟手游、剑灵、剑侠情缘等 10+ 款游戏复用，支撑互娱各类商业化抽奖活动（经典 Gacha、集卡合成、自选许愿池、无放回盲盒、阶梯递增等），各游戏通过实现差异化插件接入共享核心引擎。本人在已有抽奖组件基础上，负责多个核心模块的重构优化与新能力建设。

**系统规模**：服务 10+ 款游戏，系统设计容量 50 万 QPS；实际峰值出现在王者荣耀春节无双抽奖活动，峰值 1 万+ QPS（付费抽奖场景，转化链路长，QPS 不等同于 DAU）。抽奖接口 P99 < 50ms，限量道具零超发。

**核心工作与技术亮点**：

1. **主导插件化引擎的扩展与新游戏接入**：系统采用插件化架构，核心主流程固定（参数校验→概率组确定→多维过滤→加权采样→限量检查→状态持久化→响应缓存），提供 15+ 个 Hook 点（ProbIdGen、FilterChain、LuckyValue、FixProb、ReplaceItem、BuildRsp 等），后注册插件覆盖先注册实现。本人负责多个核心插件的开发与优化，并主导了多款新游戏的接入适配（如剑侠情缘按服务器等级选择奖池、LOL 定制道具响应格式等），通过实现差异化插件将新游戏接入周期从 2 周缩短至 2 天。

2. **优化分层概率模型与采样机制**：在原有概率模型基础上，优化了 Fixed/Unfixed 双轨概率组的缩放机制（Fixed 道具概率恒定不受过滤影响，Unfixed 被过滤后按比例自动缩放），扩展支持概率组嵌套（最大 10 层递归）。采样算法采用轮盘赌加权随机（过滤后概率组 per-user 不同且 per-request 可变，轮盘赌零预处理、天然适配动态过滤场景）；随机数生成引入 CSPRNG 安全种子 + xoshiro256** 快速 PRNG，确保统计公平且不可预测。

3. **重构并完善三级幂等保障机制**：线上出现因中间链路异常导致迟到请求覆盖缓存、活动侧拿到的结果与抽奖组件最终状态不一致的问题。针对该问题梳理并加固了抽奖组件侧的三级幂等体系——第一层响应缓存幂等（幂等key校验，命中且匹配则直接返回缓存响应，在加锁之前执行以减少 Redis 竞争）；第二层状态回退/回放（双缓存模式下回退 PrevState 重新随机，回放模式下按 PlayBackReward 逐轮重放历史结果，两种模式通过配置互斥选择）；第三层 Redis 分布式锁（SETNX 防同一用户并发抽奖），确保同一轮抽奖无论因何种原因被重复请求，抽奖组件始终返回一致结果，与下游发货组件的 `uk_actid_drawround` 唯一索引配合，实现端到端幂等。

4. **优化四层限量管控机制**：负责限量道具产出控制的稳定性优化，系统支持四层限量维度（用户累计限量、用户日限量、日期全服限量、全局永久限量），通过 Redis 原子递增计数实现并发安全——采样命中限量道具后原子递增计数器，超限则丢弃结果并触发重抽（最多 3 次），保证并发场景下绝不超发。

5. **扩展里程碑保底系统，支持业界全玩法覆盖**：在已有幸运值机制基础上，扩展支持多条规则多优先级配置，通过幸运值累加 + 阈值切换概率组，新增覆盖了软保底（概率递增）、大小保底（50/50 then 100%）、跨池继承等玩法，使保底系统能覆盖所有业界主流玩法，玩法变更仅需修改配置无需改代码。

6. **负责 Layer 多层抽奖隔离机制的设计与实现**：设计了 Layer 机制实现多轮次抽奖的完整隔离——每个 Layer 拥有独立的 Redis 用户状态、奖池道具集合、概率表和响应缓存，切换 Layer 等于开启全新一局。支持道具按 Layer 归属配置（精确匹配/通配/周期匹配），初始化时按 Layer 独立构建奖池。覆盖了多轮活动（抽完切换下一轮）、分段抽奖（不同 Layer 不同概率）、赛季制（赛季重置切新 Layer）等业务场景。

7. **完善多维过滤链，新增兜底概率组机制**：在原有过滤链（有效期→已有→背包→已抽→限量→顺序）基础上，优化过滤链的可配置能力，并新增了过滤后池为空时自动切换 complement 兜底概率组的机制（兜底也为空时返回 fallback 道具如金币），解决了极端场景下连抽空产出的线上问题。

### 腾讯音乐 | 后端开发工程师 | [起止时间请填写]

#### 项目：QQ音乐直播系统

**项目背景**：作为核心开发从 0 到 1 参与 QQ 音乐直播后台系统的建设，基于腾讯 TRTC 实时音视频服务构建，支撑开播/关播、进房/退房、推流/拉流、连麦/PK、房间管理、在线观众管理等全链路功能。本人与团队共同完成系统架构设计与落地，主要负责开关播、进退房、 WS长连接网关的落地调优与运维保障、在线观众管理等核心模块。

**系统规模**：系统设计容量为百万级 WS 长连接、十万级进退房 QPS、万级并发推流，架构上通过无状态服务 + 按房间分片可水平扩展至千万级连接。首屏 P95 < 1s，连麦端到端延迟 < 300ms。

**核心工作与技术亮点**：

1. **负责 WS 长连接网关的落地部署、容量调优与线上问题攻坚**：基于团队已有的 WS 网关组件，负责在直播场景下的适配落地与集群部署（单机 5 万连接，16 核 32G，epoll ET 模式 + goroutine pool）。深入理解网关内部实现，针对直播场景做了多项关键调优：调整 TCP socket buffer 从默认 128KB 降至 8KB 以支撑更高连接密度；实现服务端主动 Ping 的时间轮心跳管理（15 个槽位，每秒处理一个桶约 3333 个连接），替代客户端主动心跳避免惊群效应；上线后独立排查并解决了网关 OOM 导致单机连接瞬断、graceful shutdown 未通知客户端迁移等多个线上故障。架构上确保 WS 网关只做纯下行推送，上行业务请求走 API 网关，音视频流走 TRTC/CDN，三条链路物理隔离。

2. **负责分层扇出的房间消息广播机制**：实现了大房间高效消息广播链路：业务服务→MQ→Job 路由层查询房间连接分布（Redis Set `room:gateways:{room_id}`）→并行 gRPC 分发到多台网关→网关内存中找到本机该房间连接→非阻塞入队广播（消息只序列化一次）。针对慢连接设计了 `markSlow()` 机制，sendCh 队列满时标记待踢出，防止慢连接拖垮整个房间的广播。

3. **负责在线观众管理，解决热门直播间 Redis 热点问题**：热门直播间短时间大量用户涌入，同一 room_id 的 SADD 请求全打到 Redis Cluster 单分片导致延迟飙升。设计了分层在线人数方案：小房间（<1 万人）用 Redis Set 精确管理；大房间自动切换为分散计数器（`room:count:{room_id}:{user_id%16}` 打散到 16 个不同 slot）+ 环形 buffer 近似列表（只展示最近 100 人）。渐进式切换：Set 大小超 1 万时下次进房自动切换策略，小房间保留精确列表的业务价值（展示头像墙）。

4. **负责流状态机与断流保护机制**：推流状态机 `idle→pushing→live→closing→idle`，状态变更通过 MQ 广播让所有依赖方异步感知。关键设计：断流保护 30 秒窗口（网络抖动不误判关播，TRTC SDK 级自动重连 + 业务层保护期，保护期内观众看到"主播暂时离开"不踢出房间）；TRTC 事件回调乱序处理（事件序列号去重 + 时间戳检查 + 状态机合法性校验三重防护）。

5. **参与首屏秒开优化，P95 从 2.8s 优化至 680ms**：五项组合优化——① HTTPDNS + IP 直连（省 150ms）；② TCP 连接预建（推荐列表可见时预建连接池，省 200ms）；③ CDN 边缘 GOP 缓存（最大贡献项，省 800ms，观众进入时立即推送最近 I 帧）；④ 进房并行请求（房间信息 + WS 建连 + 拉流同时发起，省 200ms，拉流地址从列表页预下发）；⑤ 硬解码器预热（viewWillAppear 时预创建 VTDecompressionSession）。

6. **参与连麦信令与混流管理模块的开发**：连麦架构采用 TRTC 低延迟通信（主播 + 连麦者进同一 TRTC 房间，端到端 <300ms）+ 云端 MCU 混流合屏 + CDN 旁路推流（普通观众走 CDN 1-3s 延迟）。负责席位状态管理（空闲/邀请中/已上麦/已锁定）、TRTC UserSig 动态签发、混流布局自动计算（画中画模式）等子模块。针对连麦者弱网场景实现了自适应降级策略（网络极差→关闭视频仅保留音频→持续 30 秒不恢复自动下麦）。

7. **主导 MQ 消费积压的排查与优化**：晚高峰 MQ 消息量骤增导致消费积压，下行推送延迟从 10ms 飙到秒级。定位根因为逐条 gRPC 推送开销过大，设计了批量聚合推送方案（按 gateway 地址聚合，每 50ms 或满 500 条 flush 一次批量 gRPC），大幅减少 gRPC 调用次数，消费延迟恢复到毫秒级，CPU 利用率显著下降。
