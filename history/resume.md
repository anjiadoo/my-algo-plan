#### 项目：互娱通用抽奖系统

**项目背景**：深度参与腾讯游戏旗下通用抽奖系统的架构演进与核心优化。系统包含抽奖框架、运营配置后台和独立部署管理，以 HTTP 接口对外提供服务，每个业务每个活动独立部署实例、活动下线后自动回收资源。通过插件化架构被王者荣耀、英雄联盟、英雄联盟手游、剑灵、剑侠情缘等 10+ 款游戏复用，支撑互娱各类商业化抽奖活动（经典 Gacha、集卡合成、自选许愿池、无放回盲盒、阶梯递增等），各游戏通过实现差异化插件接入共享核心引擎。本人在早期版本的抽奖组件基础上，负责多个核心模块的重构优化与新能力建设。

**系统规模**：服务 10+ 款游戏，系统设计容量 50 万 QPS（水平扩展）；实际峰值某头部游戏春节抽奖活动，峰值 1 万+ QPS（付费抽奖场景，转化链路长，QPS 不等同于 DAU）。抽奖接口 P99 < 50ms，限量道具零超发。

**主要工作**：

1. **主导插件化引擎的扩展与新游戏接入**：系统采用插件化架构，核心主流程固定（参数校验→概率组确定→多维过滤→加权采样→限量检查→状态持久化→响应缓存），提供 15+ 个 Hook 点（ProbIdGen、FilterChain、LuckyValue、FixProb、ReplaceItem、BuildRsp 等），后注册插件覆盖先注册实现。本人负责多个核心插件的开发与优化，并主导了多款新游戏的接入适配（如剑侠情缘按服务器等级选择奖池、LOL 定制道具响应格式等），通过实现差异化插件粗略估计将新游戏接入周期从 2 周缩短至 2 天。

2. **优化分层概率模型与采样机制**：在原有概率模型基础上，优化了 Fixed/Unfixed 双轨概率组的缩放机制（Fixed 道具概率恒定不受过滤影响，Unfixed 被过滤后按比例自动缩放），扩展支持概率组嵌套（最大 10 层递归）。采样算法采用轮盘赌加权随机（过滤后概率组 per-user 不同且 per-request 可变，轮盘赌零预处理、天然适配动态过滤场景）；随机数生成引入 CSPRNG 安全种子 + xoshiro256 快速 PRNG，确保统计公平且不可预测。

3. **重构并完善三级幂等保障机制**：线上出现因中间链路异常导致迟到请求覆盖缓存、活动侧拿到的结果与抽奖组件最终状态不一致的问题。针对该问题梳理并加固了抽奖组件侧的三级幂等体系：第一层响应缓存幂等（幂等key校验，命中且匹配则直接返回缓存响应，在加锁之前执行以减少 Redis 竞争）；第二层状态回退/回放（双缓存模式下回退 PrevState 重新随机，回放模式下按 PlaybackReward 逐轮重放历史结果，两种模式通过配置互斥选择）；第三层 Redis 分布式锁（SETNX 防同一用户并发抽奖），确保同一轮抽奖无论因何种原因被重复请求，抽奖组件始终返回一致结果，与下游发货组件的 `uk_actid_drawround` 唯一索引配合，实现端到端幂等。

4. **优化四层限量管控机制**：负责限量道具产出控制的稳定性优化，系统支持四层限量维度（用户累计限量、用户日限量、全服日期限量、全服永久限量），通过 Redis 原子递增计数实现并发安全——采样命中限量道具后原子递增计数器，超限则丢弃结果并触发重抽（最多 3 次），保证并发场景下绝不超发。

5. **扩展里程碑保底系统，支持业界全玩法覆盖**：在基础幸运值机制基础上，扩展支持多条规则多优先级配置，通过幸运值累加 + 阈值切换概率组，新增覆盖了软保底（概率递增）、大小保底、跨池继承等玩法，使保底系统能覆盖所有业界主流玩法，玩法变更仅需修改配置无需改代码。

6. **负责多层抽奖隔离机制的设计与实现**：设计 Layer 机制实现多轮次抽奖的完整隔离——每个 Layer 拥有独立的 Redis 用户状态、奖池道具集合、概率表和响应缓存，切换 Layer 等于开启全新一局。支持道具按 Layer 归属配置（精确匹配/通配/周期匹配），初始化时按 Layer 独立构建奖池。覆盖了多轮活动（抽完切换下一轮）、分段抽奖（不同 Layer 不同概率）、赛季制（赛季重置切新 Layer）等业务场景。

7. **完善多维过滤链，新增兜底概率组机制**：在原有过滤链（有效期→已有→背包→已抽→限量→顺序）基础上，优化过滤链的可配置能力，新增过滤后池为空时自动切换 complement 兜底概率组的机制（兜底也为空时返回 fallback 道具如金币），解决了极端场景下连抽空产出的线上问题。

### 腾讯音乐 | 后端开发工程师 | [起止时间请填写]

#### 项目：QQ音乐直播系统

**项目背景**：参与 QQ 音乐直播后台系统的核心功能建设，基于腾讯 TRTC 实时音视频服务构建，支撑开播/关播、进房/退房、连麦/PK、弹幕消息、房间管理、在线观众管理等全链路功能。本人主要负责连麦/PK 系统、直播弹幕消息系统的设计与开发，同时参与开关播、进退房、在线观众管理等模块。

**系统规模**：系统容量设计支撑百万级在线用户、万级并发直播间、弹幕峰值万级 TPS，连麦基于 TRTC 端到端延迟 < 300ms。

**主要工作**：

1. **负责 连麦/PK 状态机设计与实现**：设计了完整的 PK 生命周期状态机（`idle→inviting→matching→pk_ing→punishing→finished`），每次状态变更携带版本号做 CAS 校验，防止并发信令导致状态跳变。核心难点在于跨房 PK 场景：两个主播分属不同 TRTC 房间，需要通过信令服务协调双方进入同一 TRTC 房间完成低延迟互动，再由云端 MCU 将混流旁路推送至各自 CDN 直播流，确保普通观众无感知切换。设计了 PK 邀请超时自动取消（30s）、对方拒绝/无响应的回退逻辑、以及 PK 进行中一方断线的容错策略（断线保护 15s，超时自动判负结算）。

2. **负责 PK 积分系统与实时结算机制**：PK 积分由支付/互动服务回调驱动（礼物折算积分、弹幕互动加分等），采用 Redis Lua 脚本原子执行去重检查（SISMEMBER 订单号）+总积分递增（INCRBY）+贡献榜递增（ZINCRBY），解决了支付回调重试导致同一笔礼物重复计分的问题。积分变更通过 IM 群聊消息广播至双方房间观众。设计了 PK 轮次计时器管理（基于 Redis Sorted Set + 500ms 定时器轮询实现精确的倒计时与自动结算触发），结算通过 Lua 脚本原子读取双方积分、比较并写入结果，同时置 pk_status=finished；加分 Lua 脚本开头检测该标志位，finished 后一律拒绝加分，通过状态标志位实现结算与加分的逻辑互斥，避免最后一刻加分改变已判定的胜负。结算后进入惩罚阶段（由 pkPunishingPool Redis ZSET 定时器池管理），惩罚结束后重置双方状态并保存排位赛积分结果。

3. **负责麦位管理与连麦信令可靠性保障**：设计了麦位的席位状态管理（`空闲→申请中→已上麦→已锁定`），主播可锁定指定麦位不接受新申请。连麦信令流程：用户申请上麦→主播审批通过→服务端签发 TRTC UserSig（临时鉴权凭证）→用户携带 UserSig 进入 TRTC 房间推流→服务端收到 TRTC 进房回调后更新席位状态并触发混流布局重算（MCU 合屏新增一路画面）→广播麦位变更事件给房间内所有观众。关键设计：上麦成功以服务端收到 TRTC 回调为最终确认，不依赖客户端本地状态；关键信令（上麦通知、PK 邀请等）通过长连接下发并要求客户端 ACK，超时未 ACK 则重试，防止弱网场景下信令丢失导致状态不一致。

4. **负责直播弹幕系统的运维与容量调优**：基于团队已有的 IM 弹幕消息组件，负责在 QQ 音乐直播场景下的适配与持续运维。系统采用长连接 + 长轮询拉模型（200ms 同步间隔，消息发送到用户收到时延 < 1s），选择拉模型而非推拉结合以保证实现简单可控。系统设计了大小房间隔离机制（以在线人数分界，80/20 原则），大房间集群使用无状态 L5 路由、小房间使用一致性 Hash 路由避免单台机器处理房间过多。消息按优先级隔离存储（大喇叭→信令→C2C 信令→礼物→普通群消息），优先拉取高优先级消息。本人负责根据业务特征调整写入限流策略（单房间每秒 200 条上限，客户端显示极限 70-100 条/s）、配置各类消息频控参数（聊天消息 15/s、礼物 PK 消息 3/s、道具消息 5/s、进房消息 70/s 等），以及合并写优化（100ms 窗口内最多 10 条合并为一次写入，减少 CKV CAS 冲突）。

5. **负责开关播流程与并发控制**：实现了 CreateShow 创建直播间接口的完整流程（前置检查→生成 ShowInfo→后置处理）。前置检查包含参数校验、开播权限校验、实名认证校验、重复开播校验等；生成 ShowInfo 阶段获取全局唯一 ShowID（CKV 自增 ID）、生成腾讯云 StreamID 及流地址、签发 TRTC 客户端密钥；后置处理发送开播事件到消息队列供下游业务消费。针对客户端 Bug 或网络抖动导致的频繁创建直播间问题，设计了 Lua 脚本保证 3s 内同一用户创建的 ShowInfo 为同一个，避免产生大量无用中间状态影响后续统计；针对跨设备重复开播问题，通过 UDID（设备标识）在前置检查中判别——若已在直播中且 UDID 一致则恢复上次直播，UDID 不一致则拒绝并提示"正在其他设备直播中"。

6. **负责进退房流程与在线观众管理**：进房流程：用户点击直播间格子→调用 JoinShow 接口获取 TRTC 进房密钥 UserSig、IM 房间命令字 RoomCMD、CDN 流地址→根据房间大小判断 TRTC 进房或 CDN 进房（TRTC 房间最大承载 10 万人，超大型直播间自动切换为 CDN 进房）→并发拉取关键数据（RestrictCheck 安全检测/踢人/拉黑/重复进房校验、GetAnchorInfo、GetIdentityInfo）→四个关键路径全部成功后进入直播间→异步发送进房广播 IM 消息及进房事件到消息队列。设计了快速进房优化：中心页已提前获取 ShowInfo，进房时并行执行 TRTC 进房与业务进房流程，用户先看到音视频画面再加载房间装扮信息，进房速度约等于 TRTC 进房耗时。在线观众管理使用在线观众组件（CKV/CKV+）+ 统一排行榜组件（CKV+ + MongoDB），用户进退房通过消息队列解耦上报至在线观众组件，维护在线状态及 Top100 活跃列表；客户端每 30s 发送用户心跳保证在线状态准确性。统一排行榜通过流水按 Key 聚合批量并行计算提升吞吐量，使用对账进程定期校验数据准确性。
